**Service (Сервис)** — это «умный сетевой адрес» для ваших подов.

В Kubernetes поды — существа временные. Они постоянно рождаются, умирают и меняют свои IP-адреса. Если твой фронтенд будет стучаться напрямую по IP пода бэкенда, то через 5 минут связь оборвется.

**Service решают две задачи:**

1. **Стабильность:** Дают единый постоянный IP и DNS-имя для подов.
2. **Балансировка:** Распределяют запросы между всеми подами, у которых есть нужная **Label** (метка).

---

Как это работает технически (Селекторы)

Сервис не «привязан» к деплойменту. Он просто сканирует кластер и говорит: «Все поды, на которых есть наклейка `project: rv`, теперь за мной! Я буду слать им трафик».

Если HPA создал 6 подов — сервис сразу видит все 6. Если осталось 2 — он шлет только на эти 2.

---

Типы сервисов (Основные 3)

1. ClusterIP (По умолчанию)

- **Для чего:** Общение внутри кластера (например, бэкенд стучится в базу).
- **Доступ:** Виден **только внутри** кластера. Снаружи (из браузера) на него не зайти.
- **DNS:** Создает имя типа `my-service.default.svc.cluster.local`.

2. NodePort

- **Для чего:** Простой способ выпустить приложение наружу.
- **Как работает:** Открывает одинаковый порт (в диапазоне 30000–32767) на **всех** нодах кластера.
- **Доступ:** Можно зайти по адресу `IP_любой_ноды:30001`.

3. LoadBalancer

- **Для чего:** Профессиональный выход в интернет (только в облаках типа AWS/GCP).
- **Как работает:** Облако само создает железный балансировщик с публичным IP.
- **В Minikube:** Чтобы этот тип заработал, нужно в отдельном окне запустить команду `minikube tunnel`.

---

Пример манифеста Service:

```yaml
# 1. СЕРВИС ДЛЯ БАЗЫ ДАННЫХ
apiVersion: v1
kind: Service
metadata:
  name: redis-service        # ЭТО ИМЯ станет доменным адресом внутри сети
spec:
  selector:
    app: my-database         # Ищем поды с этой меткой
  ports:
    - port: 6379             # Порт, на котором будет висеть Сервис
      targetPort: 6379       # Порт внутри контейнера Redis
---
# 2. ДЕПЛОЙМЕНТ БАЗЫ ДАННЫХ
apiVersion: apps/v1
kind: Deployment
metadata:
  name: redis-db
spec:
  replicas: 1
  selector:
    matchLabels:
      app: my-database
  template:
    metadata:
      labels:
        app: my-database
    spec:
      containers:
      - name: redis
        image: redis:alpine
---
# 3. ДЕПЛОЙМЕНТ БЭКЕНДА (Тот, кто обращается)
apiVersion: apps/v1
kind: Deployment
metadata:
  name: backend-app
spec:
  replicas: 2
  selector:
    matchLabels:
      app: my-backend
  template:
    metadata:
      labels:
        app: my-backend
    spec:
      containers:
      - name: js-app
        image: rvlxx/js:1.0
        env:
        - name: REDIS_URL
          value: "redis-service" # МАГИЯ: мы просто пишем имя сервиса!
```

DNS с именем сервиса работает только в подах, через coreDNS, на ноде работать не будет.

---

Основные команды для работы:

1. Проверка состояния

- **`kubectl get svc`** — список всех сервисов. Смотрим **Cluster-IP** (внутренний адрес) и **External-IP** (если это LoadBalancer).
- **`kubectl describe svc <name>`** — **самая важная команда**. Ищи строку `Endpoints`. Если там пусто — значит, селектор сервиса не нашел ни одного пода с нужной меткой, и трафик уходит в никуда.
- **`kubectl get ep <name>`** — кратко показать только **Endpoints** (реальные IP подов, на которые сервис перекидывает пакеты).

2. Создание «на лету» (без манифеста)

- **`kubectl expose deployment web --port=80 --target-port=4000 --type=NodePort`** — создаст сервис типа NodePort, который пробросит порт 80 сервиса на порт 4000 твоего приложения.
