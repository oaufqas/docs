**Service (Сервис)** — это «умный сетевой адрес» для ваших подов.

В Kubernetes поды — существа временные. Они постоянно рождаются, умирают и меняют свои IP-адреса. Если твой фронтенд будет стучаться напрямую по IP пода бэкенда, то через 5 минут связь оборвется.

**Service решают две задачи:**

1. **Стабильность:** Дают единый постоянный IP и DNS-имя для подов.
2. **Балансировка:** Распределяют запросы между всеми подами, у которых есть нужная **Label** (метка).

---

#### Как это работает технически (Селекторы)

Сервис не «привязан» к деплойменту. Он просто сканирует кластер и говорит: «Все поды, на которых есть наклейка `project: rv`, теперь за мной! Я буду слать им трафик».

Если HPA создал 6 подов — сервис сразу видит все 6. Если осталось 2 — он шлет только на эти 2.

При создании сервиса, он просто подхватывает метки указанной сущности (пода/деплоймента) и потом следит за подходящими подами. 

Сервис может не иметь собественный ip в кластере, если это явно указать в манифесте `сlusterIP: None`, но DNS имя для других подов у него все равно будет, и если по нему обратиться, можно подключиться сразу к нескольким подам, за которыми он следит: 

```yaml
apiVersion: v1
kind: Service
metadata:
  name: test-svc
  labels:
    svc: zxc
spec:
  clusterIP: None
  selector:
    po: dp
  ports:
  - name: js
    protocol: TCP
    port: 3000
    targetPort: 3000
  - name: nginx
    protocol: TCP
    port: 80
    targetPort: 80
```

## DNS

Когда ты создаешь **Service**, в кластере происходит магия на двух уровнях: «тихом» (переменные окружения) и «умном» (DNS).

1. Переменные окружения (Старый метод)

Это механизм из «каменного века» Kubernetes, который до сих пор работает. Когда создается **новый Под**, Kubernetes смотрит на список **уже существующих Сервисов** в этом неймспейсе и автоматически прописывает их адреса в переменные окружения Пода.

**Как это выглядит внутри Пода:**  
Если у тебя есть сервис `my-web-service` на порту `80`, то в новом Поде команда `env` покажет:

```bash
MY_WEB_SERVICE_SERVICE_HOST=10.107.95.142
MY_WEB_SERVICE_SERVICE_PORT=80
MY_WEB_SERVICE_PORT=tcp://10.107.95.142:80
```

**Главный минус (Ловушка):**  
Порядок важен! Если ты сначала запустил **Под**, а потом создал **Сервис**, то внутри Пода **не будет** этих переменных. Ему придется перезагрузиться, чтобы их подхватить. Поэтому на этот метод в продакшене почти не полагаются.

2. DNS (Современный метод)

Как только ты создаешь сервис с именем `db-service`, CoreDNS тут же создает запись:

- `db-service` -> `10.107.x.x` (IP сервиса).

**Как это работает для Пода:**  
Твоему приложению `(Node.js/Go)` не нужно знать IP. Оно просто делает запрос на `http://db-service`.

**Полное имя (FQDN):**  
На самом деле имя сервиса выглядит длиннее:  
`my-service.default.svc.cluster.local`

1. `my-service` — имя.
2. `default` — неймспейс.
3. `svc` — тип ресурса (Service).
4. `cluster.local` — домен кластера.

**Почему это круто?**

- **Межпространственное общение:** Если под в неймспейсе `dev` хочет постучаться в базу в неймспейсе `prod`, он просто обращается к `db.prod.svc.cluster.local`.
- **Динамичность:** Даже если ты пересоздашь сервис (с новым IP), DNS обновится мгновенно. Приложению не нужно ничего менять.


Посмотреть файл конфигурации DNS внутри любого пода:

```bash
kubectl exec -it <pod-name> -- cat /etc/resolv.conf
```

#### Типы сервисов (Основные 4)

1. ClusterIP (По умолчанию)

- **Для чего:** Общение внутри кластера (например, бэкенд стучится в базу).
- **Доступ:** Виден **только внутри** кластера. Снаружи (из браузера) на него не зайти.
- **DNS:** Создает имя типа `my-service.default.svc.cluster.local`.

![[Pasted image 20260330235359.png]]

2. NodePort

- **Для чего:** Простой способ выпустить приложение наружу.
- **Как работает:** Делает все тоже самое что и ClusterIP, и открывает одинаковый порт (в диапазоне 30000–32767) на **всех** нодах кластера.
- **Доступ:** Можно зайти по адресу `IP_любой_ноды:30001`.

![[Pasted image 20260330235142.png]]

3. LoadBalancer

- **Для чего:** Профессиональный выход в интернет (только в облаках типа AWS/GCP).
- **Как работает:** Облако само создает железный балансировщик с публичным IP.
- **В Minikube:** Чтобы этот тип заработал, нужно в отдельном окне запустить команду `minikube tunnel`.

![[Pasted image 20260330235842.png]]

4. ExternalName

- **Для чего**: Сделать DNS запись внешнего ресурса, например при подключении подов к внешней базе данных.
- **Как работает**: Создается простая DNS запись внутри кластера, которая ведет на внешний ресурс.

---

Пример манифеста Service:

```yaml
# 1. СЕРВИС ДЛЯ БАЗЫ ДАННЫХ
apiVersion: v1
kind: Service
metadata:
  name: redis-service        # ЭТО ИМЯ станет доменным адресом внутри сети
spec:
  selector:
    app: my-database         # Ищем поды с этой меткой
  ports:
    - port: 6379             # Порт, на котором будет висеть Сервис
      targetPort: 6379       # Порт внутри контейнера Redis
---
# 2. ДЕПЛОЙМЕНТ БАЗЫ ДАННЫХ
apiVersion: apps/v1
kind: Deployment
metadata:
  name: redis-db
spec:
  replicas: 1
  selector:
    matchLabels:
      app: my-database
  template:
    metadata:
      labels:
        app: my-database
    spec:
      containers:
      - name: redis
        image: redis:alpine
---
# 3. ДЕПЛОЙМЕНТ БЭКЕНДА (Тот, кто обращается)
apiVersion: apps/v1
kind: Deployment
metadata:
  name: backend-app
spec:
  replicas: 2
  selector:
    matchLabels:
      app: my-backend
  template:
    metadata:
      labels:
        app: my-backend
    spec:
      containers:
      - name: js-app
        image: rvlxx/js:1.0
        env:
        - name: REDIS_URL
          value: "redis-service" # МАГИЯ: мы просто пишем имя сервиса!
```

DNS с именем сервиса работает только в подах, через coreDNS, на ноде работать не будет.

Другой пример:

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: my-kuber-app
  labels: 
    app: backend
spec:
  containers:
  - name: kuber-app-image
    image: rvlxx/js:1.0
    ports:
	- name: http
	  containerPort: 8000
	- name: https
	  containerPort: 8100
---
apiVerion: v1
kind: Service
metadata:
  name: kuber-service
spec:
  selector:
    app: backend
  ports:
  - name: http
    targetPort: http  # Название порта пода
    port: 80  # Внешний порт для других подов в кластере
  - name: https
    targetPort: https  # Название порта пода
    port: 443  # Внешний порт для других подов в кластере	
```

Основные команды для работы:

1. Проверка состояния

- **`kubectl get svc`** — список всех сервисов. Смотрим **Cluster-IP** (внутренний адрес) и **External-IP** (если это LoadBalancer).
- **`kubectl describe svc <name>`** — **самая важная команда**. Ищи строку `Endpoints`. Если там пусто — значит, селектор сервиса не нашел ни одного пода с нужной меткой, и трафик уходит в никуда.
- **`kubectl get ep(endpoints) <svc_name>`** — кратко показать только **Endpoints** (реальные IP подов, на которые сервис перекидывает пакеты).


2. Создание «на лету» (без манифеста)

- **`kubectl expose deployment <dp_name> --port=80 --target-port=4000 --type=NodePort`** — создаст сервис типа NodePort, который пробросит порт 80 сервиса на порт 4000 твоего приложения.
