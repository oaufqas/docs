### Networking


1. **Node (Железо/ВМ):** ==В Kubernetes **одна ОС = одна Нода**==. Если у тебя 10 ПК, у тебя будет **10 Нод** У каждой ноды есть свой **реальный IP** в твоей локальной сети (например, `192.168.1.10`). Это адрес «физического» сервера. У Ноды две роли:
	-  **Инфраструктурная:** На ней работают системные штуки (`kubelet`, `kube-proxy`, `container runtime`), чтобы кластер не развалился.
	-  **Ресурсная:** Она отдает свои CPU и RAM твоим Подам. Поды — это просто процессы, которые «едят» ресурсы этой конкретной Ноды.

2. **Pod (Группа):** Внутри нод развернута виртуальная сеть кластера (через CNI). Каждый под получает свой **виртуальный IP** (например, `10.244.0.5`). ==Этот IP уникален для всего кластера==, и поды на разных нодах видят друг друга по нему.

3. **Container (Процесс):** Контейнеры внутри одного пода делят его IP (`10.244.0.5`) и различаются только по **портам** (один на 80, другой на 8080).

##### Как они связаны технически?

- **Виртуальный мост:** На каждой ноде CNI-плагин создает виртуальный коммутатор (bridge). Все поды этой ноды «воткнуты» в него виртуальными кабелями (`veth` пары).
- **Таблица маршрутизации:** Когда пакет уходит с ноды, операционная система смотрит в [[iptables|iptables]]: «Ага, пакет идет на `10.244.x.x`, это сеть соседней ноды», и отправляет его по туннелю на реальный IP той ноды.
- С любой ноды можно обращаться к подам напрямую, и это будет работать. Но k8s не открывает реальные сокеты для прослушивания на нодах, вместо этого, она создает правила в iptables, которые перехватывают трафик.

##### Когда **Scheduler** решает, что `testpod` будет жить на `Node-3`, эта связь записывается в **etcd**.

- **API-сервер** знает: «Под X находится на Ноде Y».
- Если делать `port-forward` или `logs`, API-сервер смотрит в etcd, находит имя ноды, берет её **реальный IP** и отправляет запрос **kubelet**-у на эту машину.

##### Когда один Под хочет написать другому:

- Он не идет в API-сервер.
- Он шлет пакет на **IP Пода**.
- Сетевой плагин (**CNI**) на текущей Ноде смотрит в свою таблицу маршрутизации: «Так, под с IP `10.244.3.5` сидит на Ноде с IP `192.168.1.50`».
- Пакет летит напрямую между Нодами (ПК) по локальной сети.

##### Краткий итог:

- **IP Ноды** — для управления сервером и входа извне.
- **IP Пода** — для внутреннего общения приложений между собой.
- **Порт Контейнера** — для разделения сервисов внутри одного пода.
- На каждом ПК (Ноде) работает агент **kubelet**. Он — «руки» Kubernetes на этом железе.
- **У кластера ОДНА сеть (Pod Network)**, в которой все работает, виртуальная. Остальное - вспомогательные сети нод, для связей подов между собой.

## Основные понятия kubernetes

###### 1. Инфраструктурный уровень

- **Кластер (Cluster):** Вся система целиком. Состоит из управляющего узла (Control Plane) и рабочих машин (Nodes).
- **Узел (Node):** Физический сервер или виртуальная машина, на которой работают ваши приложения.
- **Control Plane:** «Мозг» кластера. Он решает, где запустить контейнеры, следит за их здоровьем и реагирует на изменения.
- **DNS (CoreDNS)**: внутренняя DNS для обнаружения сервисов по именам (`my-service.default.svc.cluster.local`)
- **Network Plugin (CNI)**: отвечает за сеть между подами (Calico, Flannel, Cilium)
- **Metrics Server**: собирает метрики CPU/памяти для `kubectl top` и авто-масштабирования
- **Ingress Controller**: реализация Ingress (nginx-ingress, traefik)

###### 2. Уровень приложений (Объекты)

- **[[pods|Под (Pod)]]:** Самая мелкая единица в K8s. Это «оболочка» для одного или нескольких **контейнеров**, которые делят общие ресурсы (IP-адрес и порты). Контейнеры внутри одного пода всегда запускаются вместе на одном узле.
- **[[deployment|Deployment (Деплоймент)]]:** Контроллер, который описывает желаемое состояние приложения. Вы указываете: «хочу 3 копии пода версии 1.2», и Deployment следит, чтобы их всегда было именно столько.
- **[[other-essence#Jobs (Разовая задача)|Job]]**: выполняет задачу один раз и завершается (миграции БД, бэкапы)
- **[[other-essence#CroneJobs (Задачи по расписанию)|CronJob]]**: выполняет задачу по расписанию

- **[[deployment#**ReplicaSet** (controller)— это «счетчик-контролер» с одной-единственной задачей поддерживать заданное количество идентичных Подов в любой момент времени.|ReplicaSet]]:** Механизм внутри деплоймента, который отвечает за поддержание нужного количества идентичных копий подов.
- **[[other-essence#StatefulSet|StatefulSet (StatefulSet)]]**: для приложений с состоянием (базы данных, очереди). В отличие от Deployment: поды имеют **стабильные имена** (`db-0`, `db-1`), стабильное сетевое хранилище (каждый под получает свой PVC), упорядоченное развертывание и удаление
- **[[other-essence#DaemonSet|DaemonSet (DaemonSet)]]**: Запускает **один под на каждой ноде**. Используется для: сбора логов (Fluentd), мониторинга (Prometheus Node Exporter), сетевых плагинов
- **[[monitoring|Monitoring]]**: Программный стек наблюдения за работоспособностью объектов кластера
###### 3. Сетевой уровень и доступ

- **[[services|Сервис (Service)]]:** Постоянный абстрактный адрес для группы подов. Поды могут умирать и менять IP, но Service дает им единое имя и **балансирует трафик** между ними.
- **[[ingress|Ingress]]:** Входная точка в кластер. Правила, которые определяют, как внешний HTTP/HTTPS трафик из интернета попадет к конкретным сервисам внутри сети.
- **Network Policies**: Правила, которые определяют, какие поды могут общаться друг с другом. K8s по умолчанию разрешает всё. Network Policies — это **firewall внутри кластера**.
###### 4. Конфигурация и данные

- **Namespace (Пространство имен):** Виртуальное разделение кластера на части (например, `dev`, `test`, `prod`), чтобы команды или проекты не мешали друг другу.
- **[[configmaps-secrets|ConfigMap и Secret]]:** Объекты для хранения настроек (пароли, ключи, конфиги) отдельно от кода приложения.
- **[[notes/8_orchestration/kubernetes/volumes|Volume (Том)]]:** Директория с данными, которая доступна контейнерам в поде. В отличие от обычного Docker, данные в томах K8s могут сохраняться даже после перезапуска пода.
- **PV (PersistentVolume**): ресурс хранения в кластере (абстракция над физическим диском)
- **PVC (PersistentVolumeClaim**): запрос на хранилище от приложения
- **StorageClass**: Позволяет автоматически создавать PV при запросе PVC. Нужен для динамического выделения хранилища в облаках.
###### 5. Безопасность и доступ

- **ServiceAccount (Service Account)**: Учетная запись, от которой запускаются поды. Используется для доступа к API Kubernetes из пода. Каждый под по умолчанию получает ServiceAccount `default`.
- **[[rbac|RBAC (Role-Based Access Control)]]**:  Система разграничения прав:
	1. **Role / ClusterRole** — что можно делать
	2. **RoleBinding / ClusterRoleBinding** — кому можно
###### 6. Инструменты управления

- **[[kubectl|Kubectl]]:** Основная утилита командной строки для управления кластером.
- **YAML-манифест:** Файл, в котором вы описываете все вышеперечисленные объекты (поды, сервисы и т.д.) на человекочитаемом языке.
- **[[helm|Helm]]** (пакетный менеджер для K8s)
