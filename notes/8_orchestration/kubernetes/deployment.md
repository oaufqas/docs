**Deployment** (controller) — это «менеджер» или «завхоз» для ваших [[pods|Подов]]. В реальной работе никто не запускает одиночные Поды вручную (как мы делали до этого), потому что если такой Под упадет или сервер перезагрузится, он **не поднимется сам**. Deployment решает эту проблему, обеспечивая **отказоустойчивость** и **обновления**.

#### Зачем он нужен?

1. **Самолечение (Self-healing):** Ты говоришь: «Я хочу, чтобы всегда работало 3 копии приложения». Если один Под зависнет или Нода под ним сгорит, Deployment тут же скомандует запустить новый Под на другой Ноде.
2. **Масштабирование (Scaling):** Одной командой (или автоматически по нагрузке) ты можешь превратить 3 Пода в 30, и Deployment плавно их достроит.
3. **Обновления без простоя (Rolling Updates):** Когда выходит версия `2.0`, Deployment не выключает всё сразу. Он запускает один новый Под, ждет, пока тот заработает, удаляет один старый, и так по очереди. Пользователи даже не заметят процесса.
4. Если ты удалишь Под, который был создан Деплойментом (`kubectl delete pod ...`), ты увидишь магию: через секунду появится **новый Под** с другим именем. Deployment увидел, что реальность не совпадает с его планом (стало 2 пода вместо 3), и исправил это.

#### Как он устроен внутри (Матрешка)

Deployment не управляет Подами напрямую. Между ними есть еще одна прослойка — **ReplicaSet**.

- **Deployment** хранит стратегию (какой образ, как обновлять).
- **ReplicaSet** следит за количеством (чтобы их было ровно 3).
- **Pod** — это сама работающая программа.

---

Технически  — это **контроллер**, работающий внутри `kube-controller-manager`. Это не контейнер и не процесс, а **бесконечный цикл (Reconciliation Loop)**, который сверяет «желаемое» с «действительным».

Deployment не управляет Подами напрямую. Он управляет промежуточным объектом — **ReplicaSet**:

- **Deployment** — хранит «рецепт» (образ `v1`, стратегия обновления).
- **ReplicaSet** — «счетчик». Его единственная задача — держать запущенными ровно `N` подов.
- **Pod** — конечная единица с контейнером.

#### Как это работает в etcd

Когда ты применяешь манифест Deployment:

1. **API-сервер** записывает объект Deployment в базу **etcd**.
2. **Deployment Controller** видит новую запись и создает объект **ReplicaSet** (тоже запись в etcd).
3. **ReplicaSet Controller** видит, что ему нужно 3 пода, а их 0, и создает 3 записи об объектах **Pod**.
4. **Scheduler** видит поды без узла и назначает их на ноды.

#### Техника обновления (Rolling Update)

Когда ты меняешь образ в Deployment (например, с `v1` на `v2`), происходит магия:

1. Deployment создает **второй (новый) ReplicaSet** с образом `v2`.
2. Он говорит новому ReplicaSet: «сделай 1 под».
3. Он говорит старому ReplicaSet: «удали 1 под».
4. Так продолжается, пока старый ReplicaSet не обнулится, а новый не достигнет нужного количества.  
    _Если что-то пойдет не так (новый под упадет), Deployment остановит процесс, и у тебя останутся рабочие старые поды._

5. Селекторы и Владение (OwnerReference)

#### Как они находят друг друга в этой каше?

- Deployment ищет ReplicaSet по **Labels** (меткам).
- Внутри каждого Пода в поле `metadata.ownerReferences` записан ID его «родителя» (ReplicaSet).
- Если ты вручную удалишь ReplicaSet, Kubernetes по цепочке (Garbage Collection) удалит все его Поды.

#### Статус и История (Revision)

Deployment хранит историю изменений. Каждый раз, когда ты меняешь `spec.template`, создается новый ReplicaSet, а старый не удаляется совсем, а просто масштабируется в 0. Это позволяет мгновенно сделать **Rollback** (откат), просто вернув количество реплик старому ReplicaSet.


---

#### **ReplicaSet** (controller)— это «счетчик-контролер» с одной-единственной задачей: поддерживать заданное количество идентичных Подов в любой момент времени.

Если Deployment — это «стратег» (решает, какую версию софта ставить), то ReplicaSet — это «исполнитель» (следит за порядком в строю).

ReplicaSet работает как бесконечный цикл (Reconciliation Loop) обратной связи:

1. **Смотрит в конфиг:** «Пользователь хочет 3 Пода с меткой `app: web`».
2. **Считает реальность:** «Вижу в кластере только 2 Пода с меткой `app: web`».
3. **Действует:** Отправляет запрос API-серверу: «Создай еще один Под по шаблону».

Есть более старая версия - **ReplicationController (rc)** уже не используется

Ключевые технические характеристики rs:

1. **Селектор (Магнит):** ReplicaSet определяет «свои» Поды исключительно по **Labels** (меткам). Ему всё равно, как называется Под, ему важно, чтобы на нем висел нужный «стикер».
2. **Владение (OwnerReference):** Когда ReplicaSet создает Под, он записывает в его метаданные свой ID. Это позволяет Kubernetes понимать иерархию (кто чей «папа»).
3. **Шаблон (Pod Template):** Внутри ReplicaSet всегда лежит копия описания Пода. Если нужно создать новый, он просто копирует этот чертеж.

Пример манифеста:

```yaml
apiVersion: apps/v1   # Обязательно для новой версии rs
kind: ReplicaSet   # ReplicationController старая версия не используется 
metadata:
  name: test-rs
  labels:
    app: test
spec:
  replicas: 2
  selector:
    matchLabels:   # Обязательно в новой версии rs
      app: test2
  template:
    metadata:
      name: testpod
      labels:
        app: test2
    spec:
      containers:
        - name: nginx
          image: nginx:latest
          ports:
	        - containerPort: 80
	          

# Другой вариант
matchExpressions:
  - key: app
    operator: In
    values:
	  - kuber
	  - http-server
  - key: env
    operator: Exists
```

Почему мы не пишем манифесты для ReplicaSet напрямую?

Хотя ты можешь создать файл `kind: ReplicaSet`, в реальности это делают только через **Deployment**. Вот почему:

- **Обновления:** Если ты изменишь образ (image) в самом ReplicaSet, он **ничего не сделает** с уже запущенными Подами (они так и останутся старыми). Тебе придется удалять их вручную.
- **История:** Deployment умеет создавать **новые** ReplicaSet при обновлении и хранить **старые** (с 0 реплик) для быстрого отката (Rollback).

При запуске Deployment `node-app-deploy`:

```bash
kubectl get replicaset (или сокращенно rs)
```

Ты увидишь имя вроде `node-app-deploy-67fb779554`.

- Первая часть — имя **Deployment**.
- Вторая часть — **хэш** шаблона Пода (если ты изменишь хоть одну букву в шаблоне, хэш изменится и создастся новый ReplicaSet).

**Итог:** ReplicaSet — это низкоуровневый механизм обеспечения «живучести». Он тупой, но очень надежный: его не волнует версия кода, его волнует только **количество**.

Разбор манифеста (Что в нем главное?)


```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: node-app-deploy
spec:
  replicas: 3              # Сколько штук должно жить всегда
  selector:
    matchLabels:           # СЕЛЕКТОР: ищи поды с этой меткой
      app: web-server
  template:                # ШАБЛОН: какой Под создавать, если его нет
    metadata:
      labels:
        app: web-server    # МЕТКА: должна совпадать с селектором выше!
    spec:                  # Внутри этого блока — обычное описание Пода
      containers:
      - name: my-node-app
        image: rvlxx/js:1.0
```

Основные команды для работы:

- `kubectl create deploy testdp --image nginx:latest` — создать деплоймент командой
- **`kubectl get deploy`** — посмотреть статус (сколько подов из скольки готовы).
- **`kubectl scale deployment node-app-deploy --replicas=5`** — мгновенно увеличить количество копий до 5.
- **`kubectl set image deployment/<dp_name> <old_container_name>=<new_image_name>`** — запустить обновление на новую версию образа.
- **`kubectl rollout undo deployment/<dp_name> --to-revision=2`** — «отмена»! Если версия 2.0 оказалась глючной, эта команда мгновенно вернет всё к предыдущей версии 1.0.
- `kubectl rollout history deployment/<dp_name>` — история изменений
- `kubectl rollout restart deployment/<dp_name>` — пересоздать поды

**Autoscaling (HPA - HorizontalPodAutoscaler)** — автоматическое поднятие реплик, в зависимости от нагрузки на сервер

```yml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: my-autoscaling
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: web
  minReplicas: 2
  maxReplicas: 6
  metrics:
    - type: Resource
      resource:
        name: cpu
        target:
          type: Utilization
          averageUtilization: 70
    - type: Resource
      resource:
        name: memory
        target:
          type: Utilization
          averageUtilization: 80
```

###### Искуственная нагрузка на процессор

```bash
# Выбираем один из подов нашего деплоймента и грузим его
kubectl exec -it $(kubectl get pod -l project=rv -o name | head -n 1) -c f1 -- sh -c "while true; do md5sum /dev/urandom; done"
```
