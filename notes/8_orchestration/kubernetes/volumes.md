В Kubernetes контейнеры по своей природе **эфемерен** (временны). Если контейнер упадет или перезапустится, все файлы, которые он создал внутри себя, **исчезнут**.

**Volumes (Тома)** решают две задачи:

- **Сохранение данных** при перезапуске контейнера.
- **Обмен файлами** между несколькими контейнерами в одном Поде.

---

### Типы Volumes (от простого к сложному)

#### **emptyDir** (Временное хранилище)

Создается в момент запуска Пода и живет, пока жив Под. Если контейнер внутри Пода упадет — данные выживут. Если удалят сам Под — данные удалятся.

- **Зачем:** Кэши, временная папка для обработки видео, разделяемая папка для двух контейнеров в одном Поде.

```yaml
containers:
	- name: js
	  image: rvlxx/js:1.0
	  ports:
	  - containerPort: 80
	  env:
	  - name: "PORT"
	    value: "80"
	  volumeMounts:
	  - mountPath: /cache-1
	    name: cache-volume
	  
    - name: python
	  image: rvlxx/py:1.0
	  ports:
	  - containerPort: 90
	  env:
	  - name: "PORT"
	    value: "90"
	  volumeMounts:
	  - name: cache-volume
	    mountPath: /cache-2
	   readOnly: true

volumes:
  - name: cache-volume
	emptyDir: {}
```

#### **hostPath** (Связь с узлом)

Монтирует файл или директорию с жесткого диска **самой ноды** (хоста) в Под.

- **Зачем:** Когда Поду нужен доступ к системным логам ноды (`/var/log`) или Docker-сокету.
- **Минус:** Если Под переедет на другую ноду, он не найдет там своих данных.

```yaml
containers:
  - image: nginx
    name: nginx
    volumesMounts:
	- mountPath: /test-pd
	  name: test-volume
    
volumes:
    - name: test-volume
      hostPath: 
        path: /data
        type: Directory
```

##### **PersistentVolume (PV) и PersistentVolumeClaim (PVC)** — Стандарт для баз данных

##### **PersistentVolume (PV)** — это реальный кусок диска (облачный диск AWS/GCP, NFS-шара или локальный диск), который админ выделил кластеру. Это объкт кластерного уровеня, у него нет своих неймспейсов (как и node).

```yaml
apiVersion: v1
kind: PersistentVolume
metadata:
  name: aws-test-vlm
  labels:
    type: aws-pc
spec:
  capacity:
    storage: 3Gi # Размер хранилища
  volumeMod: Filesystem # Файловая система или сырой блок (Block)
  accessModes:
  - ReadWriteOnce # Кто и как может подключаться (RWO, ROX, RWX)
  persistentVolumeReclaimPolicy: Retain # Что делаеть с данными после удаления
  storageClassName: "" # Метка для связи с нужным PVC
  hostPath:
    path: "/mnt/data"
  awsElasticBlockStore:
    volumeID: "vol-askjadsg293485y"
    fsType: ext4
```

1. Разбор ключевых настроек:

- **Reclaim Policy (Политика очистки):**
    - `Retain`: (Самое частое) Если вы удалите PVC, сам PV перейдет в статус `Released`. Данные **не удаляются**, но другой PVC не сможет его занять, пока админ не почистит его вручную.
    - `Delete`: При удалении PVC удаляется и PV, и реальные данные в облаке (автоматически для облачных дисков).
    - `Recycle`: (Устарело) Просто делает `rm -rf /` внутри папки и отдает диск новому пользователю.
- **Access Modes (Режимы доступа):**
    - `ReadWriteOnce (RWO)`: Только один узел может писать/читать.
    - `ReadOnlyMany (ROX)`: Многие узлы могут только читать (например, общая база картинок).
    - `ReadWriteMany (RWX)`: Многие узлы пишут и читают одновременно (нужен NFS или Ceph).


2. Жизненный цикл (Статусы PV)

Когда вы смотрите список через `kubectl get pv`, вы видите его состояние:

- **Available**: Свободен, ждет своего принца (PVC).
- **Bound**: «Женат» на конкретном PVC.
- **Released**: PVC удален, но данные еще на диске (при политике `Retain`).
- **Failed**: Ошибка при попытке очистки или монтирования.


4. Связка PV и PVC (Как они находят друг друга?)

Чтобы PVC «забайндился» (подключился) к PV, должны совпасть 3 условия:

- **StorageClassName**: Должен быть одинаковым (или оба пустые).
- **AccessModes**: PVC не может требовать `RWX`, если PV умеет только `RWO`.
- **Размер**: В PV должно быть **не меньше** места, чем просит PVC. (Если в PV 20ГБ, а PVC просит 10ГБ — они сойдутся, но 10ГБ пропадут зря).

---

### **PersistentVolumeClaim (PVC)** — это «заявка» от разработчика: «Мне нужно 10 ГБ места с быстрым доступом».

Разработчику не нужно знать, где физически находится хранилище (в облаке AWS, на NFS-сервере или на соседнем SSD). Он просто описывает в PVC: «Мне нужно 5 ГБ с возможностью записи, примонтируйте это к моему Поду».

1. Как работает Binding

Когда вы создаете PVC, Kubernetes ищет подходящий **PV** в кластере. Чтобы они «сошлись», должны совпасть:

1. **StorageClass**: Имя класса должно быть одинаковым.
2. **AccessModes**: Если PVC требует `ReadWriteMany`, а PV умеет только `ReadWriteOnce`, они не свяжутся.
3. **Размер**: В PV должно быть **столько же или больше** места, чем просит PVC.

Как только пара найдена, статус PVC меняется с `Pending` на `Bound`. Теперь этот диск «забронирован» за вашим запросом.


2. Подробный разбор манифеста PVC

```yaml
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: mysql-data-pvc        # Имя, которое мы укажем в настройках Пода
spec:
  storageClassName: standard  # Тип диска (например, "fast", "slow", "ssd")
  accessModes:
    - ReadWriteOnce           # Режим доступа 
  resources:
    requests:
      storage: 5Gi            # Сколько места нам нужно
```

Ключевые поля:

- **`storageClassName`**: Самое важное поле для автоматизации.
    - Если указать имя (например, `gp2`), облако само создаст диск.
    - Если оставить пустым `""`, K8s будет искать только вручную созданные (Static) PV.
- **`accessModes`**:
    - `ReadWriteOnce (RWO)` — диск подключается только к одному узлу (ноде). Идеально для баз данных.
    - `ReadWriteMany (RWX)` — диск доступен многим узлам одновременно (нужен для общих папок, например, через NFS).
- **`resources.requests.storage`**: Минимальный объем. Если в кластере есть PV на 10ГБ, а вы просите 5ГБ, K8s отдаст вам этот PV (но 5ГБ «сгорят», так как PV нельзя делить между разными PVC).


3. Как использовать PVC в Поде

После того как PVC создан и связан (`Bound`), его нужно «вставить» в контейнер:

```yaml
spec:
  containers:
  - name: my-app
    image: nginx
    volumeMounts:
    - name: storage-volume
      mountPath: /data       # Путь ВНУТРИ контейнера
  volumes:
  - name: storage-volume
    persistentVolumeClaim:
      claimName: mysql-data-pvc # Имя нашего PVC
```


4. Важные нюансы

- **PVC — это неизменяемый объект**: Вы не можете просто поменять `ReadWriteOnce` на `ReadWriteMany` в уже созданном PVC. Придется удалять и создавать заново (данные могут удалиться в зависимости от политики PV).
- **Расширение диска (Resizing)**: Многие современные системы позволяют увеличивать размер диска «на лету». Вы правите `storage: 5Gi` на `10Gi`, и если `StorageClass` это поддерживает (`allowVolumeExpansion: true`), диск увеличится сам.
- **Зависание при удалении**: Если вы удалите PVC, который всё еще используется работающим Подом, он зависнет в статусе `Terminating`. K8s защищает данные и не удалит диск, пока Под жив.

---

#### **StorageClass** — автоматизатор, который сам создает облачный диск, когда видит новый PVC.

#### Пример:

```yaml
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: fast-ssd           # Имя, которое разработчик укажет в PVC
provisioner: kubernetes.io/gce-pd # "Драйвер", который пойдет в облако создавать диск
parameters:
  type: pd-ssd             # Тип диска в терминах облака
  replication-type: none
reclaimPolicy: Delete      # Что сделать с физическим диском, когда удалят PVC?
allowVolumeExpansion: true # Можно ли будет потом увеличить размер диска?
volumeBindingMode: WaitForFirstConsumer # Когда именно создавать диск? (Важно!)

```


#### Ключевые параметры (подробно)

**Provisioner (Провиженер)**

Это самое главное поле. Оно указывает, какой плагин будет создавать диск.

- `kubernetes.io/aws-ebs` — для AWS.
- `rancher.io/local-path` — для создания папок на локальном диске ноды.
- `cluster.local/nfs` — для нарезки места на NFS-сервере.

**Reclaim Policy (Политика возврата)**

Определяет судьбу реального диска в облаке после удаления PVC:

- `Delete` (по умолчанию): Удалили PVC — удалился и диск в облаке. Экономит деньги.
- `Retain`: Удалили PVC — диск в облаке остался. Полезно для очень важных баз данных, чтобы случайно не стереть терабайты данных.

**Allow Volume Expansion**

Если стоит `true`, вы сможете просто поменять `5Gi` на `10Gi` в своем PVC, и Kubernetes сам «растянет» раздел на лету (если облако это умеет).

**Volume Binding Mode**

Это тонкая настройка того, **когда** создавать диск:

1. **Immediate** (сразу): Как только вы создали PVC, облако тут же резервирует диск.
    - _Проблема:_ Диск может создаться в Зоне А, а ваш Под планировщик захочет запустить в Зоне Б. Они не смогут соединиться.
2. **WaitForFirstConsumer** (рекомендуется): Кубернетес подождет, пока появится Под, использующий этот PVC. Поймет, на какой именно ноде (в какой зоне) будет жить Под, и только тогда создаст диск именно там.

### StorageClass по умолчанию (Default)

В большинстве кластеров один из классов помечен как дефолтный. Это значит, что если разработчик в PVC вообще **не укажет** поле `storageClassName`, диск всё равно создастся через этот класс.

Посмотреть список можно командой:  
`kubectl get sc`  
_(Тот, у которого в скобках написано `(default)`, и есть основной)._


#### Как это работает вместе (Цепочка событий)

1. **Разработчик** создает PVC и пишет: `storageClassName: fast-ssd`.
2. **StorageClass** видит запрос и вызывает свой **Provisioner**.
3. **Provisioner** идет в API облака (или в софт типа Ceph) и просит: «Создай диск на 10ГБ, тип SSD».
4. Облако создает диск и возвращает его ID.
5. StorageClass автоматически создает объект **PersistentVolume (PV)** в кластере, прописывает туда ID диска и связывает его с вашим PVC.


---

Итого: что выбрать?

1. Нужно просто пошарить файлы между контейнерами в одном Поде? → **emptyDir**.
2. Нужно сохранить конфиг или секрет как файл? → **ConfigMap/Secret** (они тоже монтируются как Volumes).
3. Нужно хранить данные базы данных (Postgres, MongoDB)? → **PVC + StorageClass**.

### Если в облаке yc после команды delete не удаляются pv и pvc:

Если подов нет, а статус все равно `Terminating`, значит в объекте остался «финализатор» — метка, которая ждет подтверждения от облачного контроллера Yandex Cloud, что диск физически удален. Иногда связь обрывается, и подтверждение не приходит.

**Для PVC:**

```bash
kubectl patch pvc ИМЯ_PVC -p '{"metadata":{"finalizers":null}}' --type=merge
```

**Для PV:**

```bash
kubectl patch pv ИМЯ_PV -p '{"metadata":{"finalizers":null}}' --type=merge
```

### RideWriteMany

Если нужно общее хранилище, из которого сможет читать несколько подов, а не 1 (как в `ReadWriteOnce`) можно поднять nfs сервер в кластере, который выделит у облака pv с ReadWriteOnce, тк облака в основном напрямую не поддерживают RWM, в кластере появится новый StorageClass, который сможет создавать для подов RWM от куска выделенного для сервера диска. 

[[helm#Установка nfs сервера|Установка через Helm]]

Создание PVC уже для подов с RWM:

```yaml
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: # Имя pvc
spec:
  accessModes:
    - ReadWriteMany # !       
  storageClassName: ## Название StorageClass'a от nfs сервера
  resources:
    requests:
      storage: ## Размер диска
```
