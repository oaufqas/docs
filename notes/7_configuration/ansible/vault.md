## Ansible Vault - это встроенный инструмент для шифрования данных, который позволяет безопасно хранить пароли, ключи API, сертификаты и другие секреты в плейбуках и ролях.


#### Основные команды:

```bash
# Создание зашифрованного файла
ansible-vault create <filename>

# Редактировать содержимое уже зашифрованного файла
ansible-vault edit <filename>

# Зашифровать существующий текстовый файл
ansible-vault encrypt <filename>

# Навсегда расшифровать файл (вернуть в открытый вид).
ansible-vault decrypt <filename>

# Просмотреть содержимое без открытия в редакторе
ansible-vault view <filename>
```


Как использовать в Playbook

При запуске плейбука, содержащего зашифрованные данные, Ansible нужно предоставить пароль. Сделать это можно двумя способами:
1. **Интерактивно**: добавить флаг `--ask-vault-pass` при запуске `ansible-playbook`.
2. **Через файл**: указать путь к файлу с паролем через параметр `--vault-password-file` или переменную окружения `ANSIBLE_VAULT_PASSWORD_FILE`.


```yml
- name: Deploy app
  hosts: all
  vars_files:
    - secrets.yml
  tasks:
    - name: DB password
      debug:
        msg: "Пароль: {{ db_password }}"

```


### Основные встроенные переменные, доступные в ansible playbooks:

#### 1. Самые популярные переменные:

- **`hostvars`** — позволяет получить значения переменных других хостов из инвентаря.
- **`inventory_hostname`** — имя текущего хоста так, как оно прописано в файле инвентаря.
- **`ansible_play_hosts`** — список всех хостов, участвующих в текущем «плее» (play).
- **`groups`** — словарь всех групп в инвентаре и хостов, входящих в них.
- **`group_names`** — список групп, к которым принадлежит текущий хост.
- **`role_path`** — путь к текущей роли (полезно для указания путей к файлам внутри роли). 

#### 2. Часто используемые факты (Ansible Facts), факты собираются автоматически модулем `setup` (если не указано `gather_facts: false`): 


- **`ansible_distribution`** — название операционной системы (например, `Ubuntu`, `CentOS`).
- **`ansible_os_family`** — семейство ОС (например, `Debian` или `RedHat`), часто используется в условиях `when` для выбора пакетного менеджера.
- **`ansible_default_ipv4.address`** — основной IP-адрес хоста.
- **`ansible_memtotal_mb`** — общий объем оперативной памяти в мегабайтах.
- **`ansible_processor_vcpus`** — количество виртуальных ядер процессора.
- **`ansible_fqdn`** — полное доменное имя узла.

#### 3. Переменные соединения, используются для переопределения параметров подключения к хостам:


- **`ansible_user`** — имя пользователя для подключения по SSH.
- **`ansible_host`** — реальный IP-адрес или доменное имя для подключения (если оно отличается от `inventory_hostname`).
- **`ansible_port`** — порт для подключения (по умолчанию 22).
- **`ansible_ssh_private_key_file`** — путь к приватному ключу SSH.
- **`ansible_become`** — логический флаг (true/false) для активации повышения 

#### 4. Среда и контекст выполнения (Special Variables):


- **`ansible_check_mode`** — принимает значение `true`, если плейбук запущен с флагом `--check`. Полезно для пропуска задач, которые не могут работать в режиме проверки.
- **`ansible_diff_mode`** — `true`, если используется флаг `--diff`, показывающий изменения в файлах.
- **`ansible_version`** — словарь с информацией о версии Ansible (полезно для проверки совместимости ролей).
- **`play_name`** — имя текущего «плея» (play), указанное в параметре `name`.
- **`inventory_dir`** — абсолютный путь к директории, в которой находится файл инвентаря.
- **`ansible_play_batch`** — список имен хостов, которые находятся в текущей «партии» обработки (актуально при использовании `serial`). 

#### 5. Расширенные системные факты (Hardware & System):

- **`ansible_mounts`** — список словарей с информацией о примонтированных дисках (путь, файловая система, свободное место).
- **`ansible_interfaces`** — список всех сетевых интерфейсов, доступных на хосте.
- **`ansible_env`** — словарь переменных окружения удаленного пользователя (например, `ansible_env.PATH` или `ansible_env.HOME`).
- **`ansible_date_time`** — подробная информация о времени на целевом узле (дата, час, часовой пояс, epoch).
- **`ansible_pkg_mgr`** — определенный Ansible пакетный менеджер для данной ОС (например, `apt`, `yum` или `dnf`).
- **`ansible_virtualization_type`** — тип виртуализации (например, `docker`, `kvm`, `vmware` или `physical`). 

#### 6. Переменные управления соединением:

- **`ansible_connection`** — тип соединения (например, `ssh`, `local` для запуска на своей машине или `docker`).
- **`ansible_python_interpreter`** — путь к интерпретатору Python на удаленном хосте (критично для систем, где Python 3 не является стандартным).
- **`ansible_become_method`** — способ повышения привилегий (по умолчанию `sudo`, но можно сменить на `su` или `pbrun`). 

#### 7. Переменные результата (через `register`):

Когда вы пишете `register: my_result`, Ansible создает объект со следующими стандартными ключами:

- **`my_result.stdout`** — стандартный вывод команды (одной строкой).
- **`my_result.stdout_lines`** — вывод команды, разбитый на список строк (удобно для циклов).
- **`my_result.stderr`** — вывод ошибок.
- **`my_result.rc`** — код возврата (Return Code). `0` — успех, всё остальное — ошибка.
- **`my_result.changed`** — булево значение: изменила ли задача состояние системы.
- **`my_result.failed`** — `true`, если задача завершилась с ошибкой.
- **`my_result.msg`** — текстовое пояснение от Ansible (например, почему модуль упал).

#### 8. Магические переменные результата:

Эти переменные доступны внутри плейбука автоматически для оценки статуса:

- **`ansible_failed_task`** — содержит данные о последней упавшей задаче (доступна в блоках `rescue`).
- **`ansible_failed_result`** — результат упавшей задачи (аналог `register` для упавшей задачи).
- **`ansible_loop_var`** — позволяет переименовать переменную цикла `item` (полезно при вложенных циклах).
- **`ansible_index_var`** — если задано в `loop_control`, хранит текущий индекс итерации (0, 1, 2...).

#### 9. Переменные для поиска и путей (Lookup & Paths):

Часто используются для динамического получения данных:

- **`playbook_dir`** — абсолютный путь к директории, где лежит текущий плейбук.
- **`role_name`** — имя текущей исполняемой роли.
- **`ansible_search_path`** — список путей, где Ansible ищет файлы, шаблоны и роли.
- **`ansible_parent_role_paths`** — пути к ролям, которые вызвали текущую роль (при наследовании).

#### 10. Специальные факты "Local":

- **`ansible_local`** — содержит кастомные факты, которые вы положили в `/etc/ansible/facts.d/*.fact` на удаленном сервере (в формате JSON или INI).

#### 11. Переменные окружения и управления:

- **`ansible_run_tags`** — список тегов, которые были переданы при запуске (флаг `--tags`).
- **`ansible_skip_tags`** — список пропущенных тегов (флаг `--skip-tags`).
- **`ansible_limit`** — значение флага `--limit`, если вы ограничивали выполнение на группу хостов.
