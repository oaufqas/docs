## [[notes/7_configuration/ansible/installation|Ansible]] — это инструмент автоматизации ИТ-инфраструктуры, работающий через SSH *без агентов*. [Документация](https://docs.ansible.com/projects/ansible/latest/index.html)

##### Идемпотентность (Idempotency) - Ansible сам проверяет, соответствует ли текущее состояние описанному. Если да — ничего не делает. Если нет — приводит систему к нужному виду. Можно запускать один и тот же сценарий сто раз — система не сломается и лишний раз ничего не сделает.

###### Основные компоненты: 
**[[inventory|Inventory]]** (список серверов), 
**[[modules|Modules]]** (инструменты Ansible)
**[[playbooks|Playbooks]]** (сценарии YAML), 
**[[roles|Roles]]** (структурированные задачи), 
**[[vault|Vault]]** (шифрование секретов)
**[[jinja2|Jinja2]]** шаблонизатор (динамические файлы). 
Все это управляет серверами, описывая желаемое состояние, а не просто команды

---

#### Консольные команды Ansible строятся по логичному шаблону. Их можно разделить на **Ad-hoc команды** (однострочники) и запуск **Playbooks**.

##### 1. Структура Ad-hoc команд (модульные вызовы):

```bash
ansible <pattern> -m <module> -a "<arguments>" -i <inventory>
```

- **`<pattern>`**: На каких хостах запустить? (`all`, `web_servers`, `webserv1`).
- **`-m <module>`**: Какой модуль использовать? (`ping`, `shell`, `apt`, `copy`).
- **`-a "<arguments>"`**: Аргументы для модуля. Если их несколько, пишутся в кавычках.
- **`-i <inventory>`**: Путь к файлу инвентаря (если не указан, ищется в `/etc/ansible/hosts`).
- _`--become` добавляет права sudo_


#### 2. Структура запуска Playbook:

```bash
ansible-playbook <file.yml> -i <inventory> [options]
```

- **`ansible-playbook`**: Основная утилита.
- **`<file.yml>`**: Путь к твоему плейбуку.
- **`[options]`**: Дополнительные флаги.

**Частые флаги (опции):**

- `-K` (прописная): Спросить пароль для `sudo` (become).
- `-C` (`--check`): **Dry run** — имитация запуска без изменений на сервере.
- `-v`, `-vv`, `-vvv`: Уровни детализации логов (verbosity).
- `-l <pattern>` (`--limit`): Запустить плейбук только на определенном хосте из инвентаря.
- `--tags <tag_name>`: Выполнить только задачи с определенным тегом.


#### 3. Другие полезные утилиты

- **`ansible-inventory -i hosts.ini --list`** — проверить, как Ansible видит твой инвентарь (в формате JSON).
- **`ansible-vault encrypt <file>`** — зашифровать файл с секретами.
- **`ansible-galaxy install <author.role>`** — скачать чужую готовую роль.
- **`ansible-doc <module_name>`** — справочник по любому модулю прямо в консоли (например, `ansible-doc apt`).
---




**Схема работы выглядит так**:

3. Ты запускаешь плейбук (`ansible-playbook my-playbook.yml`).
    
4. Ansible смотрит в **Inventory**, чтобы понять, на какие серверы нацелен этот плейбук.
    
5. Для каждого сервера он по SSH подключается и выполняет описанные в плейбуке задачи.
    
6. Для каждой задачи Ansible использует соответствующий **модуль**. Он передает на удаленный сервер небольшой кусочек кода (модуль), который выполняется там, делает свою работу и возвращает результат в формате JSON.
    
7. После выполнения модуль удаляется с сервера. Никакого мусора не остается.
    
8. Ansible анализирует результат и, если нужно, переходит к следующей задаче.


##### Когда ansible подключается по ssh (security shell) к серверу, он начинает делать сбор фактов системы (архитектура, ip-адреса и тд) и заполняет свои переменные. 

###### Далее идет выполнение задач, выполняются задачи в указанном порядке (но есть и асинхронный вариант). После всех задач, выполняется handler при необходимости.
