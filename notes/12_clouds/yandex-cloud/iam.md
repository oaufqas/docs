#### **IAM (Identity and Access Management)** — это «пропускная система» облака. Она отвечает на два вопроса:

1. **Кто ты?** (Аутентификация: проверка логина/пароля или ключа).
2. **Что тебе можно делать?** (Авторизация: проверка ролей и прав).

**Основные объекты IAM:**

- **Ресурс** (виртуалка, диск, сеть).
- **Субъект** (пользователь или сервисный аккаунт).
- **Роль** (набор прав).

---

### Что такое Service Account

**Service Account (SA)** — это «робот», а обычный **User Account** — это человек.

Когда ты заходишь в облако под своим логином и паролем, ты — **User**. Но когда твоему скрипту или приложению (например, серверу на Python или агенту CI/CD) нужно зайти в облако, чтобы загрузить файл в хранилище, ему тоже нужны права.

Создавать для программы личный логин/пароль человека — плохая практика (небезопасно и неудобно). Поэтому создается **Service Account**.

- **У него нет пароля** (в привычном понимании). Вместо этого используются специальные ключи (API Keys) или временные токены.
- **Он привязывается к ресурсу.** Можно «приклеить» сервисный аккаунт к виртуальной машине, и она автоматически получит все его права.

### Что такое Роли (Roles)

**Роль** — это набор разрешений (Policy). Сами по себе роли ничего не делают, пока их кому-то не назначили.

Как это работает в связке:

1. Ты создаешь **Роль** (например, `S3-ReadOnly` — право только смотреть файлы).
2. Ты создаешь **Service Account** (робота).
3. Ты «вручаешь» эту Роль этому Аккаунту.
4. Теперь любая программа, действующая от имени этого аккаунта, может только смотреть файлы.

Типы ролей (на примере Yandex Cloud / AWS)

- **Примитивные (Basic):**
    - `Viewer` — только смотреть.
    - `Editor` — создавать и удалять ресурсы.
    - `Admin` — полный доступ + управление правами других.
- **Гранулярные (Fine-grained):**
    - Конкретные права на конкретный сервис (например, только перезагрузка виртуальных машин, без права их создавать).

### Как подключаться к [[examples|Terraform]]?

Terraform — это внешняя программа. Чтобы она могла управлять твоим облаком, ей нужно «представиться».

1. Создать в облаке **Сервисный аккаунт** (специально для Terraform).
2. Назначить этому аккаунту **Роль** (обычно `admin` или `editor` на уровне всего облака/каталога).
3. Создать **Ключи** для этого аккаунта. В зависимости от провайдера (AWS или Yandex Cloud) названия чуть разные, но суть одна. Тебе понадобятся **два типа данных**:

А) Идентификаторы (ID)

- **Cloud ID / Folder ID:** Указывают, в какое именно «пространство» (облако или каталог) Terraform должен вносить изменения.
- **Key ID (или Access Key):** Публичный идентификатор твоего ключа (как номер банковской карты).

Б) Секреты (Tokens / Secret Keys)

- **IAM-токен (временный):**
    - _За что отвечает:_ Короткоживущий «билет» (обычно на 12-24 часа).
    - _Минус:_ Его нужно постоянно обновлять вручную или скриптом.
    - `yc iam create-token`
- **Статический ключ (Access Key + Secret Key):**
    - _За что отвечает:_ Постоянный доступ. Access Key — логин, Secret Key — пароль.
    - _Где используется:_ Часто нужен для работы с S3-хранилищем (хранение `terraform.tfstate`).
- **Авторизованный ключ (JSON-файл / Key File):**
    - _За что отвечает:_ Это файл с полными данными сервисного аккаунта.
    - _Как работает:_ В коде Terraform ты просто указываешь путь к этому файлу: `service_account_key_file = "key.json"`.

4. Передать эти ключи в Terraform через файлы конфигурации или переменные окружения. Как это выглядит в коде (Пример для Yandex Cloud) `main.tf`:

```hcl
provider "yandex" {
  token     = "<твой_iam_токен>" # Или путь к JSON-ключу
  cloud_id  = "<id_облака>"
  folder_id = "<id_каталога>"
  zone      = "ru-central1-a"
}
```
