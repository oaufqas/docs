### Основные понятия:

`Context` - разделы в файлах конфигурации, в которых могут использоваться директивы
`Directives` - конкретные опции конфигурации, которые состоят из имени и значения опции

### Архитектура:

- Nginx использует концепцию воркеров (workers)
- В главном процессе nginx создаются дочерние процессы (workers)
- По умолчанию количество воркеров будет равно количеству физических потоков

### Рабочие процессы:

- Важнейший элемент внутренней архитектуры NGINX — это **главный процесс (master process)**. Сразу после запуска NGINX порождает процесс, который управляет всеми остальными потоками и процессами внутри приложения.

- Помимо прочего, главный процесс создает пару процессов для работы с кешем (один отвечает за чтение данных с диска, другой за обновление кеша). Однако наиболее интересными являются **рабочие процессы (worker processes)**. Именно они берут на себя основную нагрузку по обработке входящих подключений и запросов.

- При настройке NGINX в режиме «auto» (это значение по умолчанию) приложение автоматически создает столько рабочих процессов, сколько ядер ЦПУ доступно в системе. Количество рабочих процессов может меняться, если в системе включена технология **Нyper-Threading**.

- **Hyper-Threading** позволяет одному физическому ядру ЦПУ выглядеть для операционной системы как два логических ядра. За счет этого ОС способна более эффективно распределять задачи между потоками одного физического ядра.

- Если в машине два физических ядра и Hyper-Threading включен, операционная система «увидит» четыре логических ядра. Решение о том, использовать ли Hyper-Threading, принимается, исходя из требуемой производительности.

- Соотношение «один рабочий процесс одно ядро» (1:1) — это всего лишь подход к настройке, который предполагает, что операции в основном зависят от вычислительных ресурсов (CPU-bound). Но если большинство запросов «упираются» в операции ввода-вывода (I/O-bound), например, активно читают данные из базы или взаимодействуют с другим удаленным сервисом, то процесс тратит основную часть времени на ожидание ответа, а не на загрузку процессора. Кроме того, многое зависит от используемого протокола.



### Управление соединениями: 

Каждый рабочий процесс «слушает» свой порт (чаще всего 80 для HTTP). При создании «слушающего» сокета в ядре операционной системы формируются две очереди: 1. **SYN-очередь** (SYN queue) 2. **АССЕРТ-очередь** (Accept queue) 

#### Обе очереди контролируются ядром ОС:

- Когда клиент пытается соединиться с портом 80, он начинает трехстороннее рукопожатие TCP (SYN - SYN-ACK ACK).

- Сначала пакет SYN попадает в SYN-очередь; после ответа сервера (SYN-ACK) и подтверждения от клиента (АСК) соединение становится установленным.

- Завершенное ТСР-соединение перемещается в АССЕРТ-очередь, из которой **рабочие процессы** NGINX «забирают» соединения на обработку.

- В момент принятия соединения создается файловый дескриптор, и теперь рабочий процесс отвечает за чтение и запись данных по этому соединению. 

Существует несколько способов распределения новых соединений между рабочими процессами. Иногда все процессы слушают один и тот же порт, а распределением соединений занимается само ядро. В другом варианте главный процесс первым принимает соединения и затем перенаправляет их рабочему процессу. 

В любом случае один рабочий процесс может обрабатывать множество подключений одновременно.


### Обработка запросов

- Рассмотрим пример: у нас есть два клиента, которые подключены к NGINX-балансировщику. Первый клиент «сидит» на рабочем процессе No3 (Worker 3), а второй на рабочем процессе No4 (Worker 4).

- Допустим, клиент, работающий с Worker 3, запрашивает HTML-страницу. Что происходит под капотом?

1. Клиент отправляет HTTP-запрос в виде байтового потока через ТСР-соединение.
2. Поток данных доходит до буфера в ядре операционной системы.
3. Worker 3 вызывает метод чтения (read) на своем соединении.
4. Байты копируются из буфера ядра в память рабочего процесса.
5. Worker 3 парсит полученные данные и распознает, что это HTTP-запрос.
6. Дальше он обрабатывает запрос (например, читает нужный файл с диска).
7. После формирования ответа Worker 3 отправляет HTTP-ответ обратно по соединению.
8. Байты ответа попадают в буфер отправки (send buffer).
9. Ядро пересылает их клиенту.
