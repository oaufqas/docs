Контейнер (Docker, LXC или systemd-nspawn, containerd) - это самый обычный процесс в Linux, изолированный по ресурсам (CPU, RAM...), ограниченный по сети, корню. Технологии ограничения процессов будут рассмотрены подробно ниже:
- namespaces - ограничение видимости среди процессов
- chroot - ограничение файлового пространства процессов
- cgroups - ограничение ресурсов для процессов


## Namespaces (Пространства имен)

Это технология ядра Linux, которая изолирует системные ресурсы процессов друг от друга. Ядро Linux умеет изолировать 8 разных категорий ресурсов. Когда процесс перемещается в контекст контейнера, для него создается свой уникальный набор этих пространств:

1) **PID Namespaces (Процессы)**

Внутри контейнера, скрипт любого приложения видит себя как `PID 1` (будто он главный init-процесс системы). Но если зайти на реальный хост и ввести `ps aux`, этот же самый процесс будет виден под своим настоящим системным PID. Механизм namespaces позволяет создавать отдельное ответвление дерева процессов с собственным PID 1. Процесс, который создаёт такое ответвление, являются частью основного дерева, но его дочерний процесс уже будет корневым в новом дереве. Процессы внутри PID Namespace не могут видеть и убивать процессы снаружи. 

Для создания новых пространств имён PID используется системный вызов **clone()** c флагом `CLONE_NEWPID`. С помощью этого флага можно запускать новый процесс в новом пространстве имён и в новом дереве.

![[Pasted image 20260620150931.png|437]]

2) **NET Namespace (Сеть)**

Изолирует сетевые интерфейсы, таблицы маршрутизации, порты и правила iptables. Каждый контейнер имеет свой собственный виртуальный сетевой интерфейс `lo` и свой собственный изолированный стек портов. Именно поэтому вы можете запустить на одном сервере 10 разных контейнеров, и внутри каждого из них база данных будет слушать стандартный порт `3306` — они не будут конфликтовать, так как их сетевые пространства изолированы.

Сетевые пространства имён можно создавать с помощью системного вызова `clone()` с флагом `CLONE_NEWNET`. Также это можно сделать с помощью iproute2:  `ip netns add netns1`. Как можно помещать процессы в новое сетевое пространство имён?  
  
Во-первых, процесс, создавший новое пространство имён, может порождать другие процессы, и каждый из этих процессов будет наследовать сетевое пространство имён родителя.  
  
Во-вторых, в ядре имеется специальный системный вызов — `setns()`. С его помощью можно поместить вызывающий процесс или тред в нужное пространство имён. Для этого требуется файловый дескриптор, который на это пространство имён ссылается. Он хранится в файле `/proc/<PID процесса>/ns/net`. Открыв этот файл, мы можем передать файловый дескриптор функции `setns()`.  
  
Можно пойти и другим путём. При создании нового пространства имён с помощью команды ip создаётся файл в директории /var/run/netns/ (см. в выводе трассировки выше). Чтобы получить файловый дескриптор, достаточно просто открыть этот файл.  
  
Сетевое пространство имён нельзя удалить при помощи какого-либо системного вызова. Оно будет существовать, пока его использует хотя бы один процесс.


3) **MNT Namespace (Точки монтирования / Файловая система)**

Изолирует список примонтированных дисков и папок. Процесс видит только ту файловую систему, которую ему выделили (обычно через системный вызов `chroot` или `pivot_root`). Если контейнер примонтирует себе папку внутри своего пространства, основная хост-система этого монтирования не увидит.

![[Pasted image 20260620170432.png|425]]

Для изоляции файловой системы используется системный вызов `clone()` c флагом `CLONE_NEWNS` (newns а не newmnt потому что изначально изоляция рассматривалась только как файловая `chroot`, так и оставили):  
  
```bash
clone(child_fn, child_stack+1048576, CLONE_NEWPID | CLONE_NEWNET | CLONE_NEWNS | SIGCHLD, NULL)
```
  
Сначала дочерний процесс «видит» те же точки монтирования, что и родительский. Как только дочерний процесс перенесён в отдельное пространство имён, к нему можно примонтировать любую файловую системы, и это никак не затронет ни родительский процесс, ни другие пространства имён.


4) **IPC Namespace (Межпроцессное взаимодействие)**

Изолирует очереди сообщений, разделяемую память (Shared Memory) и семафоры POSIX/SysV. Процессы из разных контейнеров не могут использовать общую память для скрытой передачи данных друг другу, даже если они запущены на одном физическом процессоре. Вызов `clone()` с флагом `NEWIPC`.

5) **UTS Namespace (Имена хоста)**

Изолирует имя компьютера (`hostname`) и доменное имя (`domainname`). Программа внутри контейнера может вызвать команду `hostname myserver` и изменить имя хоста для себя, но для реального физического сервера имя останется прежним. Вызов `clone()` с флагом `NEWUTS`.

6) **UID (USER) Namespace (Пользователи и права)**

Изолирует идентификаторы пользователей и групп (UID/GID). Это важнейший элемент безопасности. Внутри контейнера вы можете быть полноценным пользователем `root` (UID 0) и делать всё что угодно. Но на реальном хосте этот же самый процесс будет выполняться с правами обычного непривилегированного пользователя (например, UID 10005). Если злоумышленник взломает ваш контейнер и выйдет из него, у него не будет прав `root` на вашем реальном сервере. Вызов `clone()` с флагом `NEWUSER`.

7) **CGROUP Namespace (Контрольные группы)**

Изолирует видимость самих контрольных групп. Процесс внутри контейнера не может увидеть, какие лимиты по CPU и RAM на него наложены со стороны хоста.

8) **TIME Namespace (Системное время)**

Появился относительно недавно. Позволяет процессам внутри контейнера иметь свое собственное системное время и аптайм, отличные от основных часов сервера.


#### Главные системные вызовы для управления

Вы можете из программы создать пространства имен вручную с помощью трех системных вызовов ядра:

1. **`clone()`** — заменяет стандартный `fork()` . Он создает новый дочерний процесс, но с флагами изоляции (например, `CLONE_NEWPID | CLONE_NEWNET`).
2. **`unshare()`** — позволяет текущему уже запущенному процессу «отпочковаться» от родительских ресурсов и создать для самого себя новое чистое пространство имен.
3. **`setns()`** — позволяет процессу принудительно «войти» в уже существующее чужое пространство имен (именно так работает команда `docker exec` — она берет новый процесс вашей консоли и подсаживает его в Namespaces работающего контейнера).


Для чего нужны ns, если процессы и так изолированы в своей виртуальной памяти? Благодаря **MMU (Memory Management Unit)** процессора, каждый процесс заперт в своем изолированном куске виртуальной памяти. Процесс Nginx физически не может прочитать или изменить память процесса MySQL.

Но виртуальная память защищает только **данные в RAM**. Она никак не защищает **глобальные системные сущности операционной системы**. Если представить, что Namespaces нет, но есть виртуальная память. В этом случае любой процесс в системе:

1. **Видит все остальные процессы:** Он может сделать системный вызов `kill(1420, SIGKILL)` и убить чужую программу, если запущен от того же пользователя.
2. **Видит всю сеть:** Он видит все сетевые карты хоста. Если один процесс занял порт `80` на сетевой карте, второй процесс уже не сможет запуститься на этом же порту.
3. **Видит то же имя хоста:** Если процесс изменит имя компьютера через `sethostname()`, оно изменится для всей ОС.

**Смысл Namespaces:** Виртуальная память изолирует _адреса в RAM_, а Namespaces изолируют _инфраструктуру ОС_ (дерево PID, сетевые порты, имена и файлы). Без Namespaces было бы физически невозможно запустить два изолированных контейнера на одном сервере.

#### Взаимодействие с Namespaces в консоли

Можно посмотреть пространства имен каждого процесса через виртуальную фс `/proc`:

```bash
# lrwxrwxrwx 1 rvlx rvlx 0 Jun 19 17:32 ipc -> 'ipc:[4026531839]'
# lrwxrwxrwx 1 rvlx rvlx 0 Jun 19 17:32 mnt -> 'mnt:[4026531832]'
# lrwxrwxrwx 1 rvlx rvlx 0 Jun 19 17:32 net -> 'net:[4026531833]'
# lrwxrwxrwx 1 rvlx rvlx 0 Jun 19 17:32 pid -> 'pid:[4026531836]'

```

Цифры в квадратных скобках (например, `4026531836`) — это **уникальные системные идентификаторы (индексы) пространств имен** внутри ядра Linux

Посмотреть все namespaces на сервере можно командой `lsns`, по умолчанию (без контейнеров), их будет базовых 8 (на каждый тип по одному), но при создании контейнеров это число будет кратно увеличиваться. 1 контейнер +~ 8ns.


Создать недо-контейнер без docker можно используя только ns и отдельную директорию с оболочкой и busybox

```bash
sudo unshare --mount --uts --ipc --net --pid --fork -R <directory> --mount-proc /bin/sh
# sudo — запускает команду с правами суперпользователя хоста, так как создание пространств имен (особенно сетей и точек монтирования) требует максимальных привилегий ядра.

--mount # создает новый MNT Namespace (изолирует таблицу смонтированных дисков и папок).
--uts # создает UTS Namespace (изолирует имя компьютера / hostname).
--ipc # создает IPC Namespace (изолирует общую память и семафоры).
--net # создает NET Namespace (выдает пустой сетевой стек без карт и портов).
--pid # создает PID Namespace (создает новое изолированное дерево процессов).

--fork (-f) # заставляет утилиту сделать вызов fork(), чтобы дочерний процесс гарантированно стал PID 1 внутри нового пространства имен процессов.

-R <directory> # это встроенный внутрь unshare вызов chroot. Он делает указанную директорию новым корнем (/) для контейнера, полностью скрывая от него файлы хоста

--mount-proc # скомандует ядру принудительно смонтировать чистую виртуальную файловую систему процессов /proc внутрь вашей папки <directory>/proc. Благодаря этому ps внутри контейнера будет видеть только свои процессы.

/bin/sh # командная оболочка, которая запустится как PID 1 внутри этого изолированного мира.
```

 Разбор контейнера через ns с помощью ps:

```
USER   PID %CPU %MEM    VSZ   RSS TTY      STAT START   TIME COMMAND
rvlx    3416  0.2  0.1  11152  5568 pts/0    Ss   06:46   0:05 -zsh
root    3914  0.0  0.1  21920  7984 pts/0    S+   07:18   0:00   \_ sudo unshare --mount --uts --ipc --net --pid -f -R test --mount-proc busybox sh
																\
root   3916  0.0  0.0  21920  2680 pts/1    Ss   07:18   0:00    \_ sudo unshare --mount --uts --ipc --net --pid -f -R test --mount-proc busybox sh
																\
root   3917  0.0  0.0   5836  2060 pts/1    S    07:18   0:00    \_ unshare --mount --uts --ipc --net --pid -f -R test --mount-proc busybox sh
																\
root  3918  0.0  0.0   1312   416 pts/1    S+   07:18   0:00     \_ busybox sh
```

1. Основная оболочка

```text
3416 ... -zsh
```

- **За что отвечает:** Это ваша личная интерактивная консоль на хосте (ноутбуке), в которой вы сидите и руками пишете команды. Символ `-` перед `zsh` означает, что это оболочка входа (login shell). Она является **родителем всего дерева**.

2. Запрос прав суперпользователя

```text
3914 ... \_ sudo unshare --mount ...
```

- **За что отвечает:** Оболочка `zsh` сделала `fork()` и запустила утилиту `sudo`. Этот процесс отвечает за проверку ваших прав и переключение контекста безопасности с обычного пользователя `rvlx` на суперпользователя `root`. Именно этот процесс считает строку вашей команды и подготовит её к запуску.

3. Настоящий исполнитель утилиты (Клон `sudo`)

```text
3916 ...     \_ sudo unshare --mount ...
```

- **За что отвечает:** Утилита `sudo` устроена так, что она раздваивается. Процесс `3914` остается ждать завершения работы, а процесс `3916` — это его «боевой» клон, который от имени истинного `root` вызывает системную утилиту `unshare`. Обратите внимание: у него изменился терминал на **`pts/1`**. Ядро выделило ему новый псевдо-терминал для будущей изоляции.

4. Рождение пространств имен (Namespaces)

```text
3917 ...         \_ unshare --mount --uts ...
```

- **За что отвечает:** Процесс `3916` сделал вызов `exec()` и превратился в чистую утилиту `unshare`. Этот процесс `3917` — **архитектор вашего контейнера**. Он выполняет системные вызовы `unshare()` для создания новых NET, PID, MNT пространств имен, делает `chroot` в папку `test` и монтирует туда новый `/proc`.

5. Финальный житель контейнера (Ваша цель)

```text
3918 ...             \_ busybox sh
```

- **За что отвечает:** Из-за флага `-f` (`--fork`) процесс `3917` сделал финальный `fork()`, родив процесс `3918`. Этот процесс провалился внутрь всех созданных пространств имен и сделал `exec()` в оболочку `busybox sh`.
- **Важный нюанс:** Для хоста он виден под системным PID `3918`, но если вы зайдете внутрь этого `busybox` и введете `echo $$`, он честно ответит, что его **PID равен 1**, потому что в своем изолированном PID Namespace он — главный и единственный родитель.


Столбцы 6 и 7 (STAT) — Статусы процессов в планировщике ядра

- **`Ss` (у процессов 3416 и 3196):**
    - `S` (Interruptible sleep) — процессы «спят», ожидая событий (ввода клавиатуры или ответов от дочерних процессов) .
    - `s` (lowercase s) — этот процесс является **лидером сессии** (главным в своем терминале).
- **`S+` (у процесса 3914 и 3918):**
    - `+` — означает, что этот процесс находится в **Foreground-группе** (на переднем плане) своего терминала. Процесс `3914` ждет на переднем плане вашего хоста, а `3918` (`busybox`) ждет ввода ваших команд на переднем плане внутри контейнера.
- **`S` (у процесса 3917):**
    - У него нет плюсика `+`. Он выполнил свою работу по созданию контейнера, запустил `busybox` и ушел в фон планировщика ядра, просто удерживая структуры пространств имен в памяти до тех пор, пока `busybox` не завершит работу.

## Chroot (Change root)

Один из старейших и самых первых системных вызовов и способов создания контейнерной виртуализации процессов, его можно считать концептуальным прототипом MNT Namespace.

Название chroot представляет собой сокращение от change root, что дословно переводится как «изменить корень». С помощью системного вызова `chroot()` и соответствующей команды можно изменить корневой каталог. Программе, запущенной с изменённым корневым каталогом, будут доступны только файлы, находящиеся в этом каталоге. 

По умолчанию у любого процесса в Linux корнем файловой системы является папка **`/`**.

Когда процесс делает системный вызов `chroot("/home/joe")`, ядро Linux выполняет простую подмену в паспорте этого процесса: оно говорит, что отныне директория `/home/joe` является для этого процесса **абсолютным и единственным корнем (`/`)**.

Если процесс внутри этой «тюрьмы» выполнит команду `cd /`, он попадет не в настоящий корень вашего SSD, а останется внутри папки `joe`. Выйти «выше» этой папки обычными командами вроде `cd ..` он физически не может

![[Pasted image 20260620142717.png|495]]

Обратим внимание на важный момент: с помощью chroot мы создаём новую, изолированную файловую систему, которая не имеет никакого доступа к текущей, поэтому для того, чтобы просто запустить chroot, нужна командная оболочка и ее библиотеки:  

```bash
ldd /bin/bash	
	linux-vdso.so.1 =>  (0x00007fffd08fa000)
	libtinfo.so.5 => /lib/x86_64-linux-gnu/libtinfo.so.5 (0x00007f30289b2000)
	libdl.so.2 => /lib/x86_64-linux-gnu/libdl.so.2 (0x00007f30287ae000)
	libc.so.6 => /lib/x86_64-linux-gnu/libc.so.6 (0x00007f30283e8000)	
	/lib64/ld-linux-x86-64.so.2 (0x00007f3028be6000)
```

После этого выполним следующие команды:  
  
```bash
mkdir test test/usr test/bin
cp --parents /usr/bin/bash ./test
cp --parents /usr/lib/x86_64-linux-gnu/libtinfo.so.6 ./test
cp --parents /usr/lib/x86_64-linux-gnu/libc.so.6 ./test
cp --parents /lib64/ld-linux-x86-64.so.2 ./test
wget https://busybox.net/downloads/binaries/1.35.0-x86_64-linux-musl/busybox
mv busybox test/bin
chmod +x test/bin/busybox

chroot test
/bin/busybox --install -s

bash-4.3# 
```

Мы создали самую банальную файловую изоляцию, но у chroot есть серьезный недостаток, все необходимые файлы нужно дублировать. Есть у chroot и ряд недостатков с точки зрения безопасности: работает только на уровне дерева каталогов, нет изоляции процессов, сетей и других пространств. Если chroot запущен с правами настоящего рута, из него можно сбежать из за бага логики старого UNIX. Поэтому технология chroot сейчас заменена Namespaces

## Cgroups (Control Groups)

Это второй ключевой механизм ядра Linux, обеспечивающий работу контейнеризации и стабильность системы.

Если **Namespaces** создают иллюзию изоляции и отвечают на вопрос «Что процесс может _видеть_?», то **Cgroups** распределяют физические ресурсы компьютера и отвечают на вопрос «Сколько ресурсов процесс может _съесть_?». Благодаря cgroups один взбесившийся процесс с бесконечным циклом или утечкой памяти не может обрушить весь сервер.


Изначальный замысел был гораздо скромнее: усовершенствовать механизм [cpuset](https://access.redhat.com/documentation/en-US/Red_Hat_Enterprise_Linux/6/html/Resource_Management_Guide/sec-cpuset.html), предназначенный для распределения процессорного времени и памяти между задачами. Но со временем всё переросло в более масштабный проект. В конце 2007 года название process containers было заменено на control groups. Это было сделано, чтобы избежать разночтений в толковании термина «контейнер» (в то время уже активно развивался проект OpenVZ, и слово «контейнер» стало употребляться в новом, современном значении). В 2008 году механизм cgroups был официально добавлен в ядро Linux (версия 2.6.24).

#### Архитектура  

В ядре Linux cgroups организованы в виде **иерархического дерева**, очень похожего на обычную файловую систему. Каждая группа процессов может иметь дочерние группы, которые наследуют ограничения родителей.

Папка `/sys/fs/cgroup` — это не жесткий диск, а **виртуальный пульт управления ядром** (интерфейс файловой системы).

- **Корень — это весь ваш компьютер:** Файлы, которые вы видите прямо в корне `/sys/fs/cgroup`, настраивают лимиты и показывают статистику для **абсолютно всех процессов** в вашей системе вместе взятых.
- `cgroup.subtree_control` — файл активации категорий лимитов, в некоторых системах инициализирован и настроен с помощью systemd при запуске системы, в некоторых нужно настраивать вручную для создания своих cgroups: `echo "+cpu +memory" | sudo tee /sys/fs/cgroup/cgroup.subtree_control`.
- **Папки — это подгруппы:** Обратите внимание на папки `system.slice`, `user.slice` и `init.scope`. Их создала система `systemd`. `user.slice` — это группа для всех программ пользователя (ваш браузер, Zsh), а `system.slice` — для системных служб (Nginx, Docker). Большинство директорий - служебные, особо заострять внимание на них не стоит.
- Директории, созданные пользователем в корне, это и есть cgroups, в них автоматически появляются служебные файлы, такие как cpu.max, memory.max, где и устанавливаются лимиты.

Механизм cgroups состоит из двух составных частей: ядра (cgroup core) и подсистем: каждый конкретный тип физического ресурса отвечает свой специализированный программный модуль ядра, который называется **контроллером (или подсистемой)**:

- `cpu` — обеспечивает доступ процессов в рамках контрольной группы к тактам CPU;
- `memory` — управляет выделением памяти для групп процессов;
- **`io`** (в старых версиях `blkio`) — устанавливает лимиты на чтение и запись с блочных устройств;
- `pids` — используется для ограничения количества процессов в рамках контрольной группы.
- `cpuacc` — генерирует отчёты об использовании ресурсов процессора;
- `cpuset` — распределяет задачи в рамках контрольной группы между процессорными ядрами;
- `devices` — разрешает или блокирует доступ к устройствам;
- `freezer` — приостанавливает и возобновляет выполнение задач в рамках контрольной группы
- `hugetlb` — активирует поддержку больших страниц памяти для контрольных групп;
- `net_cls` — помечает сетевые пакеты специальным тэгом, что позволяет идентифицировать пакеты, порождаемые определённой задачей в рамках контрольной группы;
- `netprio` — используется для динамической установки приоритетов по трафику;

**У каждой подсистемы есть свой корень, второго быть не может**

Какие в ядре поддерживаются группы:

```bash
cat /proc/cgroups
lssubsys -a
lscgroup
cat /sys/fs/cgroup/cgroup.controllers
```

Посмотреть группы текущего процесса:

```bash
cat /proc/self/cgroup
```

Вывести корень иерархии cgroups:  

```bash
ls -l /sys/fs/cgroup
```

Посмотреть все cgroups на сервере

```bash
systemd-cgls
```

#### Смена поколений: Cgroups v1 vs Cgroups v2

![[Pasted image 20260621142824.png]]

Технология cgroups пережила глобальную архитектурную революцию.

>**Cgroups v1 (Старая версия)**

В первой версии каждый контроллер (`cpu`, `memory`) жил в своем собственном отдельном дереве процессов. Процесс мог находиться в группе `A` для процессора, но в группе `B` для памяти. Это приводило к дикой путанице в коде ядра, порождало баги и не позволяло нормально синхронизировать, например, кэш диска и лимиты памяти.

Создание cgroup в старой версии:

```bash
cgcreate -t kkv:kkv -g memory,cpu:/mycgroup

-t uid:gid -# пользователи, получающие права на перемещение заданий в группу
-a uid:gid -# пользователи, получающие права на управление параметрами группы
-g -# список подсистем(контроллеров):путь

cgdelete -g memory,cpu:/mycgroup
```

При создании группы, появляется файл `tasks`, в который можно записывать PID'ы процессов, подконтрольных этой cgroup. `echo 123 >> ...tasks`. Вручную это делать неудобно, поэтому есть `sgclassify`:

```bash
cgclassify -g cpu:/mycgroup 66433 3662
-g -# список подсистем(контроллеров):путь PID [PID PID ...] 

echo 6433 > /sys/fs/cgroup/cpu/mycgroup/tasks
```

При удалении группы, входящие в нее задачи перемещаются в родительскую группу.


>**Cgroups v2 (Современный стандарт, с ядра 4.5+)**

Разработчики полностью переписали механизм, создав **Единую унифицированную иерархию (Unified Hierarchy)**. Теперь в системе есть только одно дерево процессов. Если контейнер попадает в группу `/docker/container_1`, то все контроллеры (`cpu`, `memory`, `io`) применяются к этой единой папке одновременно. Это сделало лимитирование ресурсов невероятно точным и предсказуемым. 

Изначально все процессы находятся в одной корневой группе, без лимитов (`/sys/fs/cgroup/cgroup.procs`), и могут существовать одновременно только в одной группе, при добавлении в другую, с корневой они автоматически удаляются. Также после удаления cgroup, все процессы возвращаются в корневую.

Создание cgroup в новой версии:

1) Чтобы создать cgroup, нужно создать директорию в `/sys/fs/cgroup/`
2) Активировать нужные лимиты через родительский `/sys/fs/cgroup/cgroup.subtree_control` (они могут быть активированы при загрузке системы инит процессом `systemd`, автоматически), команда для активации:

```bash
echo "+cpu +memory +io +pids" | sudo tee /sys/fs/cgroup/cgroup.subtree_control
```

3) После создания директории и настройки файла, появится новая cgroup (в директории появится много служебных файлов). В своей группе, можно создать еще 1 группу, и дать ей свои лимиты, получается иерархия группю. Важные файлы для конфигурации:

- `cgroup.tasks` - Файл с PID'ами процессов, которые относятся к этой cgroup.
- `cpu.max` - содержит два значения через пробел: `квота период` (по умолчанию `max 100000`). Период измеряется в микросекундах и обычно равен 100 000 мкс (100 миллисекунд). Если мы хотим выделить процессам в группе ровно 1 ядро CPU, мы пишем: `100000 100000`. Если хотим выделить **половину ядра (50%)**, мы пишем `50000 100000`:

```bash
echo "50000 100000" | sudo tee /sys/fs/cgroup/test/cpu.max

# Команда для загрузки процессора
while true; do true; done &
```

- `memory.max` - по умолчанию содержит слово `max` (без ограничений). Запишем туда лимит, например, в 100 Мегабайт (значение указывается строго в байтах: 100 × 1024 × 1024 = 104857600):

```bash
echo "104857600" | sudo tee /sys/fs/cgroup/test/memory.max

# Команда для загрузки памяти
data="A"; while true; do data="$data$data"; done
```

 - `cgroup.subtree_control` - конфигурационный файл, чтобы добавлять типы контроллеров ограничений в дочерние группы.

Чтобы удалить cgroup, нужно удалить каталог группы в директории `/sys/fs/cgroup` с помощью `sudo rmdir`
