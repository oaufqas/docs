## VFS (Virtual File System)

**VFS (Virtual File System / Виртуальная файловая система)** — это еще один абстрактный слой в ядре Linux, который находится между программами и конкретными файловыми системами (Ext4, NTFS, Btrfs или сетевыми NFS). Именно благодаря VFS, программам без разницы, на каком диске или флешке лежит файл. Код всегда вызывает одну стандартную функцию `open()`, а VFS сама разбирается, как эту команду выполнить для конкретной файловой системы.

Без VFS разработчикам программ пришлось бы писать разный код для работы с разными дисками. Программа бы думала: _«Так, этот файл лежит на флешке FAT32, значит мне нужно вызвать функцию `read_fat32()`. А этот лог пишется на диск Ext4, вызываю `read_ext4()`»_.

VFS берет эту головную боль на себя. Она предоставляет программам **единый интерфейс** (тот самый стандарт POSIX). Программа делает обычный вызов `read(fd)`, а VFS на лету превращает его в специфические команды для конкретного железа и файловой системы.


#### Главные объекты (структуры данных) в VFS

Мир VFS держится на четырех основных объектных типах.

1. **`superblock` (Суперблок)** — Описывает конкретную подключенную файловую систему (ее размер, тип, статус). Это «главный управляющий» диска.
2. **`inode` (Индексный дескриптор)** — Описывает конкретный файл (его размер, права, владельца). В VFS существует структура `struct inode`, которая является универсальной абстракцией для паспорта файла, независимо от того, где он лежит.
3. **`dentry` (Directory Entry / Элемент директории)** — Связывает имя файла с его Inode. Именно `dentry` склеивает пути (например, `/home/user/file.txt`) в цепочку. Ядро кэширует их в оперативной памяти (`dentry cache`), чтобы мгновенно находить файлы по имени, не опрашивая медленный жесткий диск при каждом чихе.
4. **`file` (Открытый файл)** — Описывает связь между процессом и открытым файлом. Это та самая строка во **Втором слое** (Системной таблице открытых файлов), которая хранит File Offset (смещение).


Поскольку VFS — это чистая абстракция, ей все равно, откуда брать данные. Это позволяет Linux создавать «ненастоящие» файловые системы, с которыми пользователь работает как с обычными папками, ниже представлены псевдо-фс, которых не существует на жестком диске, это чистая абстракция VFS в оперативной памяти:

- **`/proc` (procfs)** — Виртуальная файловая система, которая живет только в оперативной памяти ядра. Когда вы делаете `cat /proc/sys/fs/file-nr`, драйвер `procfs` перехватывает операцию `read()`, находит в оперативной памяти структуру `task_struct` для процесса `1234`, вытаскивает оттуда строку запуска и отдаёт её вам в текстовом виде.
- **`/sys` (sysfs)** — Еще одна виртуальная папка для управления «железом» (например, через нее можно поменять яркость экрана или скорость кулера, просто записав цифру в текстовый файл). Драйвер `sysfs` перехватывает операцию `write()`, берет вашу цифру `0` и мгновенно передает её драйверу видеокарты, который физически понижает напряжение на подсветке экрана.
- `tmpfs` — **(быстрый диск в ОЗУ):** Это псевдо-ФС, которая выглядит как папка, но данные пишутся напрямую в оперативную память. Всё, что там лежит, сотрется при перезагрузке, но скорость работы с ней феноменальная (используется для `/tmp` и кэшей).
- **Сетевые папки (NFS / Samba)** — Файлы физически лежат на сервере в другой стране, но благодаря VFS вы видите их в своем файловом менеджере Linux как обычную локальную папку.

Когда вы запрашиваете файл из `/proc` или `/sys`, VFS не делает запросов к контроллеру диска. Вместо этого она перенаправляет ваш запрос **функциям обратного вызова (callbacks)** внутри ядра.


`Программа (Код)`  
⬇️ _вызывает read(3)_  
`Локальная таблица дескрипторов процесса (Слой 1)`  
⬇️ _ведет на..._  
`Системная таблица открытых файлов VFS (Слой 2 / struct file)`  
⬇️ _знает смещение и ведет на..._  
`Универсальный VFS Inode (Слой 3 / struct inode)`  
⬇️ _вызывает функцию конкретной ФС (например, ext4_read_block)_  
`Драйвер Ext4 / Диспетчер дисков`  
⬇️ _считывает физические байты_  
`Жесткий диск (SSD / Флешка)`


#### Команда `mount` - главный инструмент управления VFS

Она связывает физическое устройство или виртуальный диск с конкретной директорией в системе (точкой монтирования)

Чтобы операционная система увидела подключенный диск или флешку, нужно выполнить два простых шага:


```bash
# 1. Узнать имя устройства в системе
lsblk

# 2. Примонтировать раздел в нужную папку

sudo mkdir /mnt/my_flash
sudo mount /dev/sdb1 /mnt/my_flash

# Чтобы размонтировать, используется команда umount
sudo umount /mnt/my_flash
```

Когда вы выполняете команду `sudo mount /dev/sdb1 /mnt/my_flash`, вы не указываете тип файловой системы. Ядро Linux находит нужный драйвер автоматически.

Происходит это по следующему алгоритму:

1. **Проверка суперблока:** Утилита `mount` обращается к специальной системной библиотеке `libblkid`. Она считывает первые несколько килобайт диска (`/dev/sdb1`), где записаны метаданные (сигнатуры) файловой системы (суперблок).
2. **Определение типа:** По этим сигнатурам система мгновенно понимает что там за файловая система.
3. **Загрузка модуля:** Ядро проверяет, загружен ли в оперативную память драйвер для `ext4`. Если нет, ядро динамически загружает нужный модуль (драйвер) из папки `/lib/modules/`
4. **Принудительный выбор:** Вы можете заставить систему использовать конкретный драйвер вручную с помощью флага `-t` (type):

```bash
sudo mount -t ext4 /dev/sdb1 /mnt/my_flash
```

Можно посмотреть список поддерживаемых драйверов в виртуальном файле:

```bash
cat /proc/filesystems
```

_Слово `nodev` перед названием означает, что драйвер работает без физического устройства (это псевдо-ФС)_

**Чтобы посмотреть, какой драйвер отвечает за какие файлы, нужно использовать команду `mount` без параметров, это покажет полную карту монтирования VFS**:

`[ЧТО] on [КУДА] type [КАКОЙ ДРАЙВЕР] ([ПАРАМЕТРЫ ДОСТУПА])`

#### Монтирование

Образ диска (`.img` или `.iso`) — это просто один большой обычный файл, внутри которого записаны побайтовые копии секторов настоящего диска. Благодаря VFS, этот файл можно примонтировать как полноценный жесткий диск.

Шаг 1: Создаем пустой файл под образ (например, на 1 ГБ)

Используем утилиту `dd` (побайтовое копирование нулей из виртуального генератора `/dev/zero`):

```bash
sudo dd if=/dev/zero of=my_disk.img bs=1M count=1024
```

Шаг 2: Форматируем этот файл в нужную файловую систему

Мы заставляем утилиту форматирования думать, что файл `my_disk.img` — это диск:

```bash
sudo mkfs.ext4 my_disk.img
```

Шаг 3: Монтируем образ диска

Чтобы VFS могла работать с файлом как с блочным устройством, `mount` автоматически использует скрытый механизм **Loop-устройств** (виртуальных петель). Файл виртуально превращается в `/dev/loop0`:

```bash
sudo mkdir /mnt/iso_disk
sudo mount -o loop my_disk.img /mnt/iso_disk
```

_Всё! Теперь вы можете зайти в `/mnt/iso_disk`, создавать там папки, копировать файлы. Они будут физически записываться внутрь одного единственного файла `my_disk.img`._

1. Файл `/etc/fstab` (Автомонтирование)

Вы уже знаете, как монтировать диски вручную через `mount`. Но если компьютер перезагрузится, все ваши точки монтирования исчезнут.  
Чтобы диски подключались автоматически при старте Linux, их прописывают в конфигурационный файл `/etc/fstab` (File System Table). Каждая строка в нем указывает VFS: _«При старте найди диск с таким-то UUID (уникальным ID), примонтируй его в такую-то папку, используй драйвер ext4 и такие-то права»_.

2. Механизм Журналирования (Защита от сбоев)

Современные файловые системы (Ext4, NTFS, XFS) называются **журналируемыми**. Внутри них есть специальная скрытая круговая зона — **журнал**.

- **Как это работает:** Когда вы сохраняете файл, ФС сначала быстро записывает план своих действий в журнал (_«Я собираюсь изменить блоки 5 и 6 для файла doc.txt»_), и только потом меняет сами блоки.
- **Зачем нужно:** Если посреди записи у вас внезапно отключится электричество, Linux при следующем включении не будет сканировать весь терабайтный диск часами. Она просто заглянет в журнал, увидит незавершенную операцию и либо безопасно откатит её назад, либо допишет до конца. Данные не повредятся.

3. Кэш страниц (Page Cache) и команда `sync`

Когда программа делает вызов `write()`, Linux ради скорости **не пишет данные на SSD мгновенно**. Она сохраняет их в оперативной памяти (в кэше страниц VFS) и сразу говорит программе: _«Всё готово, я всё записала!»_, а на физический диск сбрасывает их в фоне через несколько секунд. Такие данные в ОЗУ называются _грязными страницами (dirty pages)_.

- **Практический вывод:** Именно поэтому флешки нельзя выдергивать просто так — данные из кэша могут еще не успеть долететь до физических транзисторов. Команда **`sync`** в консоли принудительно заставляет VFS немедленно сбросить весь кэш из памяти на физические диски.
