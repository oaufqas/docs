### Файловые системы 

- **Ext4 (Fourth Extended Filesystem)** - Стандарт по умолчанию для большинства дистрибутивов Linux. Очень стабильная, проверенная временем и быстрая файловая система. Поддерживает огромные файлы и диски, использует журналирование (защиту от сбоев питания).
- **Btrfs (Butter FS)** - Современная «умная» система. Главные фишки — создание мгновенных снимков системы (снапшотов), встроенный RAID (объединение дисков) и сжатие данных «на лету» для экономии места. Используется по умолчанию в Fedora и openSUSE.
- **XFS** - Высокопроизводительная файловая система, которая создавалась для работы с огромными массивами данных. Отлично справляется с очень большими файлами и тяжелыми нагрузками. Используется по умолчанию в Red Hat Enterprise Linux (RHEL).
- **ZFS** - Гибрид файловой системы и менеджера логических дисков. Обладает невероятной надежностью, защитой от повреждения данных, продвинутым сжатием и снапшотами. Идеальна для серверов хранения данных (NAS), но требует много оперативной памяти.
- **NTFS (New Technology Filesystem)** - Основная файловая система для Windows (диск C:, D: и т.д.). Поддерживает права доступа, шифрование, сжатие и большие файлы. Linux умеет читать и писать на NTFS, но «родной» для него она не является.
- **ReFS (Resilient Filesystem)** - Новое поколение от Microsoft для серверов и корпоративных хранилищ. Заточена под максимальную отказоустойчивость, автоматическое исправление поврежденных файлов и работу с огромными объемами данных.
- **APFS (Apple File System)** - Современный стандарт для Mac, оптимизированный специально под SSD и Flash-накопители. Поддерживает быстрое шифрование, мгновенное копирование файлов (клонирование) и динамическое распределение свободного места.
- **HFS+ (Mac OS Extended)** - Старая файловая система Apple. Сейчас используется в основном для совместимости со старыми компьютерами Mac или для резервных копий Time Machine на старых внешних жестких дисках.
- **FAT32** - Очень старый стандарт, который **понимают абсолютно все устройства**: Linux, Windows, macOS, телевизоры, магнитолы и консоли. Главный минус — жесткое ограничение: нельзя записать один файл размером **больше 4 ГБ**.
- **exFAT** - Современная замена FAT32, созданная специально для флешек и внешних дисков. У нее нет ограничений на размер файла (можно спокойно писать фильмы по 50 ГБ), и она официально поддерживается и в Windows, и в macOS, и в Linux.

### Разделы дисков для загрузки ОС

Системный раздел EFI (ESP — EFI System Partition)

Это стандарт для всех современных компьютеров, работающих в режиме **UEFI** (вытеснившем старый BIOS). Он используется в Windows, Linux и macOS.

- **Файловая система:** Строго **FAT32** (изредка FAT16/FAT12). Это требование зашито прямо в материнскую плату. Микропрограмма UEFI умеет читать файлы _только_ из разметки FAT, так как она очень простая и для нее не нужны сложные драйверы.
- **Что там лежит:** Файлы с расширением `.efi` (загрузчики ОС). Например:
    - Для Linux: загрузчик GRUB (`/EFI/BOOT/BOOTX64.EFI` или `/EFI/ubuntu/grubx64.efi`).
    - Для Windows: диспетчер загрузки (`/EFI/Microsoft/Boot/bootmgfw.efi`).
- **Размер:** Обычно небольшой, от 100 до 500 МБ.
- **Особенность:** Он имеет специальный флаг (флаг загрузки) в таблице разделов GPT, поэтому обычные операционные системы скрывают его от пользователя в проводнике, чтобы никто случайно ничего не удалил.

Разделы для старых систем (Legacy BIOS / MBR)

Если компьютер старый или настроен в режиме эмуляции BIOS, там нет файловой системы FAT32 для загрузки. Вместо этого используется классический подход:

- **MBR (Master Boot Record):** Самый первый сектор диска (512 байт). В нем нет никакой файловой системы. Там лежит крошечный кусок кода (boot sector), который указывает материнской плате, куда смотреть дальше.
- **В Windows (Раздел «Зарезервировано системой»):** 
	- Создается автоматически при установке Windows на MBR-диск. Файловая система там **NTFS**, размер около 100–500 МБ. Он скрыт, не имеет буквы диска и содержит базу данных конфигурации загрузки (BCD).
- **В Linux (Раздел `/boot` или BIOS Boot Partition):**
    - Если Linux ставится на GPT-диск, но в режиме BIOS, создается микро-раздел **BIOS Boot Partition** (размером всего 1–2 МБ) вообще без файловой системы — туда GRUB записывает свой базовый код.
    - Также часто создается отдельный раздел `/boot` с файловой системой **Ext4**. На нем лежат файлы ядра Linux (`vmlinuz`) и виртуальный диск инициализации (`initrd`). Он может быть как скрытым, так и видимым в системе, но монтируется отдельно.


### Стандарт POSIX (Portable Operating System Interface for Unix)

**POSIX — это абстрактный стандарт**, а не программа, не код и не операционная система. Это просто огромный текстовый документ (спецификация), в котором написано: _«Если вы хотите, чтобы ваша ОС считалась правильной и совместимой, она должна делать вот это, вот так и отвечать на такие-то команды»_.

Можно сказать это специальный "язык" запросов и ответов к ядру, для выполнения пользовательских программ. 
Программа --> Ядро --> Оборудование --> Необходимые ресурсы для программы

- Каждая ОС имеет свой интерфейс взаимодействия с ядром, а POSIX как стандарт, его можно придерживаться, тогда ОС будет считаться POSIX-совместимой.
- Обеспечивает совместимость UNIX-подобных операционных систем

Стандарты POSIX описывают:
- Системный интерфейс (Вызовы ядра)
- Командный интерпретатор
- Утилиты

POSIX - это работа в пространстве пользователя, системные вызовы работают в пространстве пользователя

## inode (Index Node) - индексный дескриптор

Файловая система опирается на метаданные - данные о данных. В древовидной структуре каталогов хранятся лишь **названия файлов** и **номера inode**

```
newfolder --> filename.txt
              235527628429 (inode)
```

Каждый файл имеет inode - структуру с метаданными:

- Права доступа
- Владелец и группа
- Размер
- Временные метки (atime, ctime, mtime)
- Количество жестких ссылок
- Указатели на блоки данных на жестком диске

Для inode есть своя таблица, занимающая место на диске, и они могут закончится, тогда создавать новые файлы не получится, даже если есть место на диске

Структура индексного дескриптора состоит из огромного колличества указателей:

![[Pasted image 20260605203607.png]]

Если файл слишком маленький, то его данные могут хранится в таблице inode. Например все симлинки хранятся именно в этой таблице

#### Освобождение номеров inode

Как только счетчик жестких ссылок на этот Inode в системе падает до нуля (то есть удалены все имена, связанные с ним), Linux делает следующее:

1. Помечает этот номер Inode в специальной битовой карте как **«свободный»**. Теперь система может выдать этот паспорт новому файлу.
2. Помечает блоки данных на диске, где лежал контент, как **«свободные для записи»**.  
    _(Сами данные физически не затираются нулями сразу ради скорости — именно поэтому случайно удаленный файл иногда можно восстановить, если на диск еще ничего не успели записать поверх)._


`ls -i file`              # Показать inode файла
`stat file`               # Детальная информация о файле
`mkfs.ext4`               # Управление inode
`df -hi`                     # Информация об inode на ФС, сколько всего inode в системе.

`find / -xdev -type d -print0 | xargs -0 -I {} sh -c 'echo $(find "{}" -maxdepth 1 | wc -l) "{}"' | sort -n -r | head -n 20`    # Команда для поиска папок, с наибольшим колличеством занятых inode.

### Ссылки — это указатели на файлы или папки.

 Вместо того чтобы копировать огромный файл и занимать место, вы можете создать ссылку на него.
##### Жесткая ссылка (Hard Links)

Жесткая ссылка — это **еще одно имя для того же самого Inode**. 

- **Как это устроено:** Создавая жесткую ссылку, вы просто добавляете новую запись в таблицу директории (Новое Имя <—> Старый Номер Inode).
- **Размер:** Занимает 0 байт на диске, так как блоки данных не дублируются.
- **Ограничения:**
    1. Нельзя создать ссылку на **папку** (чтобы избежать бесконечных циклов в системе).
    2. Нельзя создать ссылку на файл, который лежит на **другом диске или другой файловой системе** (ведь на другом диске таблица Inode своя, и там этот номер может принадлежать чужому файлу).
- **Удаление:** Если вы удалите оригинал, файл **не исчезнет**. Он будет доступен по жесткой ссылке. Данные удалятся только тогда, когда вы сотрете _все_ жесткие ссылки на этот Inode (счетчик ссылок в паспорте файла станет равен 0).
- Любое имя файла в любой папке - это и есть hardlink, системе без разницы что было первое. У одной inode могут быть разные именя жестких ссылок.
- При изменении прав на одну жесткую ссылку, они изменятся и на другую, потому что права хранятся в inode, а она у жестких ссылок одна.
- При создании папки, у нее есть сразу 2 hardlink'а:
	1. **Первая ссылка:** Имя `my_folder` в родительской директории.
	2. **Вторая ссылка:** Скрытое имя `.` (точка) внутри самой `my_folder`.
	3. Каждая **поддериктория** в этой папке будет дополнительной ссылкой

`ln target link_name`
`ls -li`                  # Одинаковый inode


##### Символические ссылки (Symlinks / Мягкие ссылки)

Символическая ссылка — это **отдельный маленький файл особого типа**, внутри которого написан текстовый путь к оригинальному файлу. Это полный аналог «Ярлыков» из Windows.

- **Как это устроено:** У симлинка создается **свой собственный, новый номер Inode** и свои блоки данных, где записан текст (например: `/var/www/index.html`).
- **Преимущества:**
    1. Можно ссылаться на **любые папки**.
    2. Можно ссылаться на файлы на **других дисках, флешках или даже сетевых папках**.
- **Удаление:** Если вы удалите оригинал, симлинк останется, но он «сломается» (станет подсвечиваться красным в консоли). Ссылка будет вести в никуда. Если удалить сам симлинк — с оригиналом ничего не случится.
- **Fast Symlinks (Быстрые ссылки):** Если путь к оригиналу короткий (меньше 60 символов в ext4), Linux не тратит блоки данных на диске. Он записывает текстовый путь прямо внутрь самого Inode — туда, где обычно хранятся адреса блоков данных. Такая ссылка занимает **0 блоков** на диске и считывается мгновенно.
- **Slow Symlinks (Медленные ссылки):** Если путь длинный (например, больше 60 символов), выделяется стандартный блок данных на диске (обычно 4 КБ), и этот текстовый путь записывается туда.
- Права доступа всегда `lrwxrwxrwx`, проверка идет из прав на сам ссылаемый объект.
- Многое зависит от того, абсолютный путь или относительный.

`ln -s target symlink_name`
`ls -l`                   # Видно куда указывает

Различия:

Жесткая - нельзя на директории, нельзя между ФС
Символическая - можно на всё, если удалить оригинал - ссылка сломается

## Файловые дескрипторы

Файловый дескриптор (file descriptor, FD) — это **целое неотрицательное число**, которое операционная система (ядро Linux) использует для идентификации **открытого файла**, **сокета**, **канала (pipe)** или любого другого **ресурса ввода-вывода (I/O)** внутри процесса .

#### Как это выглядит на практике?

Когда процесс запускается, он по умолчанию получает три предопределенных (базовых) файловых дескриптора от ядра:

| Канал | Имя               | Назначение                                  |
| ----- | ----------------- | ------------------------------------------- |
| **0** | `STDIN` (stdin)   | Стандартный поток ввода (обычно клавиатура) |
| **1** | `STDOUT` (stdout) | Стандартный поток вывода (обычно экран)     |
| **2** | `STDERR` (stderr) | Стандартный поток ошибок (обычно экран)     |

Любые другие файлы, сетевые соединения или устройства, к которым обращается программа по ходу своей работы, получают новые порядковые номера: `3`, `4`, `5`, `6` и так далее.

Как только программа делает системный вызов `open()` (открывает файл на диске) или создает сетевое подключение (сокет), Linux ищет первый свободный номер в таблице процесса и присваивает его этому объекту.

Представьте веб-сервер (например, Nginx). При старте у него открыты стандартные `0`, `1`, `2`. Затем он начинает работать:

- Дескриптор **`3`** — он открывает конфигурационный файл `nginx.conf`.
- Дескриптор **`4`** — он открывает файл логов `access.log` для записи.
- Дескриптор **`5`** — он открывает сетевой сокет, чтобы слушать порт 80 (интернет-трафик).
- Дескрипторы **`6`, `7`, `8`... `1002`** — это новые сетевые сокеты для каждого пользователя, который прямо сейчас зашел на сайт.

Когда пользователь закрывает вкладку сайта, сервер закрывает сокет, и этот номер дескриптора снова становится свободным для следующего гостя.

Файловые дескрипторы лежат в основе всего сетевого и файлового взаимодействия в Linux. Когда процессу нужно выполнить операцию ввода-вывода (I/O), он не передает ядру путь к файлу или адрес сокета каждый раз. Вместо этого он использует короткое, эффективное **число**, которое ядро выдало при открытии ресурса.

**Основные системные вызовы (syscalls), работающие с FD:**
- **`open()` / `socket()` / `pipe()`:** Создают ресурс и возвращают новый файловый дескриптор.
- **`read()` / `write()`:** Используют FD для чтения или записи данных в ресурс.
- **`close()`:** Говорят ядру, что процесс закончил работу с ресурсом, и FD можно освободить.

Концепция «все есть файл» в Linux работает на полную. FD используется не только для обычных файлов:

- **Сокеты (Sockets):** В веб-сервере (Nginx, Apache) каждое новое соединение клиента — это новый сокет, т.е. новый FD. Ошибка `too many open files` для сервера — часто результат того, что он не справляется с количеством одновременных подключений. 

- **Каналы (Pipes):** Когда вы пишете `command1 | command2`, создается канал. `stdout` первой команды (FD 1) подключается к `stdin` второй команды (FD 0) через этот канал.

- **Epoll:** Это специальный механизм, когда процесс создает FD для наблюдения за множеством других FD («список наблюдения»). Когда на любом из наблюдаемых FD происходит событие (появились данные, закрылся сокет), ядро сообщает об этом через epoll FD. Это основа работы высокопроизводительных серверов (Nginx, Node.js) в Linux.

#### Примеры работы дескрипторов и перенаправления в консоли

**Пример в bash:** Их можно увидеть в `/proc/<PID>/fd/`.

```bash
# Запустите команду sleep в фоне
sleep 300 &
[1] 12345
# Посмотрите дескрипторы процесса
ls -l /proc/12345/fd/
```

**Вывод будет похож на этот:**

```text
lrwx------ 1 user user 64 May 30 10:00 0 -> /dev/pts/0
lrwx------ 1 user user 64 May 30 10:00 1 -> /dev/pts/0
lrwx------ 1 user user 64 May 30 10:00 2 -> /dev/pts/0
```

Это показывает, что все три стандартных потока связаны с текущим терминалом `/dev/pts/0`

##### Простой пример (На языке C)

Допустим, вы хотите открыть файл `test.txt`. Вот пример кода на C, который наглядно это демонстрирует:

```c
#include <fcntl.h>
#include <stdio.h>
#include <unistd.h>
int main() {
    // Попытка открыть файл. open() возвращает файловый дескриптор (fd).
    int fd = open("test.txt", O_RDWR | O_CREAT, 0644);
    
    if (fd == -1) {
        // Ошибка открытия, fd = -1.
        perror("open");
        return 1;
    }
    
    printf("Файл открыт, файловый дескриптор: %d\n", fd);
    
    // Пишем что-то в файл, используя fd.
    write(fd, "Hello, file descriptor!\n", 23);
    
    // Закрываем файл, освобождая fd.
    close(fd);
    printf("Файл закрыт.\n");
    
    return 0;
}
```

**Вывод:**

```text
Файл открыт, файловый дескриптор: 3
Файл закрыт.
```

Первый открытый файл для процесса, помимо стандартных потоков (0,1,2), получает следующий доступный номер — **3**.

> Перенаправления логов

```bash
tar -czf backup.tar.gz /var/www/ 1> success.log 2> errors.log
```

`1`, `2` - это номера соответствующих дескрипторов STDOUT, STDERR и их перенаправление в соответствующие файлы.

- Например `> /dev/null` — отправляет дескриптор `1` (успешный вывод) в «черную дыру».
- Или `>./alllogs 2>&1` — отправляет `1` в файл `alllogs`, а `2` отправляет по ссылке туда же, куда и `1`.
- `&>` — синтаксический сахар, означает тоже самое что и `> file 2>&1`, на старых оболочках типо `sh` не работает.
- `command >> file.log` — дописывает успешный вывод в конец файла.
- `command 2>> errors.log` — дописывает только ошибки в конец файла.
- `command >> file.log 2>&1` — дописывает и вывод, и ошибки в один файл.

> Механизм Пайпов / Конвейеров (Конвейер данных через `|`)

Пайп (символ вертикальной черты `|`) — это мощнейший инструмент Linux. На уровне дескрипторов он делает следующее: **забирает дескриптор `1` (stdout) первой программы и втыкает его в дескриптор `0` (stdin) второй программы.**

```bash
cat /var/log/nginx/access.log | grep "404" | wc -l
```

1. `cat` читает лог и выводит его в дескриптор `1`.
2. Пайп перенаправляет этот вывод в дескриптор `0` команды `grep`.
3. `grep` фильтрует только строки с ошибкой "404" и выдает их в свой дескриптор `1`.
4. Второй пайп передает эти строки в дескриптор `0` команды `wc -l`, которая считает количество строк.  

> Одновременный вывод на экран и в файл (`tee`)

Если вы перенаправляете вывод через `>`, экран остается пустым. Если вы хотите **и видеть, что происходит в терминале, и одновременно сохранять это в лог**, используется утилита-тройник `tee`:

```bash
make compile 2>&1 | tee build.log
```

_Конструкция `2>&1 |` сливает ошибки с выводом и отправляет их в пайп, а `tee` раздваивает этот поток: одну копию пускает на экран, вторую — записывает в `build.log`._

> Чтение из строки на лету (Here-string: `<<<`)

Обычно для передачи текста в программу используют пайп: `echo "my text" | grep "text"`. Но в Bash можно передать строку напрямую в дескриптор `0` (ввод) без запуска лишнего процесса `echo`:

```bash
grep "text" <<< "my text"
```

> Передача многострочного текста (Here-doc: `<<`)

Используется в скриптах, когда нужно записать целый блок текста или конфигурации в файл из самого скрипта. Вы сами задаете слово-маркер (обычно `EOF` — End Of File):

```bash
cat << EOF > config.conf
server_name=localhost
port=80
db_user=root
EOF
```


Без дескрипторов нам пришлось бы сохранять промежуточные результаты в кучу временных файлов на диске.

#### Утилита `lsof` для просмотра всех дескрипторов

```bash
lsof -p <PID>      # Для процесса
lsof /path/to/file # Для файла
lsof -i :8080      # Для порта
```

Вы увидите список всех открытых FD для процесса (`cwd`, `txt`, `mem`, `3u`, `4u` и т.д.). Это незаменимая команда для диагностики утечек.

```bash
ls -l /proc/<id>/fd/   # В файловой системе процессов
```

Почему в `/proc/[PID]/fd` меньше строк, чем в выводе `lsof`?

Директория `/proc/[PID]/fd/` содержит **только классические числовые дескрипторы** (`0`, `1`, `2`, `3`...), которые программа открыла через явные системные вызовы вроде `open()` или `socket()`.

Полноценные системные маркеры (`cwd`, `txt`, `mem`, `rtd`) там **не показываются**, потому что ядро Linux не выделяет под них номера в таблице дескрипторов процесса. Но они отображаются в других файлах внутри `/proc`:

- **`cwd`** — в `/proc/[PID]/` есть отдельная символическая ссылка `cwd`, ведущая в текущую папку.
- **`txt`** — там же есть ссылка `exe`, указывающая на исполняемый файл.
- **`mem`** — список замапленных библиотек хранится в текстовом файле `/proc/[PID]/maps`.

Утилита `lsof` (List Open Files) делает огромную аналитическую работу: она заглядывает во _все_ эти скрытые файлы и ссылки внутри `/proc`, собирает информацию воедино и выводит вам красивую общую таблицу, где склеивает и числовые дескрипторы, и системные мета-связи (`cwd`, `txt`).

---

#### Текстовые маркеры (Специальные дескрипторы)

Этим объектам ядро Linux не присваивает стандартные порядковые номера `0`, `1`, `2`, но процессу они критически важны для работы:

- **`cwd` (Current Working Directory)** — Текущая рабочая директория. Это папка, внутри которой процесс сейчас «находится». Если вы в терминале выполните `cd /var/log`, то для этого терминала `cwd` станет указывать на `/var/log`.
- **`txt` (Text / Executable code)** — Программный код. Это ссылка на сам бинарный исполняемый файл программы (например, файл `/usr/bin/nginx`), который сейчас загружен в память и выполняется.
- **`mem` (Memory-mapped file)** — Файл, проецируемый в память. Обычно это системные библиотеки (`.so`-файлы в Linux), которые программа замапила в ОЗУ для быстрого вызова общих функций.
- **`rtd` (Root Directory)** — Корневая директория процесса. Обычно это просто `/`, но может быть другой папкой, если процесс запущен в изолированном окружении через `chroot`.

#### Цифровые дескрипторы с буквами (`3u`, `4u`, `0r`, `1w`)

Когда вы видите число, за которым следует буква — это **порядковый номер дескриптора** (из таблицы процесса) + **режим, в котором этот файл открыт**.

Буквы после цифры означают следующее:

- **`u` (read and write / универсальный)** — Файл открыт одновременно и на чтение, и на запись. В вашем примере `3u` и `4u` означают: дескриптор №3 и дескриптор №4 открыты в двустороннем режиме (так часто открываются сетевые сокеты или терминалы).
- **`r` (read)** — Файл открыт строго только для чтения (например, `0r` — стандартный ввод читает данные).
- **`w` (write)** — Файл открыт строго только для записи (например, `1w` — стандартный вывод пишет логи или данные в файл).
- **`W` (write lock)** — Процесс открыл файл на запись и заблокировал его, чтобы другие программы не могли параллельно в него писать и портить данные.


- **Просмотр лимитов процесса:**

```bash
cat /proc/<PID>/limits
```

Здесь, в частности, можно увидеть параметр **`Max open files`** — максимальное количество файлов, которое может открыть процесс. Его превышение приводит к знаменитой ошибке **`too many open files`**.

Как проверить свои лимиты?

В консоли Linux можно узнать ограничения текущего пользователя с помощью команды `ulimit`:
`ulimit -Sn` — **мягкий лимит** (soft limit). Это ограничение, которое программа может сама временно увеличить при необходимости.
`ulimit -Hn` — **жесткий лимит** (hard limit). Потолок, выше которого обычный пользователь прыгнуть не может (его может поднять только `root` в системных настройках).


Файловые дескрипторы дают **единый абстрактный интерфейс**. Системе и программам абсолютно всё равно, куда они пишут данные: в файл на SSD, в консоль пользователю, в сетевой сокет на другой конец планеты или в другую программу. Для кода это всегда просто операция `write(дескриптор_номера, данные)`

## Системная таблица открытых файлов

Когда программа открывает файл или сокет, ядро Linux не просто выдает ей случайное число дескриптора. На самом деле в глубинах ядра создается целая **трехслойная структура данных ввода-вывода**.

**Системная таблица открытых файлов** (System-wide Open File Table) — это центральный, «средний» слой этой структуры. Она едина для всей операционной системы и хранит состояние абсолютно всех файлов, которые считываются или пишутся прямо сейчас.

Чтобы понять, зачем она нужна и как работает, давайте посмотрим на все три слоя, сверху вниз:

> **Слой 1: Таблица дескрипторов процесса (Локальная)**

У каждого процесса (например, у вашего Python-скрипта или браузера) есть своя личная таблица.

- Индексы в этой таблице — это и есть те самые **числа дескрипторов (`0`, `1`, `2`, `3`...)**.
- В этой таблице нет информации о файле. В ней содержатся только флаги (например, «закрывать ли этот дескриптор при запуске другой программы») и **указатель (ссылка)**, ведущий на второй слой.
- Локальная таблица дескрипторов процесса полностью материализована в виртуальной папке `/proc/[PID]/fd/`. Каждая строка в этой таблице представлена в виде символической ссылки.

```bash
ls -l /proc/self/fd/
```

Вывод будет подобный:  
`lrwx------ 1 user user 64 Jun 6 16:00 0 -> /dev/pts/0`

- **Цифра `0`** — это индекс (строка) в локальной таблице дескрипторов процесса.
- **Стрелочка `->`** — это и есть визуализированный ядром **указатель** на системный объект открытого файла.
- **`/dev/pts/0`** — это имя файла (или устройства), к которому привязана строка в системной таблице открытых файлов.


> **Слой 2: Системная таблица открытых файлов (Глобальная)**

Она одна на всю систему. Каждый раз, когда любая программа вызывает `open()`, в этой таблице создается новая строка (запись).

Она хранит динамическое состояние работы с файлом:

1. **Текущее смещение (File Offset / Позиция чтения-записи):** Это указатель на байт, у которого сейчас остановилась программа. Если программа прочитала первые 100 байт, смещение станет равно `100`. Следующий вызов `read()` начнется со 101-го байта.
2. **Статусные флаги:** В каком режиме открыт файл (`O_RDONLY` — только чтение, `O_APPEND` — дописывать в конец, `O_NONBLOCK` — неблокирующий ввод-вывод).
3. **Счетчик ссылок:** Сколько дескрипторов из разных (или одного) процессов прямо сейчас указывают на эту конкретную строку.
4. **Указатель на третий слой** (таблицу Inode).

Посмотреть на саму систему можно несколькими способами:

```bash
cat /proc/sys/fs/file-nr
```

Вывод будет по типу:  
`1536 0 9223372036854775807`

**Как это читать (это и есть мета-параметры системной таблицы):**

1. **Первое число (`1536`)** — Количество записей (строк), занятых в системной таблице открытых файлов _прямо сейчас_ по всей операционной системе.
2. **Второе число (`0`)** — Количество выделенных, но освободившихся строк (в современных ядрах Linux этот показатель часто равен 0, так как память выделяется динамически).
3. **Третье число (`9223372036854775807`)** — Максимально возможный размер системной таблицы открытых файлов (`fs.file-max`). Выше этого значения операционная система физически не сможет открыть ни одного файла или сокета.

Чтобы буквально сделать дамп всей системной таблицы открытых файлов:

```bash
sudo lsof
```

У каждого процесса есть структура `task_struct` (паспорт процесса). Внутри нее есть указатель на структуру `files_struct` (Локальная таблица). А внутри нее лежит массив указателей:

Каждый элемент этого массива `fd_array[0]`, `fd_array[1]` — это и есть дескриптор. А тип `struct file *` — это и есть **указатель-ссылка** на строку в Глобальной системной таблице открытых файлов.


> **Слой 3: Таблица Inode (Глобальная виртуальная память VFS)**

Это кэш «паспортов» файлов в оперативной памяти. Здесь хранится статическая информация о файле: его тип, права доступа, размер и где его блоки лежат на физическом диске.

---

Такая трехслойная схема создана ради реализации ключевых фишек Unix/POSIX-систем. Это открывает потрясающие возможности:

А. Разделение позиции чтения между процессами (`fork()`)

Когда процесс создает свою копию с помощью функции `fork()`, дочерний процесс полностью копирует локальную Таблицу 1. В итоге дескриптор `3` у родителя и дескриптор `3` у потомка начинают указывать на **одну и ту же строку** в системной Таблице 2.

- _Эффект:_ Если родитель прочитал из файла 50 байт (смещение сдвинулось на 50), то потомок автоматически продолжит читать с 51-го байта. Они не мешают друг другу и читают файл как одну общую очередь.

Б. Независимое чтение одного файла (`open()` + `open()`)

Если две абсолютно разные программы (например, два плеера) одновременно открывают один и тот же файл `video.mp4`:

- В системной Таблице 2 создаются **две разные строки**. У каждого плеера будет свое собственное смещение (один смотрит на 5-й минуте, второй — на 40-й).
- Но обе эти строки в Таблице 2 будут указывать на **один и тот же Inode** в Таблице 3. Система знает, что файл физически один, права доступа проверяются один раз, а данные кэшируются в памяти совместно.

Поэтому в Linux существует несколько разных лимитов на открытые файлы:

1. Лимит на уровне `ulimit -n` (например, 1024) — это максимальный размер **Локальной таблицы (Слой 1)** для конкретного процесса.
2. Системный лимит `fs.file-max` (можно посмотреть через `cat /proc/sys/fs/file-max`) — это максимальная вместимость всей **Глобальной таблицы открытых файлов (Слой 2)** для всего ядра Linux. Если сервер откроет миллионы сокетов, место закончится именно в этой системной таблице.

## VFS (Virtual File System)

**VFS (Virtual File System / Виртуальная файловая система)** — это еще один абстрактный слой в ядре Linux, который находится между программами и конкретными файловыми системами (Ext4, NTFS, Btrfs или сетевыми NFS). Именно благодаря VFS, программам без разницы, на каком диске или флешке лежит файл. Код всегда вызывает одну стандартную функцию `open()`, а VFS сама разбирается, как эту команду выполнить для конкретной файловой системы.

Без VFS разработчикам программ пришлось бы писать разный код для работы с разными дисками. Программа бы думала: _«Так, этот файл лежит на флешке FAT32, значит мне нужно вызвать функцию `read_fat32()`. А этот лог пишется на диск Ext4, вызываю `read_ext4()`»_.

VFS берет эту головную боль на себя. Она предоставляет программам **единый интерфейс** (тот самый стандарт POSIX). Программа делает обычный вызов `read(fd)`, а VFS на лету превращает его в специфические команды для конкретного железа и файловой системы.


#### Главные объекты (структуры данных) в VFS

Мир VFS держится на четырех основных объектных типах.

1. **`superblock` (Суперблок)** — Описывает конкретную подключенную файловую систему (ее размер, тип, статус). Это «главный управляющий» диска.
2. **`inode` (Индексный дескриптор)** — Описывает конкретный файл (его размер, права, владельца). В VFS существует структура `struct inode`, которая является универсальной абстракцией для паспорта файла, независимо от того, где он лежит.
3. **`dentry` (Directory Entry / Элемент директории)** — Связывает имя файла с его Inode. Именно `dentry` склеивает пути (например, `/home/user/file.txt`) в цепочку. Ядро кэширует их в оперативной памяти (`dentry cache`), чтобы мгновенно находить файлы по имени, не опрашивая медленный жесткий диск при каждом чихе.
4. **`file` (Открытый файл)** — Описывает связь между процессом и открытым файлом. Это та самая строка во **Втором слое** (Системной таблице открытых файлов), которая хранит File Offset (смещение).


Поскольку VFS — это чистая абстракция, ей все равно, откуда брать данные. Это позволяет Linux создавать «ненастоящие» файловые системы, с которыми пользователь работает как с обычными папками, ниже представлены псевдо-фс, которых не существует на жестком диске, это чистая абстракция VFS в оперативной памяти:

- **`/proc` (procfs)** — Виртуальная файловая система, которая живет только в оперативной памяти ядра. Когда вы делаете `cat /proc/sys/fs/file-nr`, драйвер `procfs` перехватывает операцию `read()`, находит в оперативной памяти структуру `task_struct` для процесса `1234`, вытаскивает оттуда строку запуска и отдаёт её вам в текстовом виде.
- **`/sys` (sysfs)** — Еще одна виртуальная папка для управления «железом» (например, через нее можно поменять яркость экрана или скорость кулера, просто записав цифру в текстовый файл). Драйвер `sysfs` перехватывает операцию `write()`, берет вашу цифру `0` и мгновенно передает её драйверу видеокарты, который физически понижает напряжение на подсветке экрана.
- `tmpfs` — **(быстрый диск в ОЗУ):** Это псевдо-ФС, которая выглядит как папка, но данные пишутся напрямую в оперативную память. Всё, что там лежит, сотрется при перезагрузке, но скорость работы с ней феноменальная (используется для `/tmp` и кэшей).
- **Сетевые папки (NFS / Samba)** — Файлы физически лежат на сервере в другой стране, но благодаря VFS вы видите их в своем файловом менеджере Linux как обычную локальную папку.

Когда вы запрашиваете файл из `/proc` или `/sys`, VFS не делает запросов к контроллеру диска. Вместо этого она перенаправляет ваш запрос **функциям обратного вызова (callbacks)** внутри ядра.


`Программа (Код)`  
⬇️ _вызывает read(3)_  
`Локальная таблица дескрипторов процесса (Слой 1)`  
⬇️ _ведет на..._  
`Системная таблица открытых файлов VFS (Слой 2 / struct file)`  
⬇️ _знает смещение и ведет на..._  
`Универсальный VFS Inode (Слой 3 / struct inode)`  
⬇️ _вызывает функцию конкретной ФС (например, ext4_read_block)_  
`Драйвер Ext4 / Диспетчер дисков`  
⬇️ _считывает физические байты_  
`Жесткий диск (SSD / Флешка)`


#### Команда `mount` - главный инструмент управления VFS

Она связывает физическое устройство или виртуальный диск с конкретной директорией в системе (точкой монтирования)

Чтобы операционная система увидела подключенный диск или флешку, нужно выполнить два простых шага:


```bash
# 1. Узнать имя устройства в системе
lsblk

# 2. Примонтировать раздел в нужную папку

sudo mkdir /mnt/my_flash
sudo mount /dev/sdb1 /mnt/my_flash

# Чтобы размонтировать, используется команда umount
sudo umount /mnt/my_flash
```

Когда вы выполняете команду `sudo mount /dev/sdb1 /mnt/my_flash`, вы не указываете тип файловой системы. Ядро Linux находит нужный драйвер автоматически.

Происходит это по следующему алгоритму:

1. **Проверка суперблока:** Утилита `mount` обращается к специальной системной библиотеке `libblkid`. Она считывает первые несколько килобайт диска (`/dev/sdb1`), где записаны метаданные (сигнатуры) файловой системы (суперблок).
2. **Определение типа:** По этим сигнатурам система мгновенно понимает что там за файловая система.
3. **Загрузка модуля:** Ядро проверяет, загружен ли в оперативную память драйвер для `ext4`. Если нет, ядро динамически загружает нужный модуль (драйвер) из папки `/lib/modules/`
4. **Принудительный выбор:** Вы можете заставить систему использовать конкретный драйвер вручную с помощью флага `-t` (type):

```bash
sudo mount -t ext4 /dev/sdb1 /mnt/my_flash
```

Можно посмотреть список поддерживаемых драйверов в виртуальном файле:

```bash
cat /proc/filesystems
```

_Слово `nodev` перед названием означает, что драйвер работает без физического устройства (это псевдо-ФС)_

**Чтобы посмотреть, какой драйвер отвечает за какие файлы, нужно использовать команду `mount` без параметров, это покажет полную карту монтирования VFS**:

`[ЧТО] on [КУДА] type [КАКОЙ ДРАЙВЕР] ([ПАРАМЕТРЫ ДОСТУПА])`

#### Монтирование

Образ диска (`.img` или `.iso`) — это просто один большой обычный файл, внутри которого записаны побайтовые копии секторов настоящего диска. Благодаря VFS, этот файл можно примонтировать как полноценный жесткий диск.

Шаг 1: Создаем пустой файл под образ (например, на 1 ГБ)

Используем утилиту `dd` (побайтовое копирование нулей из виртуального генератора `/dev/zero`):

```bash
sudo dd if=/dev/zero of=my_disk.img bs=1M count=1024
```

Шаг 2: Форматируем этот файл в нужную файловую систему

Мы заставляем утилиту форматирования думать, что файл `my_disk.img` — это диск:

```bash
sudo mkfs.ext4 my_disk.img
```

Шаг 3: Монтируем образ диска

Чтобы VFS могла работать с файлом как с блочным устройством, `mount` автоматически использует скрытый механизм **Loop-устройств** (виртуальных петель). Файл виртуально превращается в `/dev/loop0`:

```bash
sudo mkdir /mnt/iso_disk
sudo mount -o loop my_disk.img /mnt/iso_disk
```

_Всё! Теперь вы можете зайти в `/mnt/iso_disk`, создавать там папки, копировать файлы. Они будут физически записываться внутрь одного единственного файла `my_disk.img`._

1. Файл `/etc/fstab` (Автомонтирование)

Вы уже знаете, как монтировать диски вручную через `mount`. Но если компьютер перезагрузится, все ваши точки монтирования исчезнут.  
Чтобы диски подключались автоматически при старте Linux, их прописывают в конфигурационный файл `/etc/fstab` (File System Table). Каждая строка в нем указывает VFS: _«При старте найди диск с таким-то UUID (уникальным ID), примонтируй его в такую-то папку, используй драйвер ext4 и такие-то права»_.

2. Механизм Журналирования (Защита от сбоев)

Современные файловые системы (Ext4, NTFS, XFS) называются **журналируемыми**. Внутри них есть специальная скрытая круговая зона — **журнал**.

- **Как это работает:** Когда вы сохраняете файл, ФС сначала быстро записывает план своих действий в журнал (_«Я собираюсь изменить блоки 5 и 6 для файла doc.txt»_), и только потом меняет сами блоки.
- **Зачем нужно:** Если посреди записи у вас внезапно отключится электричество, Linux при следующем включении не будет сканировать весь терабайтный диск часами. Она просто заглянет в журнал, увидит незавершенную операцию и либо безопасно откатит её назад, либо допишет до конца. Данные не повредятся.

3. Кэш страниц (Page Cache) и команда `sync`

Когда программа делает вызов `write()`, Linux ради скорости **не пишет данные на SSD мгновенно**. Она сохраняет их в оперативной памяти (в кэше страниц VFS) и сразу говорит программе: _«Всё готово, я всё записала!»_, а на физический диск сбрасывает их в фоне через несколько секунд. Такие данные в ОЗУ называются _грязными страницами (dirty pages)_.

- **Практический вывод:** Именно поэтому флешки нельзя выдергивать просто так — данные из кэша могут еще не успеть долететь до физических транзисторов. Команда **`sync`** в консоли принудительно заставляет VFS немедленно сбросить весь кэш из памяти на физические диски.
