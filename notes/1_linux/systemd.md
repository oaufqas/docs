### Отличие демона от процесса

- **Процесс** — это любая запущенная программа (калькулятор, браузер, команда `ls`). Обычно он привязан к твоему окну или терминалу: закрыл терминал — процесс умер.
- **Демон (Daemon)** — это процесс, который **отвязался** от терминала и ушел работать в фон. Он живет сам по себе, не имеет графического окна и обычно запускается системой при старте.

|Характеристика|Обычный процесс|Демон (Service в systemd)|
|---|---|---|
|**Взаимодействие**|Ты видишь его окно или вывод в консоли.|Работает незаметно «под капотом».|
|**Жизненный цикл**|Завершается, когда ты закрываешь программу.|Работает постоянно, пока не остановят службу.|
|**Привязка**|Зависит от пользователя (сессии).|Зависит от системы (работает даже если никто не залогинен).|
|**Примеры**|`python script.py`, `ls`, `chrome`.|`httpd` (Apache), `sshd`, `mysql`.

### Загрузка системы (Boot process)

##### Этапы загрузки

```text
1. BIOS/UEFI
   ↓
2. Bootloader (GRUB)
   ↓
3. Kernel (загрузка ядра)
   ↓
4. initramfs (временная ФС)
   ↓
5. init/systemd (PID 1)
   ↓
6. Запуск сервисов
   ↓
7. Login prompt
```

#### GRUB (Grand Unified Bootloader)

```bash
/boot/grub/grub.cfg     # Конфигурация (не редактировать вручную!)
/etc/default/grub       # Настройки для генерации конфига
update-grub             # Обновить конфиг (Debian/Ubuntu)
grub2-mkconfig          # Обновить конфиг (RHEL/CentOS)
```

#### initramfs (initrd)

Временная корневая ФС в памяти:
- Содержит драйверы для монтирования реальной ФС
- Загружается до настоящей корневой ФС

|0|poweroff.target|Выключение|
|1|rescue.target|Однопользовательский режим|
|2|multi-user.target|Мультипользовательский (без GUI)|

```bash
systemctl get-default               # Текущий target

```

### `Systemd` (системный демон):

- Это первый процесс (PID 1), который запускается при старте Linux
- Он управляет всеми остальными процессами: запускает, останавливает, следит за ними
- Отвечает за загрузку системы, монтирование дисков, запуск сервисов
- Работает в фоне постоянно
- Имеет модульный функционал

После загрузки OC (когда загрузчик передает управление яндру), ядро запускает `initrd` процесс, в своем окружении нужных утилит, а уже этот процесс запускает `systemd`

`Systemd`, после запуска, запускает других демонов, для работы операционной системы:
`systemd-logind`, `systemd-timesyncd`, `systemd-networkd` и др, а также **`dbus`** - шина для управления демонами через апи.

И все заимодействие клиентов по типу `systemctl`, `timedatectl`, `resolvectl`, `networkctl` с демонами, идет через эту шину **`dbus`** c помощью библиотеки `libdbus`

##### **Юнит (Unit)** — это ==базовый объект управления в== `systemd`
Обычно это обычный файл в директории `/lib/systemd/system/` или `/etc/systemd/system/`
Внутри он разбит на секции:

1. **`[Unit]`** — общая информация. Описание («что это?») и зависимости («после чего запускать?»).
2. **`[Тип]`** — настройки конкретного типа (например, `[Service]` для программ или `[Timer]` для расписания). Тут пишут пути к исполняемым файлам или интервалы времени.
3. **`[Install]`** — правила автозагрузки. Инструкция о том, в какой момент юнит должен активироваться при старте системы.

Тип юнита определяется его **расширением** (те самые `.service`, `.timer`, `.mount`)
#### Systemd unit types:

- `${name}.service` — запускает и контролирует **процессы**. `(systemctl status nginx)`
- `${name}.timer` — активирует другие юниты (обычно сервисы) по **расписанию**.
- `${name}.socket` — управляет **сетевыми портами** или сокетами. 
- `${name}.path` — следит за изменением конкретных **файлов или папок**.
- `${name}.device` — представляет **устройство** в системе.
- `${name}.mount` — управляет **точкой монтирования** файловой системы.
- `${name}.automount` — обеспечивает **автоматическое монтирование** файловой системы.
- `${name}.swap`  — управляет разделом или файлом **подкачки**.
- `${name}.slice` — создает иерархию для **ограничения ресурсов** (CPU, память, cgroups)
- `${name}.scope` — unit, запущенный не через systemd
- `${name}.target` — группировка unit'ов

- `${name}@{instance_name}.service (postgresql)` - шаблоны для запуска множества однотипных служб с разными параметрами

- `symlinks` - алиласы (ssh, sshd)

- `${name}.service.d/${01_some_override}.conf` или `systemctl edit ${name}` - возможность редактировать unit (override)

#### Где лежат unit'ы:

`/etc/systemd/system/..`
`/usr/lib/systemd/system/..`
`/run/systemd/system/`

###### `Service` считается устаревшей утилитой, для управления `init`-процессом, на ее замену пришла `systemctl`
##### Systemctl (system control) - это утилита, для управления systemd

- `systemctl start/stop ${name}` - запуск unit'ов
- `systemctl status/show ${name}` - определят состояние сервиса
- `systemctl enable/disable ${name}`- добавление `unit` в `multiuser.target`(автозагрузка)
- `systemctl kill ${name}` - если pause-сигнал не дошел
- `systemctl reload ${name}` - процесс перечитает файл конфигурации (unit), не прекращая свою работу, как nginx перечитывает conf, не сбрасывая соединений.
- `systemctl daemon-reload` - сообщение, об окончании редактирования unit'ов systemd
- `systemctl reset-failed` - сбросить лимит ошибок (при надобности)
- `systemctl cat ${name}` - увидеть содержимое unit'а и его override'ов

##### Journalctl (journal control) - утилита, для чтения и фильтрации логов от journald

```bash
journalctl -efa # Логи в структурированном виде
journalctl -efu ${unit_name} # Выбранный юнит
journalctl                    # Все логи
journalctl --since "1 hour ago"
journalctl -p err            # Только ошибки
journalctl --namespace=${namespace} # Логи указанного namespace
--grep # grep в логах
-e # Перемещение в конец логов (на текущий момент)
-f # Слежка за новыми логами
-a # Все юниты
-t # Выбранный тег
```

#### Уровни логирования (от критичных к отладочным):

```text
0 - emerg   (система неработоспособна)
1 - alert   (требуется немедленное вмешательство)
2 - crit    (критическое условие)
3 - err     (ошибка)
4 - warning (предупреждение)
5 - notice  (нормальное, но важное событие)
6 - info    (информационное сообщение)
7 - debug   (отладочная информация)
```

```bash

/var/log/messages      # Системный лог (RHEL/CentOS)

/var/log/kern.log      # Ядро
/var/log/dmesg         # Сообщения ядра при загрузке
/var/log/nginx/        # Логи nginx
/var/log/mysql/        # Логи MySQL
```
