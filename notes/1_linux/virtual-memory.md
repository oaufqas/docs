## Виртуальная память

Процесс запускается в виртуальная среде выполнения программы, которая находится в виртуальной памяти, отображается в оперативной памяти, но ей не является. Процесс думает, что вся память компьютера - его. На 32 битном процессоре, максимальный размер виртуальной памяти для процесса - 4 ГБ, на 64-бит - 256 ТБ.

Если бы процессы работали с физической оперативной памятью (RAM) на прямую, случился бы хаос, программа А могла бы перезаписать переменные программы Б. Чтобы этого избежать, ядро Linux и процессор компьютера объединяются и создают для каждого процесса **виртуальное адресное пространство**.

- Когда программа пишет в коде: «сохранить число по адресу `0x0040`», это **виртуальный адрес**.
- Специальный аппаратный блок внутри процессора — **MMU (Memory Management Unit / Блок управления памятью)** — на лету переводит этот виртуальный адрес в реальный физический адрес в RAM (например, в `0x9FFF`).

Вся память компьютера в процессе работы разрезана:

- **Страницы (Pages):** Это куски виртуальной памяти процесса. Стандартный размер одной страницы в Linux и Windows — **4 Килобайта** (4 КБ).
- **Фреймы (Page Frames):** Это такие же куски по 4 КБ, но уже внутри физической оперативной памяти (RAM).

Когда программа запрашивает память, ядро выделяет ей виртуальные страницы. Связь между ними и реальной RAM записывается в **Таблицу страниц** (Page Table).

Ядро Linux хранит для каждого процесса **Таблицу страниц (Page Table)** — это своего рода карта-переводчик. У процесса `A` виртуальный адрес `0x1000` ведет на физический кусок RAM №1, а у процесса `B` виртуальный адрес `0x1000` ведет на физический кусок RAM №2. Они используют одинаковые цифры в коде, но физически в ОЗУ никогда не пересекаются.

У каждой страницы в этой таблице есть биты-флаги, которые процессор проверяет при каждом обращении:

- **Present bit (Бит присутствия):** `1` — страница сейчас загружена в физическую RAM; `0` — страницы в RAM нет (она либо еще не создана, либо сброшена на диск в Swap).
- **Dirty bit (Бит загрязнения):** `1` — программа изменила данные в этой странице; `0` — данные не менялись. Этот флаг критически важен для свопинга.
- **Права доступа:** Запрещено ли исполнять код из этой страницы (защита от вирусов) или писать в неё.

>Виртуальную память выделяет **Ядро Linux** (его подсистема управления памятью) с помощью системных вызовов (таких как `brk`, `sbrk` или `mmap`). Это происходит в два ключевых этапа:

Этап А: При старте процесса (Вызов `exec`)

Когда вы запускаете программу, ядро создает структуру процесса и выделяет ему базовый объем виртуальной памяти:

- Загружает сегмент кода программы (Text).
- Создает фиксированный сегмент данных для глобальных переменных.
- Выделяет стартовый размер для **Стека** и **Кучи**.  
    На этом этапе память выделяется _только виртуально_ — ядро просто делает записи в своей таблице, что эти адреса теперь зарезервированы за процессом.

Этап Б: Динамически во время работы (По требованию)

Когда программа работает, ей постоянно нужны новые объемы данных (например, вы открыли новую вкладку в браузере или загрузили картинку). Программа просит память у ядра (в языках программирования это делают функции вроде `malloc()` в C или создание новых объектов/списков в Python). Ядро мгновенно раздвигает границы сегмента **Кучи (Heap)**, выделяя новые виртуальные адреса.

#### Ядро Linux не знает сколько выделять памяти процессу. Процесс сам говорит ядру, сколько ему нужно.

Процесс запрашивает память порциями по мере необходимости, но здесь кроется главный секрет оптимизации Linux — **Отложенное выделение (Lazy Allocation)**.

Когда программа просит у Linux 1 гигабайт памяти, ядро поступает очень хитро:

1. Оно **не выделяет** физическую оперативную память (RAM) сразу.
2. Оно просто проверяет: «Если этот процесс _действительно_ использует 1 ГБ, у меня в компьютере вообще хватит RAM и Свопа?». Если лимиты позволяют, ядро говорит программе: _«Да, держи свой гигабайт виртуальных адресов от `0x1000` до `0x5000`»_.
3. Программа думает, что память у неё есть. Физическая RAM при этом пуста.

В какой же момент выделяется реальная RAM?

Физическая память выделяется **постранично (обычно блоками по 4 КБ)** ровно в тот микросекундный момент, когда программа _впервые пытается что-то записать_ по выданному адресу.

Это происходит через механизм аппаратного прерывания:

1. Программа пытается записать цифру по виртуальному адресу `0x2000`.
2. Процессор (MMU) смотрит в таблицу страниц и видит что виртуальный адрес за процессом закреплен, но в реальной RAM под него еще нет физической ячейки.
3. Процессор останавливает программу и генерирует аппаратное исключение — **Page Fault (Ошибка страницы)**.
4. Ядро Linux мгновенно просыпается, перехватывает эту ошибку, находит в компьютере свободные 4 КБ реальной оперативной памяти, привязывает их к виртуальному адресу `0x2000` и незаметно запускает программу дальше.

Программа даже не замечает этой паузы. Для неё всё выглядит так, будто память была там всегда. Если программа затребовала 1 ГБ, но реально записала данные только в 10 Мегабайт — компьютер потратит на неё всего 10 МБ реальной памяти.

#### Свопинг (Swapping) - если память начинает заканчиваться

**Свопинг (Swapping)** — это механизм, который позволяет операционной системе использовать жесткий диск или SSD как продолжение оперативной памяти, если реальная RAM начинает заканчиваться.

В Linux для этого выделяется отдельный раздел диска или файл (**Swap**), в Windows — файл **`pagefile.sys`**.

Когда ядро Linux видит, что свободная RAM подходит к концу, запускается фоновый системный демон (в Linux это `kswapd`). Его задача — освободить фреймы в RAM.

1. Он ищет страницы процессов, к которым **дольше всего никто не обращался** (например, свернутый три часа назад браузер).
2. Демон проверяет **Dirty bit**:
    - Если страница «чистая» (программа её не меняла), ядро просто стирает её из RAM. Зачем тратить время на запись, если её можно в любой момент заново прочитать с диска (например, код самой программы).
    - Если страница «грязная» (измененная), ядро физически записывает эти 4 КБ данных на жесткий диск в область Swap.
3. В таблице страниц процесса **Present bit** сбрасывается в `0`, а вместо адреса в RAM записывается координата: _«эта страница теперь лежит в Swap на диске в секторе Х»_.
4. Физический фрейм в RAM объявляется свободным и отдается активной программе.

>Вы разворачиваете браузер, который три часа лежал в Swap. Браузер пытается прочитать свою переменную по виртуальному адресу.

1. Процессор (MMU) заглядывает в таблицу страниц и видит: **`Present bit = 0`**. Страницы в RAM нет!
2. Процессор бьет тревогу и генерирует аппаратное прерывание — **Major Page Fault** (тяжелая ошибка страницы).
3. Процесс полностью замирает. Ядро Linux берет управление на себя, смотрит в таблицу страниц, видит адрес сектора в Swap, считывает 4 КБ с диска и копирует их обратно в любой свободный фрейм RAM.
4. Ядро меняет `Present bit` обратно на `1`, прописывает новый адрес RAM и запускает процесс с той же микрокоманды.

#### Опасная сторона иллюзии: OOM Killer (Out of Memory)

Поскольку Linux раздает виртуальную память направо и налево, не подкрепляя её реальной RAM (это называется _Overcommit_ — оверкоммит / перевыделение), в системе может наступить критический момент.

Если все запущенные программы одновременно начнут активно писать данные в свои виртуальные гигабайты, реальная физическая оперативная память в компьютере внезапно закончится на 100%.

В этот момент активируется жесткий системный механизм ядра — **OOM Killer (Out of Memory Killer)**. Чтобы компьютер не завис намертво, ядро запускает алгоритм, который выбирает самый невежливый и тяжелый процесс в системе (часто это бывает Chrome, база данных или зависший скрипт) и мгновенно уничтожает его сигналом `SIGKILL`, освобождая память для остальных.

#### Чем отличаются Стек (Stack) и Куча (Heap)?

Это два разных региона в виртуальной памяти процесса. Они созданы для совершенно разных задач и управляются по-разному.

| Критерий            | Стек (Stack)                                                                                           | Куча (Heap)                                                                                                     |
| ------------------- | ------------------------------------------------------------------------------------------------------ | --------------------------------------------------------------------------------------------------------------- |
| **Для чего нужен**  | Для мгновенного сохранения локальных переменных внутри функций.                                        | Для хранения больших, динамических данных (массивы, объекты).                                                   |
| **Структура**       | Строгая очередь по принципу LIFO (последний пришел — первый ушел).                                     | Хаотичное «облако» памяти. Данные берутся из любого свободного места.                                           |
| **Кто управляет**   | **Процессор автоматически.** Память выделяется при входе в функцию и сама очищается при выходе из неё. | **Программист (или язык).** Нужно явно запрашивать память и вручную (или через Garbage Collector) её удалять.   |
| **Размер**          | Фиксированный и очень маленький (обычно от 1 до 8 МБ).                                                 | Практически безграничный (ограничен только размером ОЗУ + Своп).                                                |
| **Скорость работы** | **Ультра-быстрая.** Процессор просто двигает один регистр-указатель вверх-вниз.                        | **Медленнее.** Ядру нужно искать свободный блок нужного размера, регистрировать его и бороться с фрагментацией. |
