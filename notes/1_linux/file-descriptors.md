## Файловые дескрипторы

Файловый дескриптор (file descriptor, FD) — это **целое неотрицательное число**, которое операционная система (ядро Linux) использует для идентификации **открытого файла**, **сокета**, **канала (pipe)** или любого другого **ресурса ввода-вывода (I/O)** внутри процесса .

#### Как это выглядит на практике?

Когда процесс запускается, он по умолчанию получает три предопределенных (базовых) файловых дескриптора от ядра:

| Канал | Имя               | Назначение                                  |
| ----- | ----------------- | ------------------------------------------- |
| **0** | `STDIN` (stdin)   | Стандартный поток ввода (обычно клавиатура) |
| **1** | `STDOUT` (stdout) | Стандартный поток вывода (обычно экран)     |
| **2** | `STDERR` (stderr) | Стандартный поток ошибок (обычно экран)     |

Любые другие файлы, сетевые соединения или устройства, к которым обращается программа по ходу своей работы, получают новые порядковые номера: `3`, `4`, `5`, `6` и так далее.

Как только программа делает системный вызов `open()` (открывает файл на диске) или создает сетевое подключение (сокет), Linux ищет первый свободный номер в таблице процесса и присваивает его этому объекту.

Представьте веб-сервер (например, Nginx). При старте у него открыты стандартные `0`, `1`, `2`. Затем он начинает работать:

- Дескриптор **`3`** — он открывает конфигурационный файл `nginx.conf`.
- Дескриптор **`4`** — он открывает файл логов `access.log` для записи.
- Дескриптор **`5`** — он открывает сетевой сокет, чтобы слушать порт 80 (интернет-трафик).
- Дескрипторы **`6`, `7`, `8`... `1002`** — это новые сетевые сокеты для каждого пользователя, который прямо сейчас зашел на сайт.

Когда пользователь закрывает вкладку сайта, сервер закрывает сокет, и этот номер дескриптора снова становится свободным для следующего гостя.

Файловые дескрипторы лежат в основе всего сетевого и файлового взаимодействия в Linux. Когда процессу нужно выполнить операцию ввода-вывода (I/O), он не передает ядру путь к файлу или адрес сокета каждый раз. Вместо этого он использует короткое, эффективное **число**, которое ядро выдало при открытии ресурса.

**Основные системные вызовы (syscalls), работающие с FD:**
- **`open()` / `socket()` / `pipe()`:** Создают ресурс и возвращают новый файловый дескриптор.
- **`read()` / `write()`:** Используют FD для чтения или записи данных в ресурс.
- **`close()`:** Говорят ядру, что процесс закончил работу с ресурсом, и FD можно освободить.

Концепция «все есть файл» в Linux работает на полную. FD используется не только для обычных файлов:

- **Сокеты (Sockets):** В веб-сервере (Nginx, Apache) каждое новое соединение клиента — это новый сокет, т.е. новый FD. Ошибка `too many open files` для сервера — часто результат того, что он не справляется с количеством одновременных подключений. 

- **Каналы (Pipes):** Когда вы пишете `command1 | command2`, создается канал. `stdout` первой команды (FD 1) подключается к `stdin` второй команды (FD 0) через этот канал.

- **Epoll:** Это специальный механизм, когда процесс создает FD для наблюдения за множеством других FD («список наблюдения»). Когда на любом из наблюдаемых FD происходит событие (появились данные, закрылся сокет), ядро сообщает об этом через epoll FD. Это основа работы высокопроизводительных серверов (Nginx, Node.js) в Linux.

#### Примеры работы дескрипторов и перенаправления в консоли

**Пример в bash:** Их можно увидеть в `/proc/<PID>/fd/`.

```bash
# Запустите команду sleep в фоне
sleep 300 &
[1] 12345
# Посмотрите дескрипторы процесса
ls -l /proc/12345/fd/
```

**Вывод будет похож на этот:**

```text
lrwx------ 1 user user 64 May 30 10:00 0 -> /dev/pts/0
lrwx------ 1 user user 64 May 30 10:00 1 -> /dev/pts/0
lrwx------ 1 user user 64 May 30 10:00 2 -> /dev/pts/0
```

Это показывает, что все три стандартных потока связаны с текущим терминалом `/dev/pts/0`

##### Простой пример (На языке C)

Допустим, вы хотите открыть файл `test.txt`. Вот пример кода на C, который наглядно это демонстрирует:

```c
#include <fcntl.h>
#include <stdio.h>
#include <unistd.h>
int main() {
    // Попытка открыть файл. open() возвращает файловый дескриптор (fd).
    int fd = open("test.txt", O_RDWR | O_CREAT, 0644);
    
    if (fd == -1) {
        // Ошибка открытия, fd = -1.
        perror("open");
        return 1;
    }
    
    printf("Файл открыт, файловый дескриптор: %d\n", fd);
    
    // Пишем что-то в файл, используя fd.
    write(fd, "Hello, file descriptor!\n", 23);
    
    // Закрываем файл, освобождая fd.
    close(fd);
    printf("Файл закрыт.\n");
    
    return 0;
}
```

**Вывод:**

```text
Файл открыт, файловый дескриптор: 3
Файл закрыт.
```

Первый открытый файл для процесса, помимо стандартных потоков (0,1,2), получает следующий доступный номер — **3**.

> Перенаправления логов

```bash
tar -czf backup.tar.gz /var/www/ 1> success.log 2> errors.log
```

`1`, `2` - это номера соответствующих дескрипторов STDOUT, STDERR и их перенаправление в соответствующие файлы.

- Например `> /dev/null` — отправляет дескриптор `1` (успешный вывод) в «черную дыру».
- Или `>./alllogs 2>&1` — отправляет `1` в файл `alllogs`, а `2` отправляет по ссылке туда же, куда и `1`.
- `&>` — синтаксический сахар, означает тоже самое что и `> file 2>&1`, на старых оболочках типо `sh` не работает.
- `command >> file.log` — дописывает успешный вывод в конец файла.
- `command 2>> errors.log` — дописывает только ошибки в конец файла.
- `command >> file.log 2>&1` — дописывает и вывод, и ошибки в один файл.

> Механизм Пайпов / Конвейеров (Конвейер данных через `|`)

Пайп (символ вертикальной черты `|`) — это мощнейший инструмент Linux. На уровне дескрипторов он делает следующее: **забирает дескриптор `1` (stdout) первой программы и втыкает его в дескриптор `0` (stdin) второй программы.**

```bash
cat /var/log/nginx/access.log | grep "404" | wc -l
```

1. `cat` читает лог и выводит его в дескриптор `1`.
2. Пайп перенаправляет этот вывод в дескриптор `0` команды `grep`.
3. `grep` фильтрует только строки с ошибкой "404" и выдает их в свой дескриптор `1`.
4. Второй пайп передает эти строки в дескриптор `0` команды `wc -l`, которая считает количество строк.  

> Одновременный вывод на экран и в файл (`tee`)

Если вы перенаправляете вывод через `>`, экран остается пустым. Если вы хотите **и видеть, что происходит в терминале, и одновременно сохранять это в лог**, используется утилита-тройник `tee`:

```bash
make compile 2>&1 | tee build.log
```

_Конструкция `2>&1 |` сливает ошибки с выводом и отправляет их в пайп, а `tee` раздваивает этот поток: одну копию пускает на экран, вторую — записывает в `build.log`._

> Чтение из строки на лету (Here-string: `<<<`)

Обычно для передачи текста в программу используют пайп: `echo "my text" | grep "text"`. Но в Bash можно передать строку напрямую в дескриптор `0` (ввод) без запуска лишнего процесса `echo`:

```bash
grep "text" <<< "my text"
```

> Передача многострочного текста (Here-doc: `<<`)

Используется в скриптах, когда нужно записать целый блок текста или конфигурации в файл из самого скрипта. Вы сами задаете слово-маркер (обычно `EOF` — End Of File):

```bash
cat << EOF > config.conf
server_name=localhost
port=80
db_user=root
EOF
```


Без дескрипторов нам пришлось бы сохранять промежуточные результаты в кучу временных файлов на диске.

#### Утилита `lsof` для просмотра всех дескрипторов

```bash
lsof -p <PID>      # Для процесса
lsof /path/to/file # Для файла
lsof -i :8080      # Для порта
```

Вы увидите список всех открытых FD для процесса (`cwd`, `txt`, `mem`, `3u`, `4u` и т.д.). Это незаменимая команда для диагностики утечек.

```bash
ls -l /proc/<id>/fd/   # В файловой системе процессов
```

Почему в `/proc/[PID]/fd` меньше строк, чем в выводе `lsof`?

Директория `/proc/[PID]/fd/` содержит **только классические числовые дескрипторы** (`0`, `1`, `2`, `3`...), которые программа открыла через явные системные вызовы вроде `open()` или `socket()`.

Полноценные системные маркеры (`cwd`, `txt`, `mem`, `rtd`) там **не показываются**, потому что ядро Linux не выделяет под них номера в таблице дескрипторов процесса. Но они отображаются в других файлах внутри `/proc`:

- **`cwd`** — в `/proc/[PID]/` есть отдельная символическая ссылка `cwd`, ведущая в текущую папку.
- **`txt`** — там же есть ссылка `exe`, указывающая на исполняемый файл.
- **`mem`** — список замапленных библиотек хранится в текстовом файле `/proc/[PID]/maps`.

Утилита `lsof` (List Open Files) делает огромную аналитическую работу: она заглядывает во _все_ эти скрытые файлы и ссылки внутри `/proc`, собирает информацию воедино и выводит вам красивую общую таблицу, где склеивает и числовые дескрипторы, и системные мета-связи (`cwd`, `txt`).

#### Текстовые маркеры (Специальные дескрипторы)

Этим объектам ядро Linux не присваивает стандартные порядковые номера `0`, `1`, `2`, но процессу они критически важны для работы:

- **`cwd` (Current Working Directory)** — Текущая рабочая директория. Это папка, внутри которой процесс сейчас «находится». Если вы в терминале выполните `cd /var/log`, то для этого терминала `cwd` станет указывать на `/var/log`.
- **`txt` (Text / Executable code)** — Программный код. Это ссылка на сам бинарный исполняемый файл программы (например, файл `/usr/bin/nginx`), который сейчас загружен в память и выполняется.
- **`mem` (Memory-mapped file)** — Файл, проецируемый в память. Обычно это системные библиотеки (`.so`-файлы в Linux), которые программа замапила в ОЗУ для быстрого вызова общих функций.
- **`rtd` (Root Directory)** — Корневая директория процесса. Обычно это просто `/`, но может быть другой папкой, если процесс запущен в изолированном окружении через `chroot`.

#### Цифровые дескрипторы с буквами (`3u`, `4u`, `0r`, `1w`)

Когда вы видите число, за которым следует буква — это **порядковый номер дескриптора** (из таблицы процесса) + **режим, в котором этот файл открыт**.

Буквы после цифры означают следующее:

- **`u` (read and write / универсальный)** — Файл открыт одновременно и на чтение, и на запись. В вашем примере `3u` и `4u` означают: дескриптор №3 и дескриптор №4 открыты в двустороннем режиме (так часто открываются сетевые сокеты или терминалы).
- **`r` (read)** — Файл открыт строго только для чтения (например, `0r` — стандартный ввод читает данные).
- **`w` (write)** — Файл открыт строго только для записи (например, `1w` — стандартный вывод пишет логи или данные в файл).
- **`W` (write lock)** — Процесс открыл файл на запись и заблокировал его, чтобы другие программы не могли параллельно в него писать и портить данные.


- **Просмотр лимитов процесса:**

```bash
cat /proc/<PID>/limits
```

Здесь, в частности, можно увидеть параметр **`Max open files`** — максимальное количество файлов, которое может открыть процесс. Его превышение приводит к знаменитой ошибке **`too many open files`**.

Как проверить свои лимиты?

В консоли Linux можно узнать ограничения текущего пользователя с помощью команды `ulimit`:
`ulimit -Sn` — **мягкий лимит** (soft limit). Это ограничение, которое программа может сама временно увеличить при необходимости.
`ulimit -Hn` — **жесткий лимит** (hard limit). Потолок, выше которого обычный пользователь прыгнуть не может (его может поднять только `root` в системных настройках).


Файловые дескрипторы дают **единый абстрактный интерфейс**. Системе и программам абсолютно всё равно, куда они пишут данные: в файл на SSD, в консоль пользователю, в сетевой сокет на другой конец планеты или в другую программу. Для кода это всегда просто операция `write(дескриптор_номера, данные)`

## Системная таблица открытых файлов

Когда программа открывает файл или сокет, ядро Linux не просто выдает ей случайное число дескриптора. На самом деле в глубинах ядра создается целая **трехслойная структура данных ввода-вывода**.

**Системная таблица открытых файлов** (System-wide Open File Table) — это центральный, «средний» слой этой структуры. Она едина для всей операционной системы и хранит состояние абсолютно всех файлов, которые считываются или пишутся прямо сейчас.

Чтобы понять, зачем она нужна и как работает, давайте посмотрим на все три слоя, сверху вниз:

> **Слой 1: Таблица дескрипторов процесса (Локальная)**

У каждого процесса (например, у вашего Python-скрипта или браузера) есть своя личная таблица.

- Индексы в этой таблице — это и есть те самые **числа дескрипторов (`0`, `1`, `2`, `3`...)**.
- В этой таблице нет информации о файле. В ней содержатся только флаги (например, «закрывать ли этот дескриптор при запуске другой программы») и **указатель (ссылка)**, ведущий на второй слой.
- Локальная таблица дескрипторов процесса полностью материализована в виртуальной папке `/proc/[PID]/fd/`. Каждая строка в этой таблице представлена в виде символической ссылки.

```bash
ls -l /proc/self/fd/
```

Вывод будет подобный:  
`lrwx------ 1 user user 64 Jun 6 16:00 0 -> /dev/pts/0`

- **Цифра `0`** — это индекс (строка) в локальной таблице дескрипторов процесса.
- **Стрелочка `->`** — это и есть визуализированный ядром **указатель** на системный объект открытого файла.
- **`/dev/pts/0`** — это имя файла (или устройства), к которому привязана строка в системной таблице открытых файлов.


> **Слой 2: Системная таблица открытых файлов (Глобальная)**

Она одна на всю систему. Каждый раз, когда любая программа вызывает `open()`, в этой таблице создается новая строка (запись).

Она хранит динамическое состояние работы с файлом:

1. **Текущее смещение (File Offset / Позиция чтения-записи):** Это указатель на байт, у которого сейчас остановилась программа. Если программа прочитала первые 100 байт, смещение станет равно `100`. Следующий вызов `read()` начнется со 101-го байта.
2. **Статусные флаги:** В каком режиме открыт файл (`O_RDONLY` — только чтение, `O_APPEND` — дописывать в конец, `O_NONBLOCK` — неблокирующий ввод-вывод).
3. **Счетчик ссылок:** Сколько дескрипторов из разных (или одного) процессов прямо сейчас указывают на эту конкретную строку.
4. **Указатель на третий слой** (таблицу Inode).

Посмотреть на саму систему можно несколькими способами:

```bash
cat /proc/sys/fs/file-nr
```

Вывод будет по типу:  
`1536 0 9223372036854775807`

**Как это читать (это и есть мета-параметры системной таблицы):**

1. **Первое число (`1536`)** — Количество записей (строк), занятых в системной таблице открытых файлов _прямо сейчас_ по всей операционной системе.
2. **Второе число (`0`)** — Количество выделенных, но освободившихся строк (в современных ядрах Linux этот показатель часто равен 0, так как память выделяется динамически).
3. **Третье число (`9223372036854775807`)** — Максимально возможный размер системной таблицы открытых файлов (`fs.file-max`). Выше этого значения операционная система физически не сможет открыть ни одного файла или сокета.

Чтобы буквально сделать дамп всей системной таблицы открытых файлов:

```bash
sudo lsof
```

У каждого процесса есть структура `task_struct` (паспорт процесса). Внутри нее есть указатель на структуру `files_struct` (Локальная таблица). А внутри нее лежит массив указателей:

Каждый элемент этого массива `fd_array[0]`, `fd_array[1]` — это и есть дескриптор. А тип `struct file *` — это и есть **указатель-ссылка** на строку в Глобальной системной таблице открытых файлов.


> **Слой 3: Таблица Inode (Глобальная виртуальная память VFS)**

Это кэш «паспортов» файлов в оперативной памяти. Здесь хранится статическая информация о файле: его тип, права доступа, размер и где его блоки лежат на физическом диске.

---

Такая трехслойная схема создана ради реализации ключевых фишек Unix/POSIX-систем. Это открывает потрясающие возможности:

А. Разделение позиции чтения между процессами (`fork()`)

Когда процесс создает свою копию с помощью функции `fork()`, дочерний процесс полностью копирует локальную Таблицу 1. В итоге дескриптор `3` у родителя и дескриптор `3` у потомка начинают указывать на **одну и ту же строку** в системной Таблице 2.

- _Эффект:_ Если родитель прочитал из файла 50 байт (смещение сдвинулось на 50), то потомок автоматически продолжит читать с 51-го байта. Они не мешают друг другу и читают файл как одну общую очередь.

Б. Независимое чтение одного файла (`open()` + `open()`)

Если две абсолютно разные программы (например, два плеера) одновременно открывают один и тот же файл `video.mp4`:

- В системной Таблице 2 создаются **две разные строки**. У каждого плеера будет свое собственное смещение (один смотрит на 5-й минуте, второй — на 40-й).
- Но обе эти строки в Таблице 2 будут указывать на **один и тот же Inode** в Таблице 3. Система знает, что файл физически один, права доступа проверяются один раз, а данные кэшируются в памяти совместно.

Поэтому в Linux существует несколько разных лимитов на открытые файлы:

1. Лимит на уровне `ulimit -n` (например, 1024) — это максимальный размер **Локальной таблицы (Слой 1)** для конкретного процесса.
2. Системный лимит `fs.file-max` (можно посмотреть через `cat /proc/sys/fs/file-max`) — это максимальная вместимость всей **Глобальной таблицы открытых файлов (Слой 2)** для всего ядра Linux. Если сервер откроет миллионы сокетов, место закончится именно в этой системной таблице.
