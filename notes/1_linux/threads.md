## Потоки  (Thread / Нить)

Поток — это наименьшая единица выполнения, которой операционная система может выделять процессорное время. Поток не может существовать самостоятельно, это как рабочая сила процесса. Планировщик Linux (`Scheduler`) распределяет кванты времени процессора **исключительно между потоками**.

Для чего тогда ядро вообще объединяет их в «Процессы»?

Раз планировщик работает только с потоками, зачем ядру помнить, к какому процессу они относятся? Это нужно для трех глобальных вещей, которые потоки сами по себе контролировать не могут:

- **Для учета ресурсов и лимитов (Менеджмент):** Ядро должно знать, сколько суммарно памяти «сожрали» все повара-потоки на этой кухне. Если лимит процесса превышен, ядро накажет весь процесс целиком.
- **Для безопасности и прав доступа:** Все потоки внутри одного процесса обязаны иметь один и тот же `UID` (идентификатор пользователя). Поток не может сам по себе стать «рутовым», если процесс запущен от обычного пользователя.
- **Для доставки Сигналов:** Если вы пишете в консоли `kill [PID]`, вы шлете сигнал процессу. Ядро перехватывает его, находит группу связанных потоков и решает, как их остановить.

По спецификации POSIX у любого процесса **не может быть 0 потоков**. Как минимум один поток есть всегда — он называется **главным потоком (main thread)**. Когда вы запускаете программу, этот главный поток начинает по строчкам выполнять функцию `main()`

#### Как потоки и процессы делят память

-  Процессы НЕ делят память между собой

У каждого процесса своя личная, изолированная Таблица страниц. Процесс `А` физически не видит Кучу (Heap) процесса `Б`. Чтобы передать данные от одного процесса к другому, ядру приходится использовать сложные механизмы межпроцессного взаимодействия (IPC) — например, гонять данные через сокеты или пайпы.

- Потоки внутри одного процесса ДЕЛЯТ память совместно

Все потоки, запущенные внутри одного процесса, **используют одну и ту же Таблицу страниц** этого процесса.

- **Куча (Heap) — ОБЩАЯ:** Если поток №1 создал в Куче большой список объектов, поток №2 может мгновенно прочитать или изменить эти данные, просто обратившись по адресу (указателю). Им не нужно просить ядро пересылать данные.
- **Сегмент кода (Text) и данных (Data) — ОБЩИЕ:** Все потоки выполняют один и тот же бинарный код программы и видят одни и те же глобальные переменные.
- **Таблица дескрипторов файлов — ОБЩАЯ:** Если поток №1 открыл сокет или файл и получил дескриптор `3`, поток №2 может сразу писать в этот дескриптор `3`.


Потоки не могут быть абсолютно одинаковыми, иначе они бы делали одно и то же. У каждого потока есть свой изолированный **микро-контекст**:

1. **Собственный Стек (Stack):** Это критически важно. У каждого потока своя личная «разделочная доска» для локальных переменных функций, чтобы они не перемешивались.
2. **Собственный указатель инструкций (PC — Program Counter):** Поток №1 может выполнять 45-ю строку кода, пока поток №2 в этот же микросекундный момент обрабатывает 120-ю строку кода.
3. **Регистры процессора:** Индивидуальные промежуточные вычисления, которые процессор сохраняет при переключении между потоками.

Есть однопоточные процессы и многопоточные. Потоки в системе это обычно колличество ядер процессора х2. На потоки можно посмотреть несколькими способами:

1) Запустить в терминале утилиту `htop`.

- Нажать клавишу **`F2`** (Настройки) -> перейти в раздел **Display options** -> отметить галочкой **Three view** (Древовидный вид) и снять галочку с _Hide userland process threads_. Зеленым будут отображатся потоки.

2) `ps -T -p [PID_процесса]` флаг `-T` добавляет колонку `SPID` - id потока

3) Через виртуальную систему `/proc`. Внутри нее есть директория `task`. Количество папок внутри `task` — это и есть все потоки процесса:

```bash
ls /proc/self/task
```

На обычном работающем ноутбуке или сервере Linux крутится в среднем **от 100 до 300 процессов**, но суммарное количество потоков в этих процессах обычно составляет **от 1000 до 4000 штук**. Большая часть из них спит, ожидая действий от пользователя или сети.

Физический лимит на максимальное количество потоков во всей системе можно посмотреть командой:

```bash
cat /proc/sys/kernel/threads-max
```

**Раз от потоков только плюсы к увеличению скорости работы программы, почему для каждой программы не использовать их огромное колличество?**

В компьютерной инженерии увеличение числа потоков работает на пользу скорости только до определенного предела. Дальше начинается жесткий спад из-за трех главных причин:

- А. Физическое ограничение ядер CPU

Потоки выполняются на физических ядрах вашего процессора. Если у вашего процессора 8 ядер (или 16 потоков гипертрединга), то в один микросекундный момент времени компьютер может физически выполнять **ровно 16 потоков**.  
Если вы запустите 100 потоков, процессору придется непрерывно переключаться между ними. Это называется **оверхед на переключение контекста (Context Switch Overhead)**. Процессор будет тратить больше сил и времени на сохранение/загрузку регистров и стеков этих 100 потоков, чем на выполнение полезного кода программы.

- Б. Закон Амдала (Не всё можно распараллелить)

Представьте, что 1 повар варит борщ за 60 минут. Если мы наймем 60 поваров, сварят ли они борщ за 1 минуту? Нет. Потому что картошка физически не сварится быстрее, а вода в кастрюле закипает фиксированное время.  
В программах то же самое: если коду нужно дождаться ответа от медленного жесткого диска или скачать файл из интернета, 100 потоков будут просто одновременно стоять и спать в RAM, ожидая этот диск. Скорость не вырастет ни на грамм.

- В. Потребление памяти

Каждый поток — хоть и легковесный, но требует ресурсов. Под **Стек** (локальную память) каждого потока операционная система сразу резервирует фиксированный объем (в Linux по умолчанию около 8 МБ, в Windows — 1 МБ). Если вы создадите 1000 потоков, программа мгновенно «сожрет» гигабайты оперативной памяти только под пустые стеки, даже не начав вычисления.

Как понять, сколько потоков может выполнять процессор ОДНОВРЕМЕННО?

Количество потоков, которые компьютер выполняет **физически в один и тот же миг**, строго равно количеству **логических ядер** (потоков) вашего CPU.

Вы можете узнать это число в Linux одной командой:

```bash
nproc
```

Поскольку потоков тысячи, а ядер всего 12, ядро Linux использует технологию **разделения времени** (Time Sharing).

Процессор выполняет потоки не «параллельно», а **по очереди, но с бешеной скоростью**. Он берет поток №1, дает ему поработать крошечный квант времени, затем замораживает его, загружает поток №2, дает поработать ему, и так далее по кругу. Для человека это выглядит так, будто музыка играет, браузер качает, а мышка двигается одновременно.

Время, которое планировщик Linux выделяет одному потоку на CPU до того, как переключить его на другой, называется **квантом времени** (Time Slice или Quantum).

- **Каков размер этого кванта?**  
	В современном Linux размер кванта времени динамический и обычно составляет **от 1 до 10 миллисекунд** (1 мс = 1/1000 доля секунды).
- **От чего зависит это время?**  
	Размер кванта вычисляется планировщиком (в Linux это алгоритм CFS — Completely Fair Scheduler) «на лету» для каждого потока отдельно на основе двух факторов:
	1. **Приоритет (`Nice`):** Чем более «невежливый» процесс (ближе к `-20`), тем больший квант времени (например, 10 мс) ему дадут. «Вежливому» (ближе к `19`) дадут всего 1 мс.
	2. **Поведение потока:** Если поток интерактивный (ждет нажатия клавиш), ядро дает ему кванты часто, но маленькие, чтобы интерфейс не дергался. Если поток пакетный (считает математику), ядро дает ему длинный квант, чтобы он реже прерывался.
