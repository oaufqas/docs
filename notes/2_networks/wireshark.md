#### Основы

Wireshark – графический анализатор сетевого трафика (сниффер). Позволяет захватывать пакеты, проходящие через сетевой интерфейс, и детально их анализировать.

**Принцип работы:** сетевой адаптер переводится в **неразборчивый режим (promiscuous mode)**, и все пакеты, которые видит интерфейс, передаются Wireshark.
#### Фильтры отображения (Display Filters)
Мощный язык для поиска нужных пакетов уже после захвата.

**Примеры:**

- `ip.addr == 192.168.1.1` – пакеты с этим IP (источник или назначение)
- `tcp.port == 80` – TCP-трафик на/с порта 80
- `http` – только HTTP-пакеты
- `tls.handshake.extensions_server_name == "example.com"` – SNI (имя сервера в TLS)
- `dns.qry.name contains "google"` – DNS-запросы, содержащие "google" 
- Логические операторы: `and`, `or`, `not`.
#### Следование потоку (Follow Stream)

Позволяет собрать все пакеты одного TCP- или UDP-соединения и увидеть их как непрерывный поток данных (например, восстановить переписку по HTTP). ПКМ по пакету → Follow → TCP Stream (или HTTP Stream).

**Wireshark показывает только TCP, а не HTTPS**

Wireshark показывает **ВСЕ уровни**, но HTTPS "спрятан" внутри TCP по двум причинам:

|Причина|Что это значит|
|---|---|
|**Шифрование (HTTPS)**|HTTP завернут в TLS/SSL. Wireshark видит только зашифрованный поток. Это как смотреть на запечатанный конверт — вы видите, что письмо есть, но прочитать не можете.|
|**HTTP/2**|Использует бинарный формат, не такой наглядный, как HTTP/1.1.|

HTTPS пакеты будут показаны либо как TCP с Length 100+, если wireshark не захватил начало диалога, либо как зашифрованные TLSv1.3. Чтобы увидеть зашифрованные данные, нужно сохранять ключи сессий.

Незашифрованные HTTP так и будут показываться в списке, но при обычной работе их никто не использует, так что они там появляться почти что не будут.

#### Итоговая таблица: Уровни и их содержимое

| Уровень OSI      | Протокол   | Что содержит                            | В Wireshark                     |
| ---------------- | ---------- | --------------------------------------- | ------------------------------- |
| **Канальный**    | Ethernet   | MAC-адреса отправителя и получателя     | `Ethernet II`                   |
| **Сетевой**      | IP         | IP-адреса отправителя и получателя, TTL | `Internet Protocol Version 4`   |
| **Транспортный** | TCP/UDP    | Порты, порядковые номера, флаги         | `Transmission Control Protocol` |
| **Прикладной**   | HTTP/HTTPS | Веб-страницы, API-запросы               | `Hypertext Transfer Protocol`   |

##### Wireshark помечает пакет **самым верхним протоколом, который он смог распознать**. А "ниже" TCP — это просто потому, что Wireshark сворачивает (скрывает) нижние уровни, чтобы не загромождать экран.

- Если это просто IP-пакет без TCP/UDP (например, ICMP) → будет `ICMP`
- Если это TCP-пакет → будет `TCP`
- Если внутри TCP распознан HTTP → будет `HTTP`
- Если внутри TCP распознан TLS → будет `TLSv1.2`

#### Почему в большинстве пакетов нет ничего "выше" TCP?

Потому что **нет данных прикладного уровня** и это служебный пакет. Например:

|Тип пакета|Есть ли HTTP?|Почему|
|---|---|---|
|**TCP SYN** (начало соединения)|❌ Нет|Это просто служебный пакет для установки соединения, данных нет|
|**TCP ACK** (подтверждение)|❌ Нет|Тоже служебный, только подтверждает получение|
|**TCP PSH, ACK** (с данными)|✅ Может быть|Если внутри есть HTTP-запрос или ответ|
|**TCP FIN** (закрытие)|❌ Нет|Служебный|

Либо Wireshark не распознал уровень выше и данные будут в поле `TCP segment data` на 3 уровне сети.

## TCP пакеты

![[Pasted image 20260603014701.png]]

- Source Port - порт отправителя, рандомно назначается операционной системой при подключении
- Destination Port - порт получателя, 443 = HTTPS порт
- Sequence Number - номер байта именно в текущем соединении
- Sequence Number (raw) - номер байта в потоке данных, **который отправляют**
- Acknowledgment Number - номер байта в потоке данных, **который ожидают**, байты до него, подтверждены
- Flags - байты флагов пакета
- Window - размер скользящего окна
- Checksum - контрольная сумма

#### Расшифровка TLS пакетов

Чтобы расшифровать TLS пакеты, нужно сохранять ключи сессий браузера ([[ssl-tls#Decryption TLS|в файле]]).

Чтобы созданный лог-файл начал использоваться для расшифровки трафика:

1. Перейдите в меню **Edit (Правка)** -> **Preferences (Настройки)**.
2. Раскройте ветку **Protocols** и найдите пункт **TLS**.
3. В поле **(Pre)-Master-Secret log filename** укажите путь к вашему файлу лога

После этих действий, захваченные пакеты TLS --> HTTP, но при условии что все пакеты сессии были захвачены wireshark'ом

### Установка TLS соединения

![[Pasted image 20260601151138.png]]

1) Client Hello + SNI (Server Name Indication) это первый пакет, запрос на создание сессии. SNI позволяет клиенту (например, вашему браузеру) сообщить имя сайта, к которому он хочет подключиться, еще до того, как начнется шифрование соединения, также он позволяет размещать сотни различных сайтов с разными сертификатами на одном IP-адресе.

![[Pasted image 20260601152009.png]]

TLS Record Layer - протокол записи, базовый протокол сообщений, в нем есть 3 поля:
- Content Type - тип протокола (для соединения используется handshake protocol).
- Version: TLS 1.0 - всегда указывается эта версия, о том какая будет использоваться в сессии, договариваются клиент и сервер в процессе установки соединения.
- Length: 1754 - длина сообщения

![[Pasted image 20260601152629.png]]

Это уже сам Handshake protocol:
- Handshake Type - указывается тип сообщения (Client Hello)
- Length: 1750 - длина сообщения
- Дальше идут параметры, варианты поддерживаемых технологий для подключения:
	- Cipher Suites - поддерживаемые TLS шифры (в TLSv1.3 - 5 поддерживаемых шифров)
	- Extension: supported_versions - поддерживаемые версии TLS
	- Extension: server_name - имя сервера, к которому идет подключение - **это и есть SNI**
	- Extension: key_share - сам обмен ключами, клиент отправляет 2 варианта информации для создания симметричных ключей



2) Сервер отправляет ответ Server Hello вместе с зашифрованной информацией, сертификатами, finished. Он может разделить ответ на несколько пакетов, в этом случае 2.

![[Pasted image 20260601155748.png]]
![[Pasted image 20260601155718.png]]

Внутри одного ответа сервера, 6 внутренних сообщений:

- Handshake Protocol: Server Hello
	- Первое что содержится в ответе сервера, почти такой же Server Hello
	- Extension: supported_versions - сервер выбрал версию 1.3
	- Extension: key_share - информация для создания симметричных ключей
	- Cipher Suite - сервер выбрал TLS шифр
- Change Cipher Spec Protocol: Change Cipher Spec - это означает, что все сообщения дальше - зашифрованы
- Handshake Protocol: Encrypted Extensions - зашифрованные расширения
- Handshake Protocol: Certificate - зашифрованные сертификаты (2 шт):
	- первый - сертификат сервера (Lets Encrypt)
		- issuer - кто выдал сертификат
		- validity - валидный ли сертификат, с какого по какое действует
		- subject - кому выдан сертификат (домен сайта должен быть)
		- subjectPublicKey - сам внешний ключ сервера
	- второй - сертификат самой организации Lets Encrypt
		- issuer - корневой удостоверяющий центр, которому доверяет OS.
- Handshake Protocol: Certificate Verify - удостоверяющее сообщение о том, что сервер владеет закрытым ключем, алгоритм цифровой подписи
- Handshake Protocol: Finished - обеспечение целостности процесса соединения. сервер берет все сообщения, считает хэш и отправляет. клиент делает тоже самое, хэши должны совпасть



3) Сервер отправляет тикет сессии, можно использовать если в дальнейшем клиент захочет возобновить эту сессию и использовать те же ключи. (0-RTT)

![[Pasted image 20260601162506.png]]

Handshake Protocol: New Session Ticket:
- Session Ticket Lifetime Hint - сколько действует этот тикет
- Session Ticket - сам ключ для восстановления сессии



4) Клиент отправляет Finished, после этого соединение установлено, начинают отправляться данные

![[Pasted image 20260601163041.png]]

- Change Cipher Spec - значит что после этого, все сообщения клиента зашифрованы
- Handshake Protocol: Finished - проверка целостности всего процесса установки соединения

### HTTP Protocol

![[Pasted image 20260601183032.png]]

После расшифровки, пакеты выглядят как обычные HTTP:

Первый пакет из списка на фото является **(Connection Preface)** в протоколе HTTP/2, которая служит обязательным индикатором начала HTTP/2-соединения.

Строка `PRI * HTTP/2.0\r\n\r\nSM\r\n\r\n` отправляется клиентом сразу после успешного TLS-рукопожатия, чтобы сервер окончательно понял: клиент будет общаться строго по протоколу HTTP/2, а не HTTP/1.1. В спецификации RFC 7540 (HTTP/2) эта последовательность байтов прописана жестко. В шестнадцатеричном виде (Hex) она выглядит так:  
`505249202a20485454502f322e300d0a0d0a534d0d0a0d0a`

Если перевести её в текст, получается ваша строка, где каждый элемент имеет смысл:

- **`PRI * HTTP/2.0`** — имитация HTTP/1.1-запроса с вымышленным методом `PRI` (от слова _Priority_). Это сделано для старых серверов: если они случайно получат этот пакет, они не сломаются, а просто вернут понятную ошибку вроде `400 Bad Request`.
- **`\r\n\r\n`** — два переноса строки (CRLF), стандартный разделитель заголовков в веб-протоколах.
- **`SM`** — аббревиатура от _SpiderMan_ (изначально HTTP/2 разрабатывался на базе протокола SPDY от Google, и разработчики оставили эту пасхалку-идентификатор).
- **`\r\n\r\n`** — финальные два переноса строки.


В терминологии HTTP/2 всё общение разделено на параллельные потоки (**Streams**).

- Потоки с данными (запросы сайтов, картинок) имеют номера `1`, `3`, `5` и так далее.
- Поток `0` резервируется для управления самим соединением.

Поскольку «магическая строка» отправляется в самом начале, когда полноценные потоки еще не созданы, анализаторы трафика группируют этот пакет под техническим именем **Stream: Magic**, подчеркивая, что это не текстовый запрос, а бинарная преамбула.
