#### Основы

Wireshark – графический анализатор сетевого трафика (сниффер). Позволяет захватывать пакеты, проходящие через сетевой интерфейс, и детально их анализировать.

**Принцип работы:** сетевой адаптер переводится в **неразборчивый режим (promiscuous mode)**, и все пакеты, которые видит интерфейс, передаются Wireshark.
#### Фильтры отображения (Display Filters)
Мощный язык для поиска нужных пакетов уже после захвата.

**Примеры:**

- `ip.addr == 192.168.1.1` – пакеты с этим IP (источник или назначение)
- `tcp.port == 80` – TCP-трафик на/с порта 80
- `http` – только HTTP-пакеты
- `tls.handshake.extensions_server_name == "example.com"` – SNI (имя сервера в TLS)
- `dns.qry.name contains "google"` – DNS-запросы, содержащие "google" 
- Логические операторы: `and`, `or`, `not`.
#### Следование потоку (Follow Stream)

Позволяет собрать все пакеты одного TCP- или UDP-соединения и увидеть их как непрерывный поток данных (например, восстановить переписку по HTTP). ПКМ по пакету → Follow → TCP Stream (или HTTP Stream).

**Wireshark показывает только TCP, а не HTTPS**

Wireshark показывает **ВСЕ уровни**, но HTTPS "спрятан" внутри TCP по двум причинам:

|Причина|Что это значит|
|---|---|
|**Шифрование (HTTPS)**|HTTP завернут в TLS/SSL. Wireshark видит только зашифрованный поток. Это как смотреть на запечатанный конверт — вы видите, что письмо есть, но прочитать не можете.|
|**HTTP/2**|Использует бинарный формат, не такой наглядный, как HTTP/1.1.|

HTTPS пакеты будут показаны либо как TCP с Length 100+, если wireshark не захватил начало диалога, либо как зашифрованные TLSv1.3. Чтобы увидеть зашифрованные данные, нужно сохранять ключи сессий.

Незашифрованные HTTP так и будут показываться в списке, но при обычной работе их никто не использует, так что они там появляться почти что не будут.

#### Итоговая таблица: Уровни и их содержимое

| Уровень OSI      | Протокол   | Что содержит                            | В Wireshark                     |
| ---------------- | ---------- | --------------------------------------- | ------------------------------- |
| **Канальный**    | Ethernet   | MAC-адреса отправителя и получателя     | `Ethernet II`                   |
| **Сетевой**      | IP         | IP-адреса отправителя и получателя, TTL | `Internet Protocol Version 4`   |
| **Транспортный** | TCP/UDP    | Порты, порядковые номера, флаги         | `Transmission Control Protocol` |
| **Прикладной**   | HTTP/HTTPS | Веб-страницы, API-запросы               | `Hypertext Transfer Protocol`   |

##### Wireshark помечает пакет **самым верхним протоколом, который он смог распознать**. А "ниже" TCP — это просто потому, что Wireshark сворачивает (скрывает) нижние уровни, чтобы не загромождать экран.

- Если это просто IP-пакет без TCP/UDP (например, ICMP) → будет `ICMP`
- Если это TCP-пакет → будет `TCP`
- Если внутри TCP распознан HTTP → будет `HTTP`
- Если внутри TCP распознан TLS → будет `TLSv1.2`

#### Почему в большинстве пакетов нет ничего "выше" TCP?

Потому что **нет данных прикладного уровня** и это служебный пакет. Например:

|Тип пакета|Есть ли HTTP?|Почему|
|---|---|---|
|**TCP SYN** (начало соединения)|❌ Нет|Это просто служебный пакет для установки соединения, данных нет|
|**TCP ACK** (подтверждение)|❌ Нет|Тоже служебный, только подтверждает получение|
|**TCP PSH, ACK** (с данными)|✅ Может быть|Если внутри есть HTTP-запрос или ответ|
|**TCP FIN** (закрытие)|❌ Нет|Служебный|

Либо Wireshark не распознал уровень выше и данные будут в поле `TCP segment data` на 3 уровне сети.
