Эти протоколы работают на транспортном уровне, отвечают за доставку данных между конкрутными программами на разных устройствах. Главное, что здесь добавляется к данным — это **заголовки с номерами портов** (отправителя и получателя), которые и указывают, какому приложению (например, браузеру или игре) предназначены пакеты. 
####  Порты - это 16-битные числа (0–65535), идентифицируют приложение на хосте.

- **Well-known ports (0–1023)**: HTTP (80), HTTPS (443), SSH (22), DNS (53)
- **Registered ports (1024–49151)**: для приложений
- **Dynamic/private (49152–65535)**: временные (эпизодические) порты клиентов

Любой транспортный протокол обязательно добавляет три ключевые вещи:

1. **Порт источника (Source Port):** программа, которая отправила данные (например, случайный порт `54321` вашего браузера).
2. **Порт назначения (Destination Port):** программа, которая должна принять данные (например, порт `443` для HTTPS или `53` для DNS).
3. **Контрольная сумма (Checksum):** математический код для проверки, не повредились ли данные при передаче.

# TCP (Transmission Control Protocol)

- **Надёжный**, с установлением соединения (connection-oriented).
- Популярный, имя TCP включено в название эталонной модели и стека.
- Гарантирует доставку, порядок пакетов, управление потоком.
- Использует трёхэтапное рукопожатие (three-way handshake) для установки соединения

- **Заголовок TCP**: от 20 до 60 байт, включает порты, sequence/acknowledge numbers, флаги (SYN, ACK, FIN, RST, PSH, URG), окно (window), контрольную сумму, опции.
- **Управление потоком** – скользящее окно (sliding window).
- **Контроль перегрузки** – алгоритмы (Reno, Cubic).

### Формат заголовка TCP

![[Pasted image 20260601233435.png]]

- Порядковый номер (Sequence number) - номер первого байта в сегменте
- Номер подтверждения (Acknowledgment number) - номер следующего байта, который ожидается в следующем сообщении
- Смещение (Data Offset) - поле для указания, где начинаются данные. нужно для того, чтобы расчитать размер заголовков пакета, тк они имеют необязательную часть параметров
- Флаги:
	- CWR, ECE для явного управления перегрузкой в сети
	- URG говорит о том, что в пакете содержатся важные данные
	- ACK говорит о том, что в поле "номер подтверждения" содержатся осмысленные данные, должен быть установлен всегда, кроме первого этапа установки соединения
	- PSH - данные нужно передать приложению
	- RST, FIN для разрыва соединения
	- SYN для синхронизации соединения
- Размер окна (Window) - размер данных, которые отправитель готов принять
- Контрольная сумма (Checksum) - поле для обнаружения ошибок
- Указатель на срочные данные (Urgent pointer) - используется вместе с флагом URG
- Параметры (options) не обязательно:
	- максимальный размер сегмента
	- масштаб окна
	- выборочное подтверждение
	- метки времени
### Поток байт 

TCP получает от приложения поток байт, и разделяет его на отдельные части - сегменты, и передает по сети отдельные, независимые части

```
 _______________________________________
|         |         |         |         |
| Сегмент | Сегмент | Сегмент | Сегмент |
|_________|_________|_________|_________|
|         |         |         |         |
|  Байт 0 |Байт 1024|Байт 2048|Байт 3072|
|_________|_________|_________|_________|
```

Получатель, на своей стороне собирает эти части в поток байт, передает эти данные приложению

Для того, чтобы сохранить порядок и отстутствие возможности дублирования данных, сегменты нумеруются номерами байт, о нумерации договариваются на стадии соединения, а процесс называется синхронизация:

### Скользящее окно

В протоколе TCP подтверждается не каждый сегмент, а несколько сегментов, отправленных друг за другом в целях повышения производительности, этим занимается механизм скользящего окна.

![[Pasted image 20260601212240.png|423]]

Пример:
- Время приема-передачи 10 мс
- Колличество сегментов в секунду 100
- Размер данных в TCP сегменте 1460б
- Скорость передачи 143 КБ/с

![[Pasted image 20260601224525.png|421]]

Размер окна - колличество байтов данных, которые могут быть переданы без получения подтверждения

Типы подтверждений в TLS:
- Кумулятивное подтверждение:
	- Подтверждение приема всех предыдущих байт, не включая указанный
	- Используется по умолчанию
- Выборочное подтверждение (Selective Acknowledgment)
	- Подтверждение диапазонов принятых байт
	- Эффективно при большом окне
	- Дополнительное поле заголовка TCP (параметр)

### Установка соединения TCP

- Установка соединения
- Передача данных
- Разрыв соединения

При установке соединения, клиент отправляет серверу пакет с флагом SYN и начальным номером байта, с которого потом будет нумерация. Нумерация идет вдухсторонняя, клиент нумерует свои байты, сервер свои. Сервер, на запрос SYN отвечает своим SYN, ACK x+1 байт, и свой начальный номер байта.

![[Pasted image 20260601195304.png|395]]

### Разрыв соединения четырехэтапный:

После того, как клиент отправил флаг FIN, он уже не может отправлять сообщения, но сервер, пока не отправит свой флаг FIN, может отправлять сообщения клиенту.

![[Pasted image 20260601195612.png|372]]

Если соединение закрывается с ошибкой, будет использоваться флаг RST (RESET), который однонаправленно закрывает соединение у обоих сторон. Также он используется после SYN, если сервер не хочет отвечать клиенту.

# UDP (User Datagram Protocol)

- **Ненадёжный**, без установления соединения.
- Не гарантирует доставку, порядок, не контролирует поток.
- Применяется там, где важна скорость: DNS, VoIP, стриминг, игры.
- **Заголовок UDP**: всего 8 байт (порт отправителя, порт получателя, длина, контрольная сумма).

#### Сравнение TCP и UDP

|Характеристика|TCP|UDP|
|---|---|---|
|Соединение|Ориентирован на соединение|Без соединения|
|Надёжность|Гарантирует доставку, порядок|Не гарантирует|
|Контроль потока|Есть|Нет|
|Заголовок|20–60 байт|8 байт|
|Скорость|Ниже|Выше|
|Применение|HTTP, SMTP, FTP, SSH|DNS, VoIP, DHCP, видео|
