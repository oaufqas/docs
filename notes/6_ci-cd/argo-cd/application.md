#### Самое главное, что есть в argoCD - это **Application**

При его создани, указывается самое важное в github:
- Source Repository ==URL==
- Source Repository ==Branch==
- Source Repository ==Path== 

При установке argo в кластер, можно будет создавать новую сущность - application, чтобы настроить CD вручную

При создании Application и включении syncPolicy: automated, Argo будет проверять Git Repository каждые 3 минуты на наличие изменений, и делать kubectl apply на все файлы в этой директории, а также Helm чарты.

### Базовая структура Application 

```yaml
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: my-app                # 1. Идентификация
  namespace: argocd
spec:
  project: default            # 2. Проект
  source:                     # 3. Источник
    repoURL: <URL репозитория>
    targetRevision: HEAD
    path: ./k8s/overlays/dev
  destination:                # 4. Целевой кластер
    server: https://kubernetes.default.svc
    namespace: my-app-dev
  syncPolicy:                 # 5. Политика синхронизации (опционально)
    automated:
      prune: true
      selfHeal: true
```

#### Детальный разбор:

#### 1. Метаданные (`metadata`)

Это стандартные поля Kubernetes, которые идентифицируют ресурс.

| Поле             | Описание                                                                                                                                                                                                                                  | Пример / Значение                          |
| ---------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------ |
| **`name`**       | Уникальное имя приложения в рамках `namespace`.                                                                                                                                                                                           | `name: my-app-prod`                        |
| **`namespace`**  | Пространство имен, где **живет** сам ресурс `Application`. Обычно используется `argocd`.                                                                                                                                                  | `namespace: argocd`                        |
| **`finalizers`** | Защищает ресурсы приложения от каскадного удаления. Если вы удалите `Application`, без `finalizer` Argo CD просто удалит объект из API. C `finalizer` он сначала выполнит `prune` (удаление всех ресурсов приложения в целевом кластере). | `- resources-finalizer.argocd.argoproj.io` |

#### 2. Проект (`spec.project`)

Определяет, к какому **AppProject** относится приложение. Проекты используются для логической группировки приложений и управления правами доступа.

| Поле          | Описание                                 | Пример             |
| ------------- | ---------------------------------------- | ------------------ |
| **`project`** | Имя существующего в кластере AppProject. | `project: default` |

### 3. Источник (`spec.source`)

Это сердце приложения — оно указывает, откуда Argo CD должен взять манифесты.

| Поле                 | Описание                                                      | Пример                                                  |
| -------------------- | ------------------------------------------------------------- | ------------------------------------------------------- |
| **`repoURL`**        | URL Git-репозитория или Helm-репозитория.                     | `repoURL: https://github.com/my-org/infrastructure.git` |
| **`targetRevision`** | Ветка, тег или коммит в Git. Для Helm-чарта — версия чарта.   | `targetRevision: main` или `targetRevision: 1.2.3`      |
| **`path`**           | Путь в репозитории к директории с манифестами.                | `path: apps/guestbook`                                  |
| **`directory`**      | Дополнительные параметры для работы с **plain YAML** файлами. | См. таблицу ниже.                                       |
| **`helm`**           | [Опционально] Параметры для **Helm**-чарта.                   | См. таблицу ниже.                                       |

#### Параметры для `spec.source.directory` (работа с обычными YAML)

Эти параметры позволяют тонко настроить, какие именно YAML-файлы и как будут загружены из директории `path`.

| Поле          | Описание                                                                  | Пример                      |
| ------------- | ------------------------------------------------------------------------- | --------------------------- |
| **`recurse`** | Следует ли рекурсивно искать манифесты во всех поддиректориях.            | `recurse: true`             |
| **`include`** | Glob-паттерн для включения **только** определенных файлов или директорий. | `include: '{*.yml,*.yaml}'` |
| **`exclude`** | Glob-паттерн для **исключения** определенных файлов или директорий.       | `exclude: 'config.json'`    |

#### Параметры для `spec.source.helm` (работа с Helm-чартами)

| Поле              | Описание                                                                   | Пример                             |
| ----------------- | -------------------------------------------------------------------------- | ---------------------------------- |
| **`values`**      | Блок для прямого указания значений в формате YAML.                         | `values: \| replicas: 2`           |
| **`valueFiles`**  | Список путей к `values.yaml` файлам в Git-репозитории.                     | `valueFiles: ['values-prod.yaml']` |
| **`releaseName`** | Имя Helm-релиза. Если не указано, используется `metadata.name` приложения. | `releaseName: my-app-release`      |

### 4. Целевой кластер (`spec.destination`)

Определяет, **куда именно** будут развернуты ресурсы.

| Поле              | Описание                                                                                                                                                                           | Пример                                   |
| :---------------- | :--------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------- |
| **`server`**      | URL API-сервера целевого кластера. Для кластера, в котором работает сам Argo CD, используется значение `https://kubernetes.default.svc`.                                           | `server: https://kubernetes.default.svc` |
|  **`namespace`**  | **Пространство имен** в целевом кластере, куда будут развернуты ресурсы. Если его не существует, а в `syncOptions` есть `CreateNamespace=true`, Argo CD создаст его автоматически. |  `namespace: production`                 |
|  **`name`**       | Альтернатива `server`. Используется для ссылки на кластер по его имени, как он зарегистрирован в Argo CD.                                                                          | `name: in-cluster`                       |

### 5. Политика синхронизации (`spec.syncPolicy`)

Этот блок контролирует, **как и когда** Argo CD будет синхронизировать состояние в кластере с состоянием в Git. Это ключевой раздел для автоматизации.

#### `spec.syncPolicy.automated` (Автоматическая синхронизация)

| Поле             | Описание                                                                                                                              | Пример / Значение  |
| ---------------- | ------------------------------------------------------------------------------------------------------------------------------------- | ------------------ |
| **`prune`**      | Если `true`, Argo CD будет **удалять** ресурсы из кластера, которых нет в Git.                                                        | `prune: true`      |
| **`selfHeal`**   | Если `true`, Argo CD будет **восстанавливать** состояние из Git при любых ручных изменениях в кластере ("лечить" дрейф конфигурации). | `selfHeal: true`   |
| **`allowEmpty`** | Если `true`, позволяет приложению синхронизироваться, даже если в результате рендеринга не было ни одного ресурса.                    | `allowEmpty: true` |

> **Важно**: Чтобы отключить автоматическую синхронизацию, вы можете не указывать блок `automated` полностью или явно задать `automated: {}`.

#### `spec.syncPolicy.syncOptions` (Опции синхронизации)

Этот список позволяет задать специфическое поведение во время синхронизации. 

|Опция|Описание|
|---|---|
|**`CreateNamespace=true`**|Автоматически создает указанное в `destination.namespace` пространство имен, если оно не существует.|
|**`PruneLast=true`**|Откладывает удаление (`prune`) ресурсов до самого конца синхронизации. Полезно, если старые ресурсы нужны новым.|
|**`ApplyOutOfSyncOnly=true`**|Синхронизирует только те ресурсы, которые находятся в состоянии `OutOfSync`, что ускоряет процесс.|
|**`ServerSideApply=true`**|Использует `kubectl apply --server-side` для применения изменений. Это более эффективно для больших ресурсов и лучше работает с CRD.|
|**`RespectIgnoreDifferences=true`**|При синхронизации учитывает поля из `spec.ignoreDifferences`, не перезаписывая их.|
|**`Replace=true`**|(**Осторожно!**) Использует `kubectl replace` или `create` вместо `apply`. Это может привести к простою приложения.|
|**`PrunePropagationPolicy=foreground`**|Определяет стратегию удаления ресурсов. `foreground` (умолч.) — сначала зависимые, `background` — сначала владелец, `orphan` — сироты.|

### 6. Игнорирование различий (`spec.ignoreDifferences`)

Этот блок позволяет указать Argo CD **не обращать внимание** на изменения определенных полей в ресурсах. Это очень полезно для полей, которые заполняются контроллерами или операторами после создания (например, `status`).

```yaml
spec:
  ignoreDifferences:
    - group: apps
      kind: StatefulSet
      name: vault
      jsonPointers:
        - /status           # Игнорировать весь блок status
        - /spec/replicas    # Игнорировать конкретное поле
```

|Поле|Описание|
|---|---|
|**`group`**|Группа API ресурса (например, `apps`, `networking.k8s.io`).|
|**`kind`**|Тип ресурса (например, `Deployment`, `Service`, `Ingress`).|
|**`name`**|(Опционально) Имя конкретного ресурса. Можно указать `*` для всех.|
|**`namespace`**|(Опционально) Пространство имен ресурса.|
|**`jsonPointers`**|Список JSON Pointer путей к полям, которые нужно игнорировать.|
