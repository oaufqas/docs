**MySQL** — это свободная реляционная система управления базами данных (СУБД), которая использует язык структурированных запросов (SQL) для управления данными. Она является одной из самых популярных в мире благодаря своей скорости, надежности и открытому исходному коду.

### 1) Подготовка и установка напрямую на сервер

##### (Установка в docker-container [[../docker/run-db-in-docker#Run MySQL in docker container|docker-container]])

```bash
# Обновляем индексы пакетов
sudo apt update

# Устанавливаем сервер
sudo apt install mysql-server -y

# Проверяем статус службы
sudo systemctl status mysql
```

### 2) Первичная настройка безопасности

После установки обязательно запустите скрипт, который удалит анонимных пользователей, тестовую БД и запретит удаленный вход для root:

```bash
sudo mysql_secure_installation
```

_В скриптах автоматизации (Ansible/Terraform) этот этап обычно заменяется прямой подачей SQL-команд._

### 3) Настройка удаленного доступа (Bind Address)

По умолчанию MySQL слушает только `127.0.0.1`. Чтобы разрешить подключения извне (например, от App-серверов):

1. Откройте конфиг: `sudo nano /etc/mysql/mysql.conf.d/mysqld.cnf`
2. Найдите строку `bind-address = 127.0.0.1`
3. Измените на `0.0.0.0` (все интерфейсы) или конкретный внутренний IP.
4. Перезапустите: `sudo systemctl restart mysql`

### 4) Создание пользователя (Best Practice)

Не используйте `root` для приложений. Создайте отдельного юзера с ограниченными правами:

```sql
-- Зайти в консоль: sudo mysql
CREATE USER 'devops_user'@'%' IDENTIFIED BY 'vash_parol';
GRANT ALL PRIVILEGES ON vasha_baza.* TO 'devops_user'@'%';
FLUSH PRIVILEGES;
```

### 5) Полезные команды

- **Логи**: `/var/log/mysql/error.log` (первым делом смотрим сюда при падении).
- **Конфиги**: `/etc/mysql/my.cnf` и папка `/etc/mysql/conf.d/`.
- **Версия**: `mysqladmin -u root -p version`.

#### Архитектура и движки

- **InnoDB**: Стандарт де-факто. Поддерживает транзакции (ACID), блокировки на уровне строк и внешние ключи.
- **MyISAM**: Устаревший движок. Не поддерживает транзакции, блокирует всю таблицу при записи. Использовать только в специфических legacy-кейсах.

#### Производительность и Тюнинг (Основные параметры)

Большинство проблем решается правильной настройкой **[[../administration/optimization#MySQL|my.cnf]]** :

- **`innodb_buffer_pool_size`**: Самый важный параметр. Рекомендуется выделять **50–80%** всей оперативной памяти сервера под кэш данных и индексов.
- **`innodb_log_file_size`**: Размер лога транзакций (redo log). Влияет на скорость записи и время восстановления после сбоя.
- **`max_connections`**: Лимит соединений. Слишком высокое значение может исчерпать RAM; лучше использовать **Connection Pooler** (например, ProxySQL).
- **`slow_query_log`**: Обязательно включите для логирования запросов, превышающих `long_query_time`. 
- [[../administration/backup-restore#Команды для бэкапа MySQL|Бекапы]]

Высокая доступность (HA) и **[[../administration/replication#MySQL (Binary Log Replication)|Репликация]]**:

- **Replication**:
    - _Asynchronous_: Стандартная, возможна небольшая потеря данных при сбое Master.
    - _Semi-synchronous_: Master ждет подтверждения записи хотя бы от одного Slave.
- **Clustering**:
    - **Group Replication / InnoDB Cluster**: Официальное решение для высокой доступности с автоматическим failover.
    - **Galera Cluster**: Синхронная мульти-мастер кластеризация (часто в MariaDB).
- **Балансировка**: Используйте **ProxySQL** или **HAProxy** для распределения трафика между мастером (запись) и репликами (чтение)
