#### 1. Индексы (Самый быстрый профит)

- **B-Tree индексы:** Стандарт для большинства запросов (`WHERE`, `ORDER BY`, `GROUP BY`).
- **Составные индексы:** Важен порядок колонок. Индекс `(a, b)` работает для `WHERE a=1` и `WHERE a=1 AND b=2`, но **не** работает для `WHERE b=2`.
- **Покрывающие индексы:** Когда все данные для `SELECT` есть в самом индексе — база не лезет в таблицу (Index Only Scan).
- **Лишние индексы:** Каждый индекс замедляет `INSERT/UPDATE/DELETE`. Удаляй неиспользуемые.

#### 2. Оптимизация запросов (SQL)

- **EXPLAIN:** Твой главный инструмент. Смотри на `type` (MySQL) или `Plan` (Postgres). Избегай `ALL` / `Seq Scan` на больших таблицах.
- _*_SELECT : Никогда не используй в коде. Тащи только нужные колонки — это экономит RAM и ширину канала.
- **N+1 Problem:** Избегай кучи мелких запросов в цикле, используй `JOIN` или `IN (...)`.
- **Пагинация:** `LIMIT 100000, 20` убивает базу. Используй "keyset pagination" (фильтрация по последнему ID) вместо большого `OFFSET`.

#### 3. Конфигурация ресурсов (Инфраструктура)

- **Память (RAM):**
    - **MySQL:** `innodb_buffer_pool_size` — ставь 70-80% от всей RAM сервера.
    - **Postgres:** `shared_buffers` (обычно 25% RAM), `work_mem` (память на один запрос/сортировку).
- **Диски:** Используй SSD/NVMe. Мониторь **IOPS** и **iowait**.
- **Connections:** База тратит ресурсы на каждое соединение. Используй **PGBouncer** для Postgres или встроенные пулы в приложении.

#### 4. Архитектурные паттерны

- **Репликация:** Выноси тяжелый `SELECT` (аналитику, отчеты) на **Replica**, оставляй **Master** только для записи.
- **Партиционирование (Partitioning):** Разбиение огромной таблицы на части (например, по месяцам). Ускоряет поиск и удаление старых данных.
- **Шардирование (Sharding):** Горизонтальное деление данных между разными серверами (крайняя мера).
- **Нормализация vs Денормализация:** Иногда лучше добавить одну колонку с готовым значением, чем делать `JOIN` пяти таблиц в каждом запросе.

#### 5. Обслуживание (Maintenance)

- **Postgres:** Следи за **Vacuum** (очистка мертвых строк). Если он не успевает, база «раздувается» (Bloat) и тормозит.
- **MySQL:** `OPTIMIZE TABLE` для дефрагментации файлов данных (особенно после массовых удалений).
- **Статистика:** База строит план запроса на основе статистики. Если она устарела, оптимизатор выберет плохой путь. (`ANALYZE` в Postgres).

DevOps-чеклист мониторинга:

1. **Slow Query Log:** Включи логирование запросов дольше 100-500мс.
2. **Connection Count:** Чтобы не упереться в лимит.
3. **Buffer Cache Hit Ratio:** Процент данных, читаемых из RAM, а не с диска (в идеале >95-99%).
4. **Transaction Logs:** Следи за местом под WAL (Postgres) или Binary Logs (MySQL).

## Настройки конфигурации бд для обеспечения скорости:

#### MySQL: 

- Debian (`/etc/mysql/my.cnf`, `/etc/mysql/mysql.conf.d/mysqld.cnf`)
- CentOs (`/etc/my.cnf`)
- Docker (`/my/custom/config.cnf:/etc/mysql/conf.d/config.cnf`)

MySQL «из коробки» настроен очень скромно. Основной фокус здесь на движок **InnoDB**.

- **`innodb_buffer_pool_size`**: Самый важный параметр. Кэширует данные и индексы.
    - _Рекомендация:_ **70-80%** от всей RAM сервера (если база выделенная).
- **`innodb_log_file_size`**: Размер лога транзакций (redo log).
    - _Зачем:_ Большой лог ускоряет запись (меньше сбросов на диск), но увеличивает время восстановления после сбоя. Обычно ставят **512MB – 2GB**.
- **`innodb_flush_log_at_trx_commit`**: Баланс между скоростью и надежностью.
    - `1` (default): Максимальная надежность (запись на диск при каждом коммите).
    - `2`: В разы быстрее (запись раз в секунду). Риск потери 1 сек данных при выключении питания.
- **`max_connections`**: Лимит соединений.
    - _Совет:_ Не ставь слишком много (например, >1000), если RAM мало, так как каждое соединение ест память.

#### PostgreSQL:

- Debian (`/etc/postgresql/[версия]/main/postgresql.conf`)
- CentOS (`/var/lib/pgsql/[версия]/data/postgresql.conf`)
- Docker (`/var/lib/postgresql/data/postgresql.conf`)  внутри контейнера

У Postgres другие названия, и он больше полагается на кэш самой операционной системы.

- **`shared_buffers`**: Память для разделяемого кэша базы.
    - _Рекомендация:_ **25%** от всей RAM (редко больше 40%).
- **`work_mem`**: Память для ОДНОЙ операции сортировки или join внутри запроса.
    - _Зачем:_ Если запросу не хватает `work_mem`, он свопится на диск (очень медленно).
    - _Совет:_ Начинай с **8-16MB**. Осторожно: если 100 соединений одновременно делают сложные запросы, они могут съесть `100 * work_mem` памяти.
- **`effective_cache_size`**: Подсказка планировщику, сколько памяти доступно в кэше ОС.
    - _Рекомендация:_ **50-75%** от всей RAM. На саму память не влияет, только на выбор плана запроса.
- **`maintenance_work_mem`**: Память для сервисных задач (`VACUUM`, создание индексов).
    - _Совет:_ Можно смело ставить **256MB – 1GB**.
- **`max_wal_size`**: Максимальный объем логов предзаписи (WAL) до принудительной точки проверки (checkpoint).
    - _Совет:_ **1GB – 4GB** для нагруженных систем.

#### Инструменты для автоматического тюнинга

Если не хочешь гадать, используй эти проверенные скрипты, которые проанализируют твою живую базу и дадут советы:

1. **MySQLTuner** (для MySQL):

```
perl mysqltuner.pl
```

2. **PGConfigurator** или **PGTune** (онлайн-сервисы): Вводишь объем RAM и CPU — получаешь готовый конфиг для Postgres.
