 **Аксиома:** Нет бэкапа — нет данных. Есть бэкап, но нет проверки восстановления — тоже нет данных.

1 ) **Виды бэкапов:**

- **Логический (mysqldump, pg_dump):** Дамп в SQL. Гибкий, можно воскресить одну таблицу. **Минус:** медленный для больших баз. Создает SQL-файл с командами для воссоздания структуры и данных (`INSERT`, `CREATE TABLE` и тд.)

- **Физический (копирует сами файлы базы данных с диска):** Быстрый слёпок всей базы. Идеален для `pg_basebackup` (PostgreSQL) или тулов вроде **WAL-G**. Быстро, но привязан к версии БД и ОС.

- **WAL-архивирование (Continuous Archiving)**: Связка физического бэкапа + бекап WAL-логов (всех изменений). Позволяет делать **point-in-time recovery (PITR)** — откатиться на любую секунду.

2 ) **Состояние базы при бэкапе:**

- **Холодный (Offline)**: База выключена. Данные гарантированно консистентны, но сервис недоступен.

- **Горячий (Online)**: База работает. Требует специальных инструментов, чтобы копия была целостной, несмотря на идущие транзакции.


# Команды для бэкапа PostgreSQL:

`pg_dump` для логических копий, `pg_basebackup` для физических

```bash
# Создание логического дампа в обычном текстовом формате:
pg_dump -U [usr] -d [db] > backup.sql


# Восстановление из текстового дампа через psql:
psql -U [usr] -d [db] -f backup.sql
```


###### Сжатый формат:

```bash
# Использование кастомного формата (`-Fc`) позволяет восстанавливать данные в несколько потоков и выбирать отдельные таблицы: 
pg_dump -U [usr] -F c -f backup.dump [db]


# Восстановление через pg_restore (быстрее и гибче):
pg_restore -U [usr] -d [db] -j 4 backup.dump 
# (Опция -j 4 запускает восстановление в 4 потока). 
```


###### Бэкап всего кластера (все базы, пользователи, права):

```bash
pg_dumpall -U postgres > all_databases.sql


psql -f all_databases.sql postgres # Восстановление 
```


###### Физический бэкап (Base Backup):

Используется для создания реплик или настройки PITR (восстановления на момент времени): 

```bash
pg_basebackup -h [хост] -D [папка_назначения] -U [пользователь] -P -v 

# (Опция -P показывает прогресс, -v — подробный вывод). 
```

`wal-g backup-push` (для PostgreSQL) и регулярный `pg_verifybackup`.


# Команды для бэкапа MySQL:

```bash
mysqldump -u [usr] -p [db] > backup.sql # Бэкап одной базы данных

mysqldump -u [usr] -p [db] table1 table2 > tables_backup.sql # Конкретных таблиц

--all-databases # все бд
--no-data [database_name] # только структура (без данных)
--single-transaction # Обязателен для таблиц InnoDB. Позволяет сделать бэкап без блокировки таблиц на чтение/запись (база продолжает работать).
--routines # Включает в бэкап хранимые процедуры и функции.
--triggers # Включает триггеры


# Для восстановления используется стандартный клиент `mysql`, который просто выполняет SQL-коды из файла.
mysql -u [user] -p [database_name] < backup.sql

```

###### Оптимизация и сжатие:

Дампы текстовые, поэтому они отлично сжимаются «на лету», что экономит место и время передачи по сети.

```bash
# Бэкап со сжатием (gzip):
mysqldump -u [user] -p [database_name] | gzip > backup.sql.gz


#Восстановление из архива:
gunzip < backup.sql.gz | mysql -u [user] -p [database_name]
```

###### Физический бэкап (Percona XtraBackup)

Если база весит сотни гигабайт, `mysqldump` будет работать часами. В таких случаях используют **XtraBackup**. Она копирует сами файлы данных (`.ibd`).

- **Создание бэкапа:**

```bash
# Создание бэкапа
xtrabackup --backup --target-dir=/path/to/backup/


# Подготовка (Prepare) перед восстановлением:
xtrabackup --prepare --target-dir=/path/to/backup/
```
