### **Что такое база данных и СУБД?**

- **База данных (БД)** — организованная коллекция структурированных данных, хранящаяся в электронном виде.

- **Система управления базами данных (СУБД)** — программное обеспечение для определения, обработки, извлечения и управления данными в БД.    

### **Реляционные vs Нереляционные базы данных**

**Ключевое различие:** SQL базы используют таблицы с предопределенной схемой, а NoSQL предлагают гибкие модели данных без жесткой схемы.

| Характеристика         | **Реляционные (SQL)**                         | **Нереляционные (NoSQL)**                       |
| ---------------------- | --------------------------------------------- | ----------------------------------------------- |
| **Модель данных**      | Таблицы (строки и столбцы)                    | Документы, ключ-значение, графы, колонки        |
| **Схема**              | Строгая, предопределенная                     | Гибкая, динамическая                            |
| **Язык запросов**      | SQL (стандартизированный)                     | Зависит от БД (API, специфичные языки)          |
| **Транзакции**         | ACID (строгая согласованность)                | BASE (согласованность в конечном счете)         |
| **Масштабирование**    | Вертикальное (больше ресурсов серверу)        | Горизонтальное (больше серверов                 |
| **Когда использовать** | Структурированные данные, финансы, транзакции | Большие данные, гибкие модели, высокая нагрузка |

### **ACID vs BASE

**ACID** — для систем, где важна надежность:

- **A**tomicity (Атомарность) — транзакция выполняется целиком или не выполняется
    
- **C**onsistency (Согласованность) — данные всегда в валидном состоянии
    
- **I**solation (Изоляция) — параллельные транзакции не мешают друг другу
    
- **D**urability (Долговечность) — зафиксированные данные не теряются
    

**BASE** — для высокомасштабируемых систем:

- **B**asically **A**vailable (Базовая доступность) — система отвечает всегда
    
- **S**oft state (Мягкое состояние) — данные могут меняться
    
- **E**ventually consistent (Конечная согласованность) — рано или поздно данные станут согласованными
    

### **Теорема CAP (теорема Брюера)**

**Суть:** В распределенной системе можно обеспечить только **2 из 3** свойств:

- **C**onsistency (Согласованность) — все узлы видят одни и те же данные
    
- **A**vailability (Доступность) — каждый запрос получает ответ
    
- **P**artition tolerance (Устойчивость к разделению) — система работает даже при разрывах сети
    

При проектировании нужно выбирать, чем жертвовать.

### **Ключевые концепции

- **Индексы** — структуры данных для ускорения поиска. Ускоряют SELECT, замедляют INSERT/UPDATE/DELETE.
    
- **Репликация** — копирование данных на другие серверы (для отказоустойчивости и распределения нагрузки).
    
    - **Синхронная** — данные подтверждаются только после записи на все реплики (медленнее, но надежнее)
        
    - **Асинхронная** — мастер не ждет реплики (быстрее, но возможна потеря данных)
        
- **Шардирование** — горизонтальное разделение данных по разным серверам.
    
- **Бэкапы**:
    
    - **Логические** (`pg_dump`, `mysqldump`) — гибкие, но медленные
        
    - **Физические** (копирование файлов) — быстрые, для больших баз
        
    - **PITR (Point-In-Time Recovery)** — восстановление на любой момент времени (WAL-архивы в PostgreSQL)
