**Grafana Loki** — это горизонтально масштабируемая, отказоустойчивая система централизованного логирования от компании Grafana Labs.

Её создавали как «Prometheus для логов». Главная идея разработчиков заключалась в том, чтобы сделать инструмент, который будет максимально дешёвым в обслуживании, потреблять минимум оперативной памяти и использовать тот же подход к метаданным (лейблам), что и Prometheus. Написан на GO.

В отличие от Elastic Stack, который индексирует каждое слово в тексте лога, **Loki не индексирует сам текст**.

- Он индексирует **только метаданные** (лейблы): имя пода (`pod`), пространство имен (`namespace`), имя контейнера (`container`), окружение (`env`).
- Сам текст лога сжимается в отдельные блоки (чанк-файлы) и отправляется в дешёвое объектное хранилище (например, в Yandex Object Storage / AWS S3).
- **Результат:** индексы Loki весят в сотни раз меньше, чем у Elastic, а требования к RAM минимальны.

Плюсы и минусы Grafana Loki

**Плюсы:**

- **Идеальная интеграция с Grafana:** Логи и метрики находятся в одном интерфейсе. Ты можешь кликнуть на график всплеска ошибок в Grafana и тут же открыть логи этого пода за эту конкретную секунду.
- **Экономия на дисках:** Благодаря сжатию и хранению в S3, стоимость удержания логов за полгода становится копеечной.
- **Знакомый синтаксис:** Язык запросов **LogQL** очень похож на PromQL (язык запросов Prometheus).

**Минусы:**

- **Низкая скорость полнотекстового поиска:** Если ты ищешь редкое слово по тексту (например, ошибку `"NullPointerException"`), а не по лейблам, Loki придется скачать из S3 гигабайты чанков и прочитать их методом `grep`. На больших объемах данных такие запросы работают медленно.
- **Проблема высокой кардинальности (High Cardinality):** В лейблы Loki **категорически нельзя** пихать динамические данные (например, IP-адреса пользователей или `user_id`). Если ты это сделаешь, Loki создаст миллионы мелких индексов и кластер упадет от нехватки памяти.

Как устроен язык запросов LogQL

Запросы в Loki делятся на две части: **селекторы потоков** (выбор по лейблам) и **фильтры** (поиск по тексту через пайпы `|`).

- _Простой поиск по тексту:_  
    `{app="rent", container="server"} |= "database error"`  
    _(Ищет строку "database error" строго внутри контейнера server твоего приложения)_.
- _Исключение из поиска:_  
    `{app="rent"} != "DEBUG"`  
    _(Покажет все логи приложения, кроме тех, где есть слово DEBUG)_.
- _Агрегация (метрики из логов):_  
    `sum(count_over_time({app="rent"} |= "error" [5m]))`  
    _(Посчитает количество ошибок в логах твоего сервера за последние 5 минут и позволит построить по этому числу график)_.

Схема развертывания через Argo CD в Yandex Cloud

В продакшене Loki часто разделяют на отдельные микросервисы (компактор, дистрибьютор, инжестор). Так же есть режим **Single Binary** (все процессы в одном поде).


```yaml
loki:
  auth_enabled: false
  commonConfig:
    replication_factor: 1
  
  # Настраиваем хранение логов в облачном бакете Яндекса вместо дисков HDD
  storage:
    type: s3
    s3:
      endpoint: storage.yandexcloud.net
      bucket: my-k8s-logs-bucket # Имя твоего бакета в YC
      region: ru-central1
      access_key_id: "ТВОЙ_КЛЮЧ"
      secret_access_key: "ТВОЙ_СЕКРЕТНЫЙ_КЛЮЧ"
```

Для сбора логов с нод в комплекте с Loki разворачивается **Promtail** или **Fluent Bit**. Они просто читают файлы `/var/log/pods/*` на ноде и отправляют их в Loki по HTTP API.
