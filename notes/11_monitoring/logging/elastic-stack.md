**Elastic Stack** (исторически известный как **ELK**) — это самый известный, мощный и коммерчески успешный комплекс инструментов для работы с большими данными, логами и системной аналитикой.

Долгое время он был монополистом в сфере централизованного логирования (Log Management), но сегодня в Kubernetes-среде его всё чаще используют для сложной бизнес-аналитики (SIEM, безопасность, умный поиск по логам), уступая место более легковесным решениям (Loki, VictoriaLogs) в повседневных задачах DevOps.

---

Из чего состоит Elastic Stack (Аббревиатура ELK)

Стек состоит из четырех основных компонентов, каждый из которых выполняет свою строгую роль:

1. **Elasticsearch (E):**
    - **Что это:** Сердце всего стека. Это распределенная база данных, основанная на поисковом движке Apache Lucene.
    - **Как работает:** Она не просто хранит логи, а строит по ним тяжелые «инвертированные индексы» (как указатель терминов в конце книги). Благодаря этому поиск фразы среди миллиарда строк происходит за миллисекунды.
2. **Logstash (L):**
    - **Что это:** Тяжелый комбайн для обработки данных (ETL-инструмент).
    - **Как работает:** Он принимает логи из сотен источников (файлы, базы данных, очереди сообщений), чистит их, парсит (например, превращает регулярными выражениями строчку лога Nginx в красивый JSON), обогащает данными (например, по IP-адресу определяет страну пользователя) и отправляет в Elasticsearch.
3. **Kibana (K):**
    - **Что это:** Веб-интерфейс и панель управления.
    - **Как работает:** Позволяет искать логи через специальный язык запросов (KQL), строить графики, карты, настраивать алерты и управлять безопасностью самого кластера Elasticsearch.
4. **Beats (Появились позже):**
    - **Что это:** Семейство очень легковесных агентов сбора данных на Go (Filebeat — для логов, Metricbeat — для метрик, Packetbeat — для сети).
    - **Зачем нужны:** Logstash слишком тяжелый, чтобы ставить его на каждый сервер. Поэтому на сервера ставят Filebeat, который просто «читает файл и отправляет в Logstash/Elasticsearch».

Плюсы и минусы Elastic Stack в Kubernetes

**Плюсы:**

- **Идеальный полнотекстовый поиск:** Если тебе нужно искать логи со сложной логикой, учетом морфологии, опечаток или нечетким соответствием (Fuzzy search) — равных Elastic нет.
- **Экосистема:** Готовые модули для Nginx, MySQL, PostgreSQL. Ты просто включаешь модуль, и Kibana сама строит красивые дашборды.
- **Enterprise-функции:** Встроенные инструменты машинного обучения (поиск аномалий в логах) и мощная подсистема безопасности.

**Минусы:**

- **Аномальное потребление ресурсов:** Elasticsearch написан на Java. Ему требуются гигабайты оперативной памяти (`JVM Heap`) просто для того, чтобы запуститься в режиме ожидания.
- **Дороговизна на дисках:** Инвертированные индексы могут занимать на диске столько же места, сколько весят сами сырые логи. 
- **Сложность эксплуатации в K8s:** Чтобы запустить его отказоустойчиво, нужен сложный **Elastic Cloud on Kubernetes (ECK) Operator**. Настройка индексов, шардов и реплик — это отдельная профессия.

Как его правильно разворачивать в проде (GitOps / Argo CD)

В современном Kubernetes связку собирают без Logstash (так как он слишком прожорливый), заменяя его на **Fluent Bit** или **Vector**, а Elasticsearch разворачивают через официальный оператор.

Для установки через Argo CD в инфраструктурный репозиторий добавляют чарт оператора.

**Манифест самого кластера Elastic (CRD, который понимает оператор):**

```yaml
apiVersion: ://amazonaws.com # пример синтаксиса ECK
kind: Elasticsearch
metadata:
  name: quickstart
  namespace: logging
spec:
  version: 8.11.0 # Фиксируем версию
  nodeSets:
  - name: default
    count: 3 # Для прода нужно минимум 3 ноды (мастер-кворум)
    config:
      node.store.allow_mmap: true
    podTemplate:
      spec:
        containers:
        - name: elasticsearch
          resources:
            requests:
              memory: 4Gi # МЕНЬШЕ ЭТОГО ОН ПРОСТО НЕ ВЗЛЕТИТ
              cpu: 2

```
